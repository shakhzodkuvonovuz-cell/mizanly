| 1 | C | stories.service.ts:484-492 | **Raw SQL uses wrong column names.** `getReactionSummary` queries `story_id`, `sticker_type`, `response_data` (snake_case) but migration DDL confirms actual DB columns are `"storyId"`, `"stickerType"`, `"responseData"` (camelCase). | 2 | C | stories.service.ts:252-289 | **`getById` does not check block/mute relationships.** A blocked user can view their blocker's stories by direct story ID. | 3 | C | stories.service.ts:252-289 | **`getById` does not enforce `closeFriendsOnly` or `subscribersOnly`.** Feed filtering (L108-116) rejects these, but direct ID access bypasses it entirely. | Close-friends and subscriber-only stories are publicly accessible via direct ID. | S — add the same closeFriendsOnly/subscribersOnly checks as the feed |
| 4 | H | stories.service.ts:381-389 | **`replyToStory` does not check `expiresAt` or `isArchived`.** Users can reply to expired or archived stories, creating DM conversations and incrementing `repliesCount` for dead content. | Data integrity — phantom replies to expired stories, inflated reply counts. | S — add expiry+archived check before L386 |
| 5 | H | stories.service.ts:568-576 | **`submitStickerResponse` does not check `expiresAt` or `isArchived`.** Users can submit sticker responses to expired stories. | Stale interaction data on expired content. | S — add expiry+archived check before L571 |
| 6 | H | stories.service.ts:280-284 | **`getById` records view via fire-and-forget upsert but does NOT increment `viewsCount`.** The `markViewed` endpoint (L327-333) does increment in a transaction. | 10 | H | stories.service.ts:427-464 | **`replyToStory` performs 3 independent writes without `$transaction`.** `message.create`, `story.update(repliesCount++)`, and `conversation.update(lastMessage)` are sequential but not atomic. | 11 | H | story-chains.service.ts:168-169 | **`joinChain` new-entry detection is unreliable.** `isNew = upserted.createdAt.getTime() > Date.now() - 1000` uses a 1-second window that breaks under clock skew, slow DB, or concurrent load. | 12 | M | schema.prisma:1101-1134 | **Missing `@@index([highlightAlbumId])` on Story.** `getHighlights` queries stories via the `highlightAlbum` relation (WHERE `highlightAlbumId = ?`). | 13 | M | schema.prisma:1101-1134 | **`Story.userId` uses `onDelete: SetNull`.** When a user is deleted, their stories become orphaned with `userId = null`. The privacy service does `story.deleteMany({ where: { userId } })` at L502, which runs BEFORE the user row deletion and should work — but if execution order changes or the `onDelete: SetNull` cascades trigger first, stories become unreachable. Should use `onDelete: Cascade` since stories ARE deleted during account deletion anyway. | Fragile deletion order dependency. If refactored, orphaned stories leak media URLs permanently. | S — change to `onDelete: Cascade` (stories already deleted explicitly) |
| 14 | M | stories.service.ts:533-540 | **`deleteHighlight` orphans `isHighlight = true` on stories.** `onDelete: SetNull` nulls `highlightAlbumId`, but `isHighlight` remains `true`. | 15 | M | privacy.service.ts:601-603 | **Account deletion removes `StoryView` and `StoryChainEntry` rows without decrementing `viewsCount` / `participantCount`.** L601: `storyView.deleteMany` — story `viewsCount` not decremented.
