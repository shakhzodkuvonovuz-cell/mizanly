# B08 — CoinBalance, CoinTransaction, GiftRecord, Cashout, Donation, PaymentMapping, ProcessedWebhookEvent

**Scope note:** `Cashout` and `Donation` do not exist as standalone Prisma models. Cashout logic lives in `GiftsService.cashout()` and `MonetizationService.requestCashout()`. Donation logic (CharityDonation, ZakatDonation, WaqfDonation) was not found in the audited files — those models exist in the schema but no service files in scope reference them. This audit covers the 5 models that exist plus all associated service code.

## Findings

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 1 | C | payments.service.ts:654-678 | **Tip webhook credits diamonds but never creates CoinTransaction record.** `handleTipPaymentSucceeded` upserts `CoinBalance.diamonds` but does NOT create a `CoinTransaction` of type `TIP_RECEIVED` for the receiver. The reconciliation audit (`auditCoinBalances` L586-606) expects `TIP_RECEIVED` transactions to exist and computes expected diamonds from them. Every tip receiver will be flagged as a balance discrepancy. Same omission in `reconcileTips` (L136-155) which also credits diamonds without a `CoinTransaction`. | Diamond audit permanently broken — produces false discrepancies for every user who received a tip. Diamonds credited but no audit trail for compliance. | S — add `tx.coinTransaction.create({ type: 'TIP_RECEIVED', ... })` inside the existing `$transaction` block |
| 2 | C | schema.prisma:3102 | **CoinBalance uses `onDelete: Cascade` on user FK.** When a user is hard-deleted (privacy.service.ts L653), `CoinBalance` is cascade-deleted. This destroys the financial balance record. While `CoinTransaction` correctly uses `SetNull` to preserve audit trail, the balance record itself (which is the source of truth for coins/diamonds) is destroyed. If the hard-delete cron runs before the reconciliation cron, balance discrepancies become undetectable. | Financial audit trail incomplete — balance at time of deletion is lost forever. Should use `SetNull` or explicit `deleteMany` after archiving. | M — change to `SetNull`, add archive step before deletion |
| 3 | H | gifts.service.ts:291-292, monetization.service.ts:466-467 | **Cashout methods have dead code after unconditional `throw`.** Both `cashout()` and `requestCashout()` throw immediately at line 1, making all validation, deduction, and transaction recording unreachable. When cashout is re-enabled by removing the `throw`, the dead code will execute WITHOUT a `$transaction` block — the `updateMany` (deduct diamonds), post-balance check, and `coinTransaction.create` are three separate DB operations, not atomic. A crash between deduct and record leaves diamonds deducted but no transaction logged. | When cashout is enabled: non-atomic deduction + record = lost money on crash. Dead code gives false confidence the flow is implemented. | M — wrap in `$transaction` before enabling |
| 4 | H | payments.service.ts:90-94, 105-109 | **PaymentMapping DB write is fire-and-forget (unawaited Promise with `.catch()`).** `storePaymentIntentMapping` and `storeSubscriptionMapping` do `this.prisma.paymentMapping.upsert(...).catch(...)` without `await`. If DB write fails (network blip, constraint error), the mapping is only in Redis. When Redis TTL expires (30 days for tips, 1 year for subscriptions), the webhook handler loses the ability to find the internal ID, causing orphaned payments. | Silent loss of durable payment mapping. Redis flush/expiry + DB write failure = permanent orphaned payment that reconciliation may not catch within TTL window. | S — add `await` (or at minimum `try/catch` with retry) |
| 5 | H | schema.prisma:3118, 3103-3104 | **CoinTransaction.amount is `Int` but serves dual purpose.** `amount` stores both coin counts (positive for purchase, negative for gift sent) AND diamond amounts (negative for cashout, positive for gift received). Coins and diamonds have different exchange rates ($0.0099/coin vs $0.007/diamond), but they share the same `Int` field. The `currency` field defaults to "USD" but is never set to anything else, creating confusion: is `amount` coins, diamonds, or USD cents? The reconciliation audit sums these mixed values together. | Ambiguous ledger — mixing coins and diamonds in the same `amount` column makes the transaction log unreliable for financial reporting or tax compliance. | L — this is a design debt; fix requires adding a `unit` column ('coins'/'diamonds') or separate tables |
| 6 | H | payments.service.ts:493-501 | **Coin purchase idempotency relies on `description CONTAINS paymentIntentId`.** The dedup check for `handleCoinPurchaseSucceeded` searches for any `CoinTransaction` where `description` contains the PI ID string. This is fragile: (a) if the description format changes, dedup breaks; (b) `CONTAINS` is a substring match — a PI ID that is a substring of another PI ID could false-match; (c) there is no index on `description`, making this a full-table scan. | Double-crediting coins if dedup check fails. Performance degrades as transaction table grows (no index on text search). | M — add a dedicated `stripePaymentIntentId` column to `CoinTransaction` with unique index |
| 7 | H | payments.service.ts:629-645 | **Tip lookup fallback matches by `senderId` + `status: pending` + optional `amount`, ordered by `createdAt desc`.** If a user has multiple pending tips (e.g., sent two tips quickly), this matches the MOST RECENT pending tip, not necessarily the one for this PaymentIntent. The wrong tip gets marked as completed, and the actual tip remains stuck as pending forever. | Money misattribution — wrong tip completed, actual tip orphaned. The receiver of tip A gets credited for tip B's payment. | M — store `stripePaymentId` on the Tip at creation time (before webhook), not just in Redis/PaymentMapping |
| 8 | M | gifts.service.ts:78-109 | **`purchaseCoins` creates a pending CoinTransaction but credits nothing and stores no PaymentIntent reference.** The method is supposed to work with Stripe webhooks, but the pending transaction has no link to a specific PaymentIntent. When the webhook fires, `handleCoinPurchaseSucceeded` matches pending transactions by `(userId, type='PURCHASE', amount=coinAmount, description CONTAINS 'pending payment')`. If the user makes two purchases of the same amount, both pending transactions match and only one gets updated — the other remains stuck as "pending" forever. | Ghost pending transactions accumulate. If user purchases same coin amount twice, one transaction record permanently shows "pending payment" even though coins were credited. | S — include PI ID in the pending transaction description, or return PI ID to client and match on it |
| 9 | M | payments.service.ts:217-218 | **Tip amount converted to cents via `Math.round(amount * 100)` with `amount` as a JS `number`.** Floating-point arithmetic means `Math.round(19.99 * 100)` could produce 1998 instead of 1999 on some runtimes. The Tip model uses `Decimal(12,2)` correctly, but the Stripe PI creation uses raw JS floating-point math. | Users could be charged 1 cent less/more than intended. Small but real for financial compliance. | S — use `Decimal` or integer cents throughout |
| 10 | M | schema.prisma:4975 | **`ProcessedWebhookEvent` has a redundant index.** The model has `eventId String @unique` (which creates a unique index) AND `@@index([eventId])` (which creates a second non-unique index on the same column). The `@unique` constraint already provides indexed lookups. | Wasted storage and write overhead for every webhook event (double index maintenance). | S — remove the `@@index([eventId])` line |
| 11 | M | stripe-webhook.controller.ts:134-137 | **Webhook handler re-throws errors after processing, returning 500 to Stripe.** If a handler (e.g., `handlePaymentIntentSucceeded`) throws, the controller re-throws, returning HTTP 500. Stripe will retry 500 responses for up to 3 days. But the event is NOT marked as processed (dedup happens after success). If the error is deterministic (e.g., invalid data), every retry will fail identically, creating noise for 3 days. | Infinite retry loop for deterministic failures. Stripe sends up to ~16 retries over 3 days, all failing identically. | M — catch deterministic errors, log to Sentry, return 200 to stop retries, persist to dead-letter queue |
| 12 | M | (none) | **No TTL / cleanup for `ProcessedWebhookEvent` table.** Events are inserted on every webhook and never deleted. Stripe retries for at most 3 days, so events older than 7 days have no dedup value. The table will grow unboundedly (one row per webhook event, forever). | Unbounded table growth. At scale (thousands of transactions/day), this becomes millions of rows of useless dedup records. | S — add a cron to delete `ProcessedWebhookEvent` records older than 30 days |
| 13 | M | (none) | **No TTL / cleanup for `PaymentMapping` table.** Similar to ProcessedWebhookEvent — mappings are created but never pruned. Completed/failed/cancelled tips and subscriptions keep their mapping rows forever. | Unbounded table growth for stale mappings. | S — add a cron to prune completed+old mappings |
| 14 | M | gifts.service.ts:216-251 | **`getHistory` ignores cursor for giftsSent and giftsReceived queries.** The `cursor` parameter is only applied to the `coinTransaction` query. The `giftsSent` and `giftsReceived` queries always fetch the latest N records with no pagination. For users with many gifts, this means the gift lists never paginate — only the transaction list does. | Broken pagination for gift history. Users with 1000+ gifts will always see the same first 20. | M — add cursor-based pagination to all three queries, or document that gifts section is not paginated |
| 15 | M | gifts.service.ts:266-278 | **Gift history uses `(g as any).receiver` and `(g as any).sender` casts.** The `include` clause correctly joins receiver/sender, but the return type doesn't reflect this. Using `as any` bypasses type safety and could hide runtime errors if the include is accidentally removed. | Type safety gap — if someone removes the `include`, the `as any` cast silences the `undefined` access instead of catching it at compile time. | S — define a proper return type or use Prisma's generated include types |
| 16 | M | counter-reconciliation.service.ts:316-335 | **Negative balance reset is not atomic with audit logging.** The `reconcileCoinBalances` method finds negative balances, then updates each one individually without a transaction. If two runs overlap (unlikely but possible with manual trigger), the balance could be double-reset or the Sentry message could log stale values. | Minor race condition in the safety net. The reset itself is idempotent (setting to 0), but the logged values could be inaccurate. | S — wrap find+update in a transaction per user |
| 17 | L | monetization.service.ts:33-34 | **Duplicate constant definitions.** `DIAMOND_TO_USD`, `DIAMONDS_PER_USD_CENT`, and `MIN_CASHOUT_DIAMONDS` are defined in both `gifts.service.ts` (L47-49) and `monetization.service.ts` (L33-35). | 18 | L | payments.service.ts:164 | **Coin price calculation uses floating-point multiplication.** `Math.round(coinAmount * 0.99)` — if `coinAmount` is large (e.g., 99999), floating-point precision could round incorrectly. For `coinAmount = 99999`, `99999 * 0.99 = 98999.01`, which rounds correctly, but the pattern is fragile for future rate changes. | Negligible for current rates but could cause penny discrepancies if the rate changes to something like 0.97. | S — use integer arithmetic: `Math.round(coinAmount * 99) / 100 * 100` or Decimal |

While harmless, it adds a needless DB round-trip inside the transaction. | Extra latency (one SELECT per gift send) for a condition that cannot occur. | I — remove or keep as defense-in-depth, but document why |
| 20 | L | payments.service.ts:614-650 | **`handleTipPaymentSucceeded` does a second query after the transaction to fetch tip for notification.** Line 681 does `prisma.tip.findFirst({ where: { stripePaymentId: paymentIntent.id } })` OUTSIDE the transaction.
