| # | Sev | File:Line | Finding | Impact | Fix Effort |

| 1 | C | counter-reconciliation.service.ts:354 | Raw SQL uses `"Thread"` (Prisma model name) instead of `"threads"` (@@map table name) in `LEFT JOIN "Thread" t`. | 2 | H | majlis-lists.service.ts:360-365 | `getTimeline()` thread query does not filter `user: { isDeactivated: false, isBanned: false, isDeleted: false }`. | Banned/deactivated/deleted users' threads appear in MajlisList timelines. | 3 | H | majlis-lists.service.ts:360-365 | `getTimeline()` does not filter `scheduledAt` (future-scheduled threads). Missing `OR: [{ scheduledAt: null }, { scheduledAt: { lte: new Date() } }]`. | Future-scheduled threads from list members leak before their intended publish time. | S — Add scheduledAt filter |
| 4 | H | bookmarks.service.ts:257-290 | `getSavedThreads()` does not filter `thread: { isRemoved: false }`. | 5 | H | threads.service.ts:73-79 | `THREAD_SELECT.repostOf` does not include `isRemoved` in the select. Moderation takedowns are partially bypassed. | S — Add `isRemoved` to repostOf select and null it in the response if true |
| 6 | M | majlis-lists.service.ts:322-325 | `removeMember()` uses `{ decrement: 1 }` without `GREATEST(0, ...)` protection. | 7 | M | majlis-lists.service.ts:265-279 | `addMember()` checks target user exists but does NOT verify `isDeactivated: false, isBanned: false, isDeleted: false`. | 8 | M | majlis-lists.service.ts:274-289 | `addMember()` existence check (L274) is outside the transaction (L282). The `ConflictException` at L278 prevents the second request from reaching the transaction in sequential flow, but concurrent requests can both pass the check simultaneously. | Counter drift: `membersCount` incremented without corresponding member row (P2002 blocks the create but the increment still fires in the batched transaction). | M — Move existence check into an interactive transaction, or catch P2002 and handle idempotently like `like()` does |
| 9 | M | threads.service.ts:555-588 | `createContinuation()` performs 3 non-atomic steps: (1) update parent chain info (L566-569), (2) count chain members (L573), (3) create continuation (L575-585). Concurrent continuation requests can get the same `chainCount`, causing duplicate `chainPosition` values. | Chain ordering corruption: two continuations with same position. Unroll/display renders incorrectly. | M — Wrap in $transaction with serializable isolation or use optimistic locking |
| 10 | M | threads.service.ts:813-829 | `likeReply()` does NOT prevent self-liking (no `reply.userId === userId` check). | 11 | M | threads.service.ts:775-810 | `getReplies()` does not filter `user: { isDeactivated: false, isBanned: false, isDeleted: false }` on reply authors. | 12 | M | threads.service.ts:958-990 | `getUserThreads()` does not check if the target user is `isDeactivated`, `isBanned`, or `isDeleted` after the `findUnique`. | 13 | M | privacy.service.ts:588,591,592 | Account deletion deletes `threadBookmark`, `threadReaction`, and `threadReplyLike` records (L588,591,592) without decrementing the corresponding `thread.bookmarksCount`, `thread.likesCount`, and `threadReply.likesCount` counters. | Counter drift on all threads/replies the deleted user interacted with. | 14 | M | majlis-lists.service.ts:360-365 | `getTimeline()` does not filter thread `visibility`. | 15 | L | schema.prisma:2549-2562 | `MajlisList` has no `@@unique([ownerId, name])` constraint. A user can create multiple lists with identical names. | User confusion with duplicate list names. No way to distinguish lists except by ID. | S — Add @@unique constraint |
| 16 | L | schema.prisma:1308-1309 | No `@@index` on `Thread.repostOfId`.
