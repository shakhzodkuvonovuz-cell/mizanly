# B11 — Report, ModerationAction, BannedHash, FlaggedContent, Appeal

**Scope note:** The Prisma schema has no `BannedHash`, `FlaggedContent`, or `Appeal` models. `ModerationAction` is an enum, not a model. The actual models in scope are `Report` and `ModerationLog` (which contains inline appeal fields). The audit covers these two models and all service code that operates on them.

| 1 | C | `content-safety.service.ts:329` | `moderatorId: 'system'` is set on ModerationLog, but `moderatorId` is a FK to User. No User record with `id = 'system'` exists. | 2 | C | `content-safety.service.ts:322-325` | `targetField` mapping for `'reel'` and `'thread'` both resolve to `'targetPostId'`. ModerationLog has no `targetReelId` or `targetThreadId` column. A reel ID or thread ID gets stored in `targetPostId`, which is a FK to the Post table. Admins reviewing logs see wrong content. | 8 | H | `moderation.service.ts:242` | `review()` only accepts reports with status `PENDING`, rejecting `REVIEWING`. Meanwhile `reports.service.ts:251` accepts both `PENDING` and `REVIEWING`. | 9 | M | `schema.prisma:2055-2060` | Report model is missing `@@index` on `reportedPostId`, `reportedCommentId`, and `reportedMessageId`. | 10 | M | `schema.prisma:2037-2039` | `reportedThreadId`, `reportedReelId`, `reportedVideoId` are bare `String?` fields with no Prisma relation. | 12 | M | `moderation.service.ts:422-440` | `getPendingAppeals` uses a manual cursor strategy (`where.id = { lt: cursor }`) that is inconsistent with every other paginated query in the codebase (which all use Prisma's `cursor: { id: cursor }, skip: 1`).
