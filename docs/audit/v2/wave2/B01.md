## Findings

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 1 | H | users.service.ts:280-282 | `getUserPosts()` does not check `isDeleted`, `isBanned`, or `isDeactivated` on the resolved user. A banned/deactivated user's posts are served via direct `GET /users/:username/posts` URL. `getProfile()` blocks these users at line 231, but `getUserPosts()` is a separate endpoint that bypasses that check. | Banned user content leaks through direct endpoint access | S — Add status check after `findUnique` |
| 2 | H | users.service.ts:338-340 | `getUserThreads()` has the same missing check as `getUserPosts()` — no `isDeleted`/`isBanned`/`isDeactivated` filter before returning threads. | Banned user threads leak through direct endpoint access | S — Same fix as #1 |
| 3 | H | users.service.ts:1101-1102 | `findByPhoneNumbers()` filters `isDeleted: false, isBanned: false` but omits `isDeactivated: false`. Deactivated users appear in contact sync results, leaking that the deactivated user has a Mizanly account. | Privacy leak: deactivated accounts discoverable via contact sync | S — Add `isDeactivated: false` to where clause |
| 4 | M | mutes.service.ts:20-24 | `mute()` verifies the target user exists but does NOT check `isDeactivated`, `isBanned`, or `isDeleted`. Users can mute banned/deleted accounts, creating orphan mute records. While harmless for security, it pollutes the muted list with ghost entries that can never be unmuted via the UI (user profile won't load). | Orphan records; confusing UX when muted list shows non-existent users | S — Add status check |
| 5 | M | restricts.service.ts:15 | `restrict()` fetches the full User object (`findUnique` without `select`) and does NOT check `isDeactivated`, `isBanned`, or `isDeleted`. Same orphan record issue as mutes, plus inefficient query (fetches all ~200+ columns instead of just `id`). | Orphan records; unnecessary full-row read | S — Add select + status checks |
| 6 | M | restricts.service.ts:107-110 | `getRestrictedIds()` is hard-capped at `take: 50`. If a user restricts more than 50 accounts, feed filtering silently ignores the overflow. Content from restricted users #51+ appears in feeds. | Feed filtering incomplete for power users; restricted content leaks into feed | S — Increase cap or paginate |
| 7 | M | schema.prisma:3371-3381 | `Restrict` model has `@@index([restrictedId])` but no `@@index([restricterId])`. Every call to `getRestrictedList()` and `getRestrictedIds()` queries by `restricterId`, requiring a full sequential scan of the `restricts` table. | Slow query performance for restrict-heavy users | S — Add `@@index([restricterId])` |
| 8 | M | schema.prisma:679 | `User.previousUsername` has no index, but `users.service.ts:211` uses `findFirst({ where: { previousUsername: username } })` for username redirect lookup. This is a sequential scan of the entire `users` table on every profile lookup that misses the primary username. | Full table scan on every username miss; latency degrades with user count | S — Add `@@index([previousUsername])` |
| 9 | M | users.service.ts:711-714 | `snapshotFollowerCounts()` cron is capped at `take: 5000` with no pagination/offset. For a platform with >5000 users with followers, only the first 5000 (by default Prisma order, likely by ID) are ever snapshotted. Later-registered users never get creator stats. | Missing analytics data for users registered after the first 5000 | M — Add cursor-based pagination loop |
| 10 | M | users.service.ts:711-712 | `snapshotFollowerCounts()` filters `isDeleted: false, isBanned: false` but omits `isDeactivated: false`. Deactivated accounts still get follower count snapshots, wasting DB writes. | Wasted I/O on accounts that cannot be viewed | S — Add `isDeactivated: false` |
| 11 | M | settings.service.ts:24-40 | `updatePrivacy()` updates `UserSettings` and `User.isPrivate` in two separate queries (lines 28-32 and 35-39) without a `$transaction`. If the second query fails (e.g., DB timeout), `UserSettings` is updated but `User.isPrivate` is not, leaving inconsistent state. | Privacy setting desync between UserSettings and User model | S — Wrap in `$transaction` |
| 12 | L | follows.service.ts:227-253 | `getFollowers()` returns follower user data without filtering out `isDeleted`, `isBanned`, or `isDeactivated` followers. Banned accounts appear in a user's followers list. The `include.follower.select` doesn't include status fields, so there's no way to filter post-query either. | Banned/deleted ghosts in followers list | S — Add where filter on follower relation or add status fields to select |
| 13 | L | follows.service.ts:278-314 | `getFollowing()` has the same issue as `getFollowers()` — no status filtering on the included `following` relation. Banned/deleted users appear in the following list. | Banned/deleted ghosts in following list | S — Same fix as #12 |
| 14 | L | blocks.service.ts:27-31 | `block()` does not check `isDeleted`, `isBanned`, or `isDeactivated` on the target before creating a block record. Users can block already-deleted accounts, creating orphan block records. | Orphan block records; minor DB clutter | S — Add status check |
| 15 | L | users.service.ts:985-1016 | `getLikedPosts()` does not filter by `post.isRemoved: false` (unlike `getSavedPosts` at line 397 which does). Removed posts appear in the liked posts list. | Removed/moderated posts visible in liked list | S — Add `post: { isRemoved: false }` filter |
| 16 | L | update-profile.dto.ts:26 vs schema.prisma:681 | `UpdateProfileDto.bio` has `@MaxLength(160)` but the Prisma schema defines `bio` as `@db.VarChar(500)`. The DTO is stricter than the DB constraint. Users who had bios set via other paths (admin, migration) up to 500 chars would be forced to truncate on next profile update. Not a bug per se, but a design mismatch that could cause confusion. | Inconsistent max length between DTO and schema | S — Align DTO to 500 or schema to 160 |
| 17 | L | devices.service.ts:99-112 | `touchSession()` uses `update` by `id` with no `userId` check. If an attacker discovers a valid device ID (CUIDs are not secret), they could update `lastActiveAt` and `ipAddress` for another user's device session. The device ID is client-provided (passed from middleware). | Minor data tampering: last-active timestamp and IP spoofing on another user's device record | M — Add `userId` to the where clause or use `updateMany` with both `id` and `userId` |
| 18 | L | users.service.ts:863-901 | `queryFollowers()` and `queryFollowing()` (private helpers) accept a `viewerId` parameter but never use it. The parameter is dead code carried from the public `resolveUsernameToUserId` call. | Dead parameter; confusing API contract | S — Remove unused parameter |
| 19 | I | blocks.service.ts:233-242 | `getBlockedIds()` has a hard cap of `take: 1000`. While 1000 blocks is unusual, it's not impossible for public figures dealing with harassment. Block entries beyond 1000 are silently ignored in feed filtering. | Theoretical feed filtering gap for extreme block counts | M — Document the limit or paginate |
| 20 | I | mutes.service.ts:1-79 | `MutesService` lacks a `getMutedIds()` bulk helper. Feed services query mutes directly via `prisma.mute.findMany`. This works but means every feed consumer reimplements the same query instead of using a shared method. | Code duplication across feed services; inconsistent caps/filtering | S — Add `getMutedIds()` helper |
| 21 | I | users.service.ts:281 | `getUserPosts()` fetches the entire User row (`findUnique` without `select`) just to get the `id`. Should use `select: { id: true, isDeleted: true, isBanned: true, isDeactivated: true }` for efficiency. Same issue in `getUserThreads()` at line 339. | Unnecessary full-row reads | S — Add select clause |
