## Summary

The payments/monetization/gifts/commerce modules total 3,611 lines of production code across 14 files. The code structure is generally well-organized with proper auth guards, DTOs with class-validator, Stripe signature verification, and webhook idempotency.

**Critical issues** center on money operations: fire-and-forget DB writes for payment mappings (#1), ambiguous tip-to-PaymentIntent matching (#2), and a race-vulnerable idempotency check for coin crediting (#3). These can result in lost money, wrong recipient, or double-crediting.

**High issues** include 6 endpoints that throw immediately (dead code), making tips, cashout, zakat, treasury, and waqf permanently non-functional. The `cancelSubscription` endpoint lacks input validation entirely (#4). No module checks `isBanned`/`isDeactivated` on financial counterparties (#12). The `subscribe` method in monetization creates DB subscriptions without payment (#21).

**Pattern-level concern:** The codebase has two parallel payment paths — `PaymentsService` (Stripe-integrated) and `MonetizationService.sendTip` (permanently disabled). This creates confusion about which is the active code path. The monetization service should either delegate to the payments service or be clearly marked as deprecated.
