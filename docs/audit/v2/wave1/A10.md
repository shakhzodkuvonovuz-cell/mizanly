## Cross-Module Analysis

### Block/Mute/Restrict Enforcement in Content Queries
The shared utility `apps/api/src/common/utils/excluded-users.ts` properly aggregates blocks (bidirectional), mutes, and restricts into an exclusion list cached in Redis for 60s. Feed services use this utility. Posts service does inline block checks on individual post views. The enforcement is reasonable but:
- The `BlocksService.getBlockedIds()` has a lower cap (1000) than the shared utility (10000).
- The `RestrictsService.getRestrictedIds()` has a severely low cap (50) vs shared utility (10000).
- Neither `getBlockedIds` nor `getRestrictedIds` is actually used by other services (only `excluded-users.ts` is used). These methods are effectively dead code or intended for future use.

### Moderation Action Reversibility
- **Content removal** is reversible via appeal (`resolveAppeal` restores `isRemoved: false`), but NOT via report dismissal (finding #2).
- **Bans** are reversible via appeal but the reversal is incomplete (finding #3: `isDeactivated` not cleared, Clerk not unblocked).
- **Warnings** (`warningsCount` increment) are NOT reversible via any mechanism.

### Audit Trail Completeness
- Report resolution creates a `moderationLog` entry (good).
- Report dismissal does NOT create a `moderationLog` entry (gap: no record of dismissed reports for audit purposes).
- Auto-flagged content uses `reporterId: null` (good fix from previous audit finding #28).
- `ModerationService.review()` skips `moderationLog` creation for "approve" (dismiss) actions — same gap as report dismissal.
