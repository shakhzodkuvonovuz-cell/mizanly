| 1 | C | live.service.ts:230,241,268 | Raw SQL uses `"LiveSession"` (Prisma model name) but `@@map("live_sessions")` means the actual PostgreSQL table is `live_sessions`. All three `$executeRaw` statements for viewer count increment/decrement will throw `relation "LiveSession" does not exist` at runtime. | Join, leave, and re-join operations all fail with 500 errors. currentViewers, totalViews, peakViewers never update. | 5 min — change `"LiveSession"` to `live_sessions` in all 3 raw SQL statements |
| 2 | C | audio-tracks.service.ts:68-78 | `delete()` accepts a `userId` param but never checks ownership. The `AudioTrack` model has no `userId` field at all (schema line 2237-2251). Data loss. | 30 min — add `userId` FK to AudioTrack model + migration + ownership check in delete |
| 3 | C | stream.service.ts:247 | Raw SQL uses `"Channel"` but `@@map("channels")` means the actual table is `channels`. | 4 | H | live.service.ts:40-54 | `create()` uses `include: { host: ... }` which returns ALL LiveSession fields including `streamKey`. The `streamKey` is the RTMPS ingest credential. Similarly, `endLive` (line 182), `cancelLive` (line 193), `startRehearsal` (line 393), `goLiveFromRehearsal` (line 419), `endRehearsal` (line 438), `setSubscribersOnly` (line 451), and `updateRecording` (line 311) all return full session objects without `select`, leaking `streamKey` in every response. | Stream key credential leaks in 8 different API responses. If any response is cached or logged by a proxy, the key is compromised. | 20 min — use `select` (matching `LIVE_SESSION_LIST_SELECT`) on all mutations that don't need to return streamKey, or create a separate HOST_SELECT that includes it only for create/start |
| 5 | H | broadcast.service.ts:156-183 | `sendMessage()` notification fan-out fetches up to 10,000 subscribers in a single query (line 159: `take: 10000`), then creates notifications in batches of 500. For a popular channel this is an unbounded memory allocation and blocks the event loop during batch creation. The `.then().catch()` fire-and-forget means any partial failure is silently lost. | At scale (10K+ subscribers), this will cause memory spikes, slow responses, and potential OOM. Notification delivery is unreliable — partial failures are silently swallowed with `.catch(() => {})` at line 175. | 1-2 hours — move to a background job queue (Bull/BullMQ). For now, reduce take to 1000 and add proper error logging |
