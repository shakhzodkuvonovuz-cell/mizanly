| 1 | H | feed.service.ts:465-474 | `featurePost` admin check bypassed when `userId` is undefined. The guard `if (userId)` means if the service method is ever called internally without a userId (signature is `userId?: string`), the admin role check is skipped entirely. The controller does use `ClerkAuthGuard` so external calls are safe, but the optional parameter is a latent auth bypass for internal callers. | Any internal service or future endpoint calling `featurePost(postId, true)` without userId will bypass admin check and feature arbitrary posts. | 10 min — make `userId` required (remove `?`) or always throw if missing |
| 2 | H | feed.controller.ts:228 | `featurePost` admin mutation endpoint has no `@Throttle()` decorator. Every other mutation endpoint in this controller has rate limiting. An attacker with admin credentials (or a compromised admin account) could feature/unfeature posts in rapid succession. | Unbounded admin mutation rate enables abuse if admin credentials are compromised. | 2 min — add `@Throttle({ default: { ttl: 60000, limit: 10 } })` |
| 3 | H | feed.service.ts:558-565 | `getNearbyContent` is missing `visibility: 'PUBLIC'` filter. Private and followers-only posts with a location name will appear in the nearby feed to any user (including unauthenticated). | Private posts with location names leak to all users via the nearby endpoint. | 5 min — add `visibility: PostVisibility.PUBLIC` to where clause |
| 4 | H | feed.service.ts:544-592 | `getNearbyContent` does not apply block/mute exclusion when `userId` is provided. Authenticated users will see content from people they have blocked or muted in the nearby feed. | Block/mute privacy contract is broken for location-based discovery. | 10 min — call `getExcludedUserIds(userId)` and add to user filter |
| 5 | H | feed.service.ts:93-106 & all feed queries | `getDismissedIds` is defined (line 103) and referenced in a comment (line 95) but NEVER called from any feed query. The dismiss/undismiss endpoints record dismissals to the DB, but getTrendingFeed, getFeaturedFeed, getNearbyContent, getCommunityTrending, and the personalized feed never exclude dismissed content. Users dismiss content but keep seeing it. | Dismiss feature is completely non-functional — users see content they explicitly dismissed. | 30 min — integrate `getDismissedIds()` into all feed query paths |
| 6 | M | feed.controller.ts:172 | `trackSessionSignal` does not `await` the async service call. Fire-and-forget means: (a) if Redis write fails, the error is an unhandled promise rejection (crashes in strict mode); (b) caller always receives `{ success: true }` even on failure; (c) violates CLAUDE.md rule "All errors explicit, never silently swallowed." | Silent error swallowing; potential unhandled rejection crash; misleading success response. | 2 min — add `await` |
| 7 | M | polls.service.ts:158-193 | `retractVote` uses `findFirst` which only finds ONE vote. The user cannot choose which vote to retract. | Multi-choice poll vote retraction is non-deterministic — user has no control over which vote is removed. | 20 min — add optional `optionId` query param to retractVote endpoint; if provided, delete that specific vote; if not, delete all votes (or require it for multi-choice) |
| 8 | M | polls.service.ts:196-247 | `getVoters` does not filter out banned/deactivated/deleted users. | 9 | M | promotions.service.ts:175 | `markBranded` injects `partnerName` into post content via string concatenation: `[Paid partnership with ${partnerName}]`. No sanitization of brackets. | Content corruption when partner name contains `]` character. | 10 | M | promotions.controller.ts:22 vs promotions.service.ts:23 | DTO `BoostPostDto.duration` allows `@Max(720)` but service `MAX_DURATION_DAYS = 30`. The service interprets `duration` as days (`duration * 24 * 60 * 60 * 1000`). DTO allows up to 720 which the service rejects at runtime. The unit is ambiguous — DTO @Max suggests hours (720h = 30d) but no documentation clarifies this. Values 31-720 pass DTO validation but fail service validation, producing confusing error messages. | Confusing developer experience; DTO and service disagree on max value; unit ambiguity could lead to bugs if either side changes independently. | 5 min — align DTO `@Max(30)` with service `MAX_DURATION_DAYS = 30` and add a comment clarifying the unit is days |
| 11 | M | personalized-feed.service.ts:601-617 | `minbar` (video) trending feed does not filter by `scheduledAt`. The Video model has a `scheduledAt` field (schema line 1466).
