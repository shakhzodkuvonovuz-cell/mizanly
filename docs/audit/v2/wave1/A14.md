## Summary of Critical Checks

**Auth guards:** All mutation endpoints correctly use `@UseGuards(ClerkAuthGuard)`. Read-only public endpoints use `OptionalClerkAuthGuard`. `@CurrentUser('id')` is used correctly with the `'id'` argument everywhere.

**SQL injection:** No `$queryRawUnsafe` or `$executeRawUnsafe` found. All raw SQL uses tagged template literals (`$queryRaw` / `$executeRaw`) which are safe.

**Islamic data integrity:** Quran metadata (surah names, ayah counts, revelation types) is stored in static TypeScript files, not AI-generated. Hadiths, duas, 99 Names, tafsir, and hajj guide are loaded from curated JSON data files. No endpoint generates Islamic content -- all endpoints serve pre-curated data or fetch from known Islamic APIs (Aladhan for prayer times, Quran.com for verse text). The prayer calculator implements standard astronomical formulas from praytimes.org. Scholar verification requires document submission and manual approval -- no auto-verification.

**Rate limiting:** All mutation endpoints have `@Throttle()` decorators. The class-level throttle on `IslamicController` (30/min) covers read endpoints, with tighter limits on create operations (5/min for campaigns, plans, challenges; 1/day for scholar verification).

**Response leaks:** No passwords, tokens, or full user objects are leaked. The mosque members endpoint returns raw membership records (finding 7) but these contain only IDs and roles, not sensitive user data.
