| 1 | C | messages.service.ts:1640,1702 | **Role casing mismatch causes authorization bypass.** `createGroupTopic` (line 1640) and `setMessageExpiry` (line 1702) compare `member.role !== 'ADMIN'` (uppercase). The Prisma schema defaults role to `"member"` (lowercase, varchar(10)). Roles stored as lowercase `admin`/`member`/`owner`. The uppercase `'ADMIN'` comparison will NEVER match, meaning admins are silently denied and only the creator (`createdById`) can create topics or set expiry. If `createdById` is null or the owner left, the feature is permanently locked. | Group admins cannot create topics or set message expiry despite having admin role. Authorization logic is broken. | 5 min — change `'ADMIN'` to `'admin'` on both lines |
| 2 | C | messages.service.ts:1714 | **Unit mismatch: `setMessageExpiry` stores minutes, consumed as seconds.** `disappearingDuration` is stored as `expiryDays * 24 * 60` (= minutes). But lines 249 and 424 consume it as `disappearingDuration * 1000` (treating the value as seconds). Result: setting "1 day" expiry stores 1440, consumed as 1440 seconds = 24 minutes. Setting "90 days" gives 90 * 24 * 60 = 129600, consumed as 129600 seconds = 36 hours. Messages expire far sooner than expected. Should be `expiryDays * 24 * 60 * 60`. | Messages auto-delete dramatically earlier than the user selected. Data loss. | 5 min — add `* 60` to line 1714 |
| 3 | H | messages.service.ts:842 | **Redis TTL re-seed is always 1 second (dead code).** `const ttl = Math.max(1, Math.floor((new Date().getTime() - Date.now()) / 1000))` computes `now - now = 0`, so `Math.max(1, 0) = 1`. The intent was `inviteExpiresAt.getTime() - Date.now()` but `inviteExpiresAt` is never used. The Redis re-seed always expires in 1 second, making the DB fallback path hit on every subsequent join attempt. | Unnecessary DB query on every invite join after the first Redis miss. Not a data loss but a performance regression and incorrect caching behavior. | 5 min — use `convo.inviteExpiresAt` instead of `new Date()` |
| 4 | H | messages.controller.ts:632,694,752 | **Three endpoints use bare `@Body('field')` without DTO class-validator validation.** `pinConversation` takes `@Body('isPinned') isPinned: boolean`, `changeGroupRole` takes `@Body('role') role: string`, and `setMessageExpiry` takes `@Body('expiryDays') expiryDays: number`. NestJS `ValidationPipe` only runs class-validator decorators on DTO classes, not on bare `@Body('field')` extractions. These values pass through unvalidated at the framework level (the controller has manual checks for `role` but not for `isPinned` type or `expiryDays` type). | `isPinned` could be any truthy/falsy JS value. `expiryDays` could be a string. While service-level checks partially mitigate, the validation is inconsistent with the rest of the codebase pattern. | 15 min — create proper DTO classes with decorators |
| 5 | H | messages.controller.ts:731-734 | **`createGroupTopic` uses inline `@Body() body: { name: string; iconEmoji?: string }` — no DTO validation.** The body type is a TypeScript interface (erased at runtime).
