# F06 — Crypto Audit: Sender Keys + Sealed Sender

**Auditor:** Claude Opus 4.6 (1M context)
**Date:** 2026-03-30
**Attacker model:** NSA TAO / compromised Mizanly server / nation-state MITM
**Scope:** `apps/mobile/src/services/signal/sender-keys.ts`, `apps/mobile/src/services/signal/sealed-sender.ts`

---

## Summary

Sender Keys implementation is solid for the Signal-spec model: HMAC symmetric ratchet, Ed25519 message authentication, AEAD encryption with domain-separated AAD, signing key isolation in SecureStore, out-of-order delivery via skipped keys, and per-sender dedup. Sealed Sender has had significant hardening across 8+ audit rounds (V4-V8) and now includes sender certificates, counter persistence, type-safe validation, and legacy rejection. However, several findings remain.

**Findings: 2 HIGH, 4 MEDIUM, 3 LOW = 9 total**

## Detailed Analysis

### F06-1 [HIGH]: Sealed sender counter is a global singleton; concurrent `sealMessage` calls race

**Location:** `sealed-sender.ts` lines 81, 137-155

**Problem:** `sealedSenderCounter` is a module-level `let` variable. The `sealMessage` function is `async` (it calls `secureStore` for persistence), but there is no mutex/lock around the counter increment + persist sequence. If two `sealMessage` calls execute concurrently (e.g., user sends two messages rapidly, or one group message triggers multiple sealed envelopes):

1. Call A reads `sealedSenderCounter = 5`, increments to 6
2. Call B reads `sealedSenderCounter = 6` (in-memory already incremented by A), increments to 7
3. Call A persists `6` to MMKV
4. Call B persists `7` to MMKV
5. Both envelopes sent with different counters — but if the `await secureStore` for A fails AFTER B already incremented, the revert `sealedSenderCounter--` reverts B's increment, not A's.

More critically: the sender key module uses `withSessionLock` for exactly this reason, but sealed sender has NO lock. The increment-then-persist-then-maybe-revert pattern is not atomic.

**Impact:** Counter desync between in-memory and persisted state. In the worst case, after an app crash between increment and persist, the counter reloads to a stale value. The recipient rejects the next message as a replay (counter <= lastSeen). This is a DoS vector: an attacker who can trigger MMKV write failures (e.g., disk full, encryption key corruption) causes permanent message send failure until the user clears E2E state.

**Recommendation:** Wrap `sealMessage` in a `withSessionLock('sealed_sender_counter')` or equivalent mutex, similar to how `encryptGroupMessage` protects its chain state.

### F06-2 [HIGH]: Sender key skipped message keys never zeroed from memory when persisted to MMKV

**Location:** `sender-keys.ts` lines 314, 380-384, 419-421

**Problem:** When out-of-order messages arrive, the decrypt path stores intermediate message keys in `newSkipped` (line 358) and carries them in `state.skippedKeys` (line 421). These `Uint8Array` message keys are:

1. Stored in the `allSkipped` array (line 380)
2. Persisted to MMKV via `storeSenderKeyState` (line 422) — which calls `toBase64()` on each key
3. But the in-memory `Uint8Array` copies are **never zeroed** after persistence

The `state.skippedKeys` array (and its constituent `messageKey` Uint8Arrays) lives on the JS heap until garbage collection. A memory dump during this window reveals all skipped keys for all active sender key sessions.

When a skipped key IS used (line 314), the individual key is zeroed at line 336: `zeroOut(skipped.messageKey)`. But all the **unused** skipped keys that remain in the array are never zeroed — they persist in memory across the entire session lifetime.

Additionally, when keys are evicted from the cap at line 382, only the evicted key is zeroed (`zeroOut(evicted.messageKey)`), which is correct. But the remaining keys in `allSkipped` are assigned directly to `state.skippedKeys` (line 421) without any zeroing of the previous `state.skippedKeys` entries' message keys (which were cloned into `allSkipped` via spread).

**Impact:** At a MAX_SENDER_KEY_SKIP of 200, up to 200 x 32-byte symmetric keys per sender per group persist on the JS heap. For a user in 10 groups with 10 active senders each, that is up to 640KB of unzeroed key material.

**Recommendation:** After `storeSenderKeyState`, zero all message keys in the `allSkipped` array that were NOT the target message key. Consider a cleanup function that walks `state.skippedKeys` and zeroes all message keys before reassignment.

### F06-3 [MEDIUM]: Sealed sender replay window uses sender-asserted clock, not monotonic-only

**Location:** `sealed-sender.ts` lines 328-333

**Problem:** The replay protection has two layers: a timestamp check (`age > 5 minutes` or `age < -60 seconds`) and a monotonic counter check. The timestamp uses `Date.now()` from both the sender (who sets `ts`) and the recipient (who checks `Date.now() - ts`).

The issue is the 5-minute window combined with the -60 second future tolerance. An attacker who compromises the Mizanly server can:

1. Buffer a sealed envelope for up to 5 minutes
2. Replay it to the recipient with the SAME counter — blocked by monotonic counter check
3. BUT: if the attacker can also manipulate the recipient's NTP time (nation-state capability on mobile networks), they can make the timestamp check pass for envelopes outside the window

The 60-second future tolerance exists for clock skew, but it means an attacker who can advance the recipient's clock by 60 seconds gets 6 minutes of replay window instead of 5.

More importantly, the `ts` and `ctr` fields are both **sender-asserted**. While the sender certificate (V7-F2) prevents third-party forgery, a compromised sender can set `ts` to any value within the window. The counter provides the real protection, not the timestamp.

**Impact:** The timestamp check provides marginal value over the counter alone. It protects against a compromised sender replaying their OWN old messages (where the counter advanced past), but only within the 5-minute window. Given the attacker model includes a compromised server, the timestamp adds confusion about what is actually providing replay protection.

**Recommendation:** Document explicitly that the monotonic counter is the PRIMARY replay defense and the timestamp is defense-in-depth for clock-skew edge cases only. Consider removing the 60-second future tolerance (reject all future-dated envelopes).

### F06-4 [MEDIUM]: Sender key distribution retry closure captures `encryptForMember` callback indefinitely

**Location:** `sender-keys.ts` lines 539-566

**Problem:** The 30-second retry `setTimeout` at line 544 captures `encryptForMember` and `uploadToServer` callbacks by closure. The code comment at line 542 acknowledges this: "This is acceptable — they hold no key material, only session encrypt/upload capability."

However, `encryptForMember` is a callback that encrypts via pairwise Double Ratchet sessions. If the pairwise session is rekeyed or rotated between the initial distribution attempt and the 30-second retry, the retry may:
1. Use a stale session state (if the callback captured a reference to session state)
2. Advance the ratchet unexpectedly (if the callback uses the live session)
3. More critically: if the user leaves the group within the 30 seconds, the retry still fires and attempts distribution to the original member list

The closure also keeps the `retryMemberIds` array alive in memory for 30 seconds unnecessarily.

**Impact:** If the user leaves or is removed from the group within 30 seconds, the retry still distributes the sender key to members who should no longer receive it. This is a forward-secrecy violation: the removed user's client may have the sender key for the new generation.

**Recommendation:** Add a guard at the start of the retry callback that checks whether the sender key state still exists and the generation matches before attempting redistribution. Alternatively, use a cancellable timer that is cleared on group leave/key rotation.

### F06-5 [MEDIUM]: Sender key chain key not zeroed in loaded state object after distribution serialization

**Location:** `sender-keys.ts` lines 510-516

**Problem:** In `distributeSenderKeyToMembers`, the function loads sender key state at line 510, serializes it at line 515, then iterates over members. The `serialized` bytes are correctly zeroed at line 537 after the loop. However, the `state` object loaded at line 510 contains the raw `chainKey` (32-byte `Uint8Array`), and this is **never zeroed** after the function completes. The `state` variable goes out of scope at function return but the `Uint8Array` persists on the heap until GC.

Similarly, in the retry closure (lines 549-562), `freshState` is loaded and `freshSerialized` is zeroed in the `finally` block, but `freshState.chainKey` is never zeroed.

**Impact:** The sender key chain key persists on the JS heap after distribution. Combined with the signing public key (also in `state`), this allows an attacker with a memory dump to advance the chain and decrypt future group messages.

**Recommendation:** Add `zeroOut(state.chainKey)` after the distribution loop completes (before return). Add `zeroOut(freshState.chainKey)` in the retry `finally` block.

### F06-6 [MEDIUM]: Sealed sender counter can overflow Number.MAX_SAFE_INTEGER without detection

**Location:** `sealed-sender.ts` line 146

**Problem:** `sealedSenderCounter` is a JavaScript `number` incremented by 1 on every `sealMessage` call. JavaScript `number` is an IEEE 754 double, with safe integer range up to 2^53 - 1 (9,007,199,254,740,991). After `Number.MAX_SAFE_INTEGER`, incrementing no longer changes the value: `Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2`.

If the counter reaches this value, two consecutive envelopes will have the SAME counter value (both `Number.MAX_SAFE_INTEGER + 1`). The recipient's replay check would reject the second envelope as a replay.

At one message per second, this would take ~285 million years, so it is not a practical concern. However, the counter is persisted as a string via `String(sealedSenderCounter)`. If a bug or corruption sets the counter to a very large value, the overflow behavior is silent.

**Impact:** Theoretical only. No practical exploit at reasonable message rates.

**Recommendation:** Add a safety check: `if (sealedSenderCounter >= Number.MAX_SAFE_INTEGER) throw new Error('Sealed sender counter exhausted')`. This is defense-in-depth against counter corruption.

### F06-7 [LOW]: Sender key generation returns full signing private key on stack despite SecureStore isolation goal

**Location:** `sender-keys.ts` lines 117-118

**Problem:** `generateSenderKey` stores the signing private key in SecureStore (line 99) and stores only a sentinel in MMKV (line 110). But then at line 118, it returns `{ ...state, signingKeyPair }` — which includes the REAL private key from `signingKeyPair` (from `generateEd25519KeyPair()` at line 91).

The returned object is used by the caller (e.g., `distributeSenderKeyToMembers` via `serializeSenderKeyForDistribution`). The serialization function correctly only includes the PUBLIC key (line 626). But the returned `SenderKeyState` object with the real private key lives on the caller's stack/heap until GC.

The `signingKeyPair.privateKey` from `generateEd25519KeyPair()` is never zeroed in `generateSenderKey`. The function creates a copy of the state with the real key and returns it, but the original `signingKeyPair` from line 91 is also never zeroed.

**Impact:** The signing private key exists in at least 2 locations in memory (original keypair + returned state) after generation, even though the design goal is SecureStore-only storage. This is a defense-in-depth concern, not a direct vulnerability.

**Recommendation:** Zero `signingKeyPair.privateKey` after storing in SecureStore (line 99), and do not include it in the return value. Callers that need the private key should load it from SecureStore via `loadSenderSigningPrivate`.

### F06-8 [LOW]: Group message dedup uses `Array.includes()` for replay check — O(n) linear scan

**Location:** `storage.ts` line 1012

**Problem:** `checkGroupMessageDedup` stores dedup IDs as a JSON array and checks membership with `set.includes(dedupId)`. This is O(n) per check, where n can be up to 500 per sender per group.

With 10 groups of 20 active senders, each inbound message triggers a linear scan of up to 500 strings. This is ~100,000 string comparisons in the worst case before the first message is processed.

More importantly, the dedup strings are formatted as `${chainId}:${counter}`, and the `includes()` call performs string comparison. An attacker sending messages with counter values that produce long string representations (e.g., counter = 999999999) increases comparison cost.

**Impact:** Performance degradation under adversarial conditions. Not a security vulnerability per se, but an attacker who floods dedup entries can slow down message processing on the victim's device.

**Recommendation:** Use a `Set` instead of an array for the in-memory check. The JSON serialization/deserialization can convert between array and Set. For persistence, the array format is fine, but the lookup should be O(1).

### F06-9 [LOW]: Sealed sender `plaintext` not zeroed after parsing in `unsealMessage`

**Location:** `sealed-sender.ts` line 238

**Problem:** After AEAD decryption at line 231, the `plaintext` Uint8Array contains the decrypted inner JSON (which includes `senderId`, `innerContent`, `senderIdentityKey`, etc.). This is parsed into `raw` via `JSON.parse(utf8Decode(plaintext))` at line 238. After parsing, the `plaintext` bytes are never zeroed.

The `plaintext` contains a JSON string with:
- `senderId` — the sender's identity (the whole point of sealed sender is to protect this)
- `senderIdentityKey` — base64 of the sender's Ed25519 public key
- `senderSignature` — base64 of the sender's signature
- `innerContent` — the encrypted message payload

The sender's identity is the most sensitive field in a sealed sender context. Leaving it unzeroed in a `Uint8Array` on the heap defeats the purpose of sealing if the attacker can dump memory.

**Impact:** Sender identity persists in memory after unsealing until GC. A memory dump reveals who sent the sealed message.

**Recommendation:** Add `zeroOut(plaintext)` after `JSON.parse` at line 238.

## Positive Findings (What Is Done Well)

| Area | Assessment |
|------|------------|
| Sender Key symmetric ratchet | Correctly implements HMAC-based chain advancement (0x01 for message key, 0x02 for next chain key). Forward secrecy within the chain. |
| Ed25519 message authentication | Signature over `(counter \|\| ciphertext)` prevents message forgery. Signing key in SecureStore prevents extraction via MMKV compromise alone. |
| AEAD with domain-separated AAD | AAD includes length-prefixed groupId + chainId + generation + counter. Prevents cross-group and cross-generation ciphertext relocation attacks. |
| Sender key distribution security | Only public signing key is distributed. Private key bytes are checked in tests (50 iterations). 0xDE sentinel poisons accidental use. |
| Key rotation on member removal | Generation counter incremented, old chain key zeroed, new key distributed only to remaining members. |
| Sealed sender DH with low-order point check | `assertNonZeroDH` called on both seal and unseal paths. Prevents small-subgroup attacks. |
| Sealed sender certificate (V7-F2) | Ed25519 signature binds senderId to identity key. TOFU verification on unseal. |
| Legacy rejection (V8-F5) | All envelopes without certificates rejected. No sv=0 bypass. |
| Counter type validation (V7-F1) | Rejects NaN, Infinity, negative, non-integer, string, null, object counter/timestamp values. Self-heals previously poisoned NaN state. |
| Counter persist failure handling (V7-F8) | Reverts in-memory counter on persist failure. Prevents memory/storage desync. |
| Session locking (sender keys) | Both encrypt and decrypt use `withSessionLock` per group+sender. Prevents concurrent chain desync. |
| Out-of-order delivery | Skipped key storage with replay protection. 200-key cap per sender. FIFO eviction of oldest keys. |
| Per-sender dedup isolation (V7-F10) | Independent 500-entry cap per sender per group. One attacker's volume cannot evict another sender's dedup entries. |
| Message padding | Both 1:1 and group messages padded to hide length. |
| Key material zeroing | Signing private key zeroed after encrypt (line 206). Old chain key zeroed after advance (line 213). Message/enc/nonce keys zeroed after use. |
