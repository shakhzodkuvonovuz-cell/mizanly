## Summary

The X3DH implementation is structurally sound: all 4 DH computations are present and correctly ordered, ephemeral keys are freshly generated, DH outputs are validated against low-order points via `assertNonZeroDH`, the HKDF derivation follows the Signal spec, and the PQXDH hybrid extension uses single-pass HKDF as specified. However, I found **9 findings** across key material hygiene, missing validation, and a spec deviation on Associated Data.

## Detailed Analysis

### F03-01 [HIGH] — `theirIdentityX25519Public` never zeroed

**Location:** `initiateX3DH` line 178, `respondX3DH` line 352

Both `initiateX3DH` and `respondX3DH` convert the remote party's Ed25519 identity key to an X25519 public key via `edToMontgomeryPub`. This X25519 public key is used in DH2 (initiator) / DH1 (responder) and then abandoned on the stack without zeroing.

```typescript
// Line 178 (initiator) — NEVER ZEROED:
const theirIdentityX25519Public = edToMontgomeryPub(bundle.identityKey);
// ... used in DH2 ...

// Line 266-270: cleanup section zeroes dh1, dh2, dh3, dhConcat, ourIdentityX25519Private
// But theirIdentityX25519Public is missing from this list.
```

The same applies at line 352 in the responder.

**Impact:** Under a memory-dump attacker model (root/jailbreak + memory forensics), the converted X25519 representation of the remote identity key persists on the JS heap until GC collects it. While the *public* identity key is not secret, the *converted* form reveals the exact DH inputs, making DH computation reconstruction easier for a forensic attacker if paired with other leaked material.

**Fix:** Add `zeroOut(theirIdentityX25519Public)` to both cleanup sections (after line 270 and after line 408).

### F03-02 [HIGH] — HKDF `derived` buffers never zeroed in session state creation

**Location:** `createInitiatorSessionState` lines 448-457, `createResponderSessionState` lines 506-515, 521-530

In `createInitiatorSessionState`:
```typescript
const derived = hkdfDeriveSecrets(dhOutput, x3dhResult.sharedSecret, 'MizanlyRatchet', 64);
zeroOut(dhOutput); // dhOutput is zeroed
const rootKey = derived.slice(0, 32);       // slice() creates a COPY
const sendingChainKey = derived.slice(32, 64); // slice() creates a COPY
// `derived` (64 bytes) is NEVER zeroed — contains rootKey + chainKey in contiguous memory
```

`Uint8Array.prototype.slice()` in JavaScript creates a **new copy** (unlike `subarray()` which creates a view). The original 64-byte `derived` buffer still contains the full `rootKey || chainKey` in contiguous memory. It is never zeroed and will persist until V8 GC collects it.

The same issue exists in `createResponderSessionState` with both `derivedReceive` (64 bytes) and `derivedSend` (64 bytes) — **192 bytes** of root + chain key material left unzeroed.

**Impact:** A heap memory dump captures root keys and chain keys in a convenient contiguous 64-byte layout, enabling full session decryption.

**Fix:** Add `zeroOut(derived)` after slicing in all three locations. Alternatively, use `subarray()` instead of `slice()` to avoid the copy entirely (but then the original buffer becomes the source of truth and must be retained).

### F03-03 [HIGH] — `x3dhResult.sharedSecret` and `signedPreKeyPrivate` never zeroed by callers

**Location:** `session.ts` lines 68-70 (initiator), 145-162 (responder)

After `initiateX3DH` returns, the `x3dhResult` object contains `sharedSecret` (32 bytes). After being consumed by `createInitiatorSessionState` (used as HKDF salt), the shared secret has no further use. But `createInitiatorSession` in `session.ts` never calls `zeroOut(x3dhResult.sharedSecret)`.

Similarly, the responder path: `x3dhResult.sharedSecret` and `x3dhResult.signedPreKeyPrivate` are consumed by `createResponderSessionState` but never zeroed.

The `X3DHInitResult` interface also holds `ephemeralKeyPair` (with `privateKey`), which is passed into the session state as a reference (not copied) — so zeroing it would corrupt the session. But `sharedSecret` is only used as HKDF salt and should be zeroed immediately after.

**Impact:** The 32-byte X3DH shared secret (the "master secret" from which all session keys derive) persists on the JS heap indefinitely. Under the attacker model, this is the single most valuable target — recovering it allows computing all root keys and chain keys for the entire session.

**Fix:** In `session.ts`, after `createInitiatorSessionState` / `createResponderSessionState`, add:
```typescript
zeroOut(x3dhResult.sharedSecret);
// Responder also:
zeroOut(x3dhResult.signedPreKeyPrivate);
```

### F03-04 [MEDIUM] — Missing `assertNonZeroDH` in session state creation functions

**Location:** `createInitiatorSessionState` line 444, `createResponderSessionState` lines 505, 520

Three DH operations in the session state creation functions lack `assertNonZeroDH`:

1. **Initiator** (line 444): `x25519DH(ephemeralKeyPair.privateKey, remoteSignedPreKey)` — same key pair used in DH3 during X3DH, so likely safe, but defense-in-depth requires checking every DH output.

2. **Responder step 1** (line 505): `x25519DH(signedPreKeyPrivate, senderEphemeralKey)` — mirrors the initiator's initial KDF_RK step. If `senderEphemeralKey` is a low-order point (supplied by a malicious initiator), this DH produces all-zeros, making the receiving chain key predictable.

3. **Responder step 2** (line 520): `x25519DH(senderRatchetKeyPair.privateKey, senderEphemeralKey)` — same risk as above.

While `@noble/curves` X25519 implementation clamps keys and rejects some low-order inputs at the library level, this is not guaranteed across all backends (native-crypto-adapter, future crypto providers). The codebase's own design principle states: "Must be called after every x25519DH() in the protocol" (crypto.ts line 113-114).

**Impact:** If the `@noble/curves` rejection is ever bypassed (native backend, library update, polyfill), a malicious party could inject a low-order public key in the PreKeySignalMessage ephemeral field, producing a predictable chain key.

**Fix:** Add `assertNonZeroDH(dhOutput, 'initiator-initial-KDF_RK')`, `assertNonZeroDH(dhReceive, 'responder-receive-KDF_RK')`, and `assertNonZeroDH(dhSend, 'responder-send-KDF_RK')`.

### F03-05 [MEDIUM] — SPK age check bypassed when `createdAt` is absent

**Location:** `initiateX3DH` line 152

```typescript
if (bundle.signedPreKey.createdAt) {
  // ... age validation ...
}
```

The `createdAt` field is optional in the `PreKeyBundle` type (line 55 of types.ts: `createdAt?: number`). If a compromised server returns a bundle without `createdAt`, the entire age validation block is skipped. The server could then replay an arbitrarily old SPK (whose private key may have been compromised during a previous breach) with a still-valid signature.

This is exactly the attack scenario described in the comment on line 148-150. The fix (V7-F6) intended to prevent it, but the optional field makes it trivially bypassable.

**Impact:** A compromised server replays a years-old SPK whose private key was exfiltrated. The client trusts it because the Ed25519 signature is still valid and no age check runs. All subsequent messages are decryptable by the attacker holding the old SPK private key.

**Fix:** Either:
1. Make `createdAt` required in the `PreKeyBundle` type and reject bundles without it, OR
2. Treat missing `createdAt` as an error: `if (!bundle.signedPreKey.createdAt) throw new Error('SPK missing createdAt — possible server manipulation.');`

### F03-06 [MEDIUM] — `otpPrivate` loaded in responder never zeroed

**Location:** `respondX3DH` lines 369-379

```typescript
const otpPrivate = await loadOneTimePreKeyPrivate(oneTimePreKeyId);
// ... used in DH4 ...
const dh4 = x25519DH(otpPrivate, senderEphemeralKey);
assertNonZeroDH(dh4, 'responder-DH4');
dhConcat = concat(PADDING, dh1, dh2, dh3, dh4);
zeroOut(dh4);
// NOTE: Do NOT delete OTP private key here.
// otpPrivate is NEVER zeroed — remains on heap
```

The comment correctly notes the OTP private key should not be *deleted from storage* here (it's deleted after session establishment). But the loaded `Uint8Array` containing the private key material is never zeroed. The `dh4` output is zeroed, but `otpPrivate` itself (32 bytes of secret key material) remains on the JS heap.

**Impact:** Under memory forensics, the OTP private key is recoverable from the heap, allowing reconstruction of DH4 and thus the complete X3DH shared secret.

**Fix:** Add `zeroOut(otpPrivate)` after the DH4 computation (the storage copy is unaffected — it's a separate Base64 string in SecureStore).

### F03-07 [LOW] — Signal spec AD construction not implemented

**Location:** Not present in x3dh.ts or the encrypt path

Per the Signal X3DH specification (Section 3.3):

> Alice then calculates an "associated data" byte sequence AD that contains identity information for both parties:
> AD = Encode(IK_A) || Encode(IK_B)

This AD is intended to be used as the associated data in the initial AEAD encryption of the first message. In the current implementation, the Double Ratchet's `ratchetEncrypt` uses only the serialized message header (ratchetKey + counter + previousCounter) as AAD. The identity keys of both parties are NOT included.

The purpose of AD in X3DH is to cryptographically bind the session to both parties' identity keys, preventing an attack where a compromised server substitutes one party's identity key after the DH computations. Without AD, the AEAD tag does not authenticate WHO the message is from/to at the crypto layer (only at the session metadata layer).

**Impact:** LOW because the identity keys are already bound into the shared secret via DH1 and DH2 — an attacker who substitutes an identity key would compute a different shared secret and decryption would fail. The AD provides defense-in-depth, not primary protection. Signal implements it; this implementation omits it.

**Fix:** In `createInitiatorSessionState` / `createResponderSessionState`, compute `AD = concat(localIdentityKey, remoteIdentityKey)` and store it in `SessionState`. Pass it as additional AAD to `ratchetEncrypt`/`ratchetDecrypt` alongside the header bytes.

### F03-08 [LOW] — Unsafe type cast to access `pqPreKey` field

**Location:** `initiateX3DH` line 214

```typescript
const bundlePqPreKey = (bundle as unknown as Record<string, unknown>).pqPreKey;
```

This double-cast (`as unknown as Record<string, unknown>`) bypasses TypeScript's type system entirely. The `PreKeyBundle` type does not include `pqPreKey`, so this is accessing an untyped field. If the server sends an unexpected type (e.g., an object instead of a string or Uint8Array), the subsequent type checks (lines 217-223) catch it, but:

1. The cast violates the project rule "No `any` in non-test code" (the `Record<string, unknown>` cast is effectively equivalent to `any` for property access).
2. It makes the PQXDH extension invisible to TypeScript's type checker — no compilation error if the field name changes.

**Fix:** Extend the `PreKeyBundle` interface to include the optional PQXDH fields:
```typescript
interface PreKeyBundle {
  // ... existing fields ...
  pqPreKey?: Uint8Array | string;
  pqPreKeySignature?: string;
  pqPreKeyId?: number;
}
```

### F03-09 [INFO] — Custom HKDF info string instead of spec-compliant value

**Location:** Line 57

```typescript
const X3DH_INFO = 'MizanlySignal';
```

The Signal X3DH specification uses `"WhisperText"` as the info string. Using a custom string (`"MizanlySignal"`) is not a security vulnerability (HKDF info provides domain separation, and any unique string works), but it means this implementation is not wire-compatible with any other Signal-protocol implementation. This is intentional (the system is self-contained), but worth noting for future interoperability considerations.

Similarly, the ratchet info string is `"MizanlyRatchet"` instead of Signal's values. Both are consistent within the codebase.

**Impact:** None for security. Prevents accidental interop with other Signal implementations (which could be a feature, not a bug).
