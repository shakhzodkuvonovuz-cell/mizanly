## What is correct (not exhaustive)

- KDF_RK follows Signal spec: HKDF(dhOutput, rootKey, info, 64) splitting into rootKey + chainKey
- KDF_CK follows Signal spec: HMAC(chainKey, 0x01) = messageKey, HMAC(chainKey, 0x02) = nextChainKey
- DH ratchet step correctly generates new keypair, derives both receiving and sending chains, zeros DH outputs and old private key
- Small-subgroup protection: `assertNonZeroDH` called on both DH outputs in ratchet step, checking all 7 low-order X25519 points
- Counter overflow guard at 2^32 prevents header serialization wrap-around
- MAX_SKIPPED_KEYS=2000 enforced with hard cap of 200 (defense-in-depth)
- Skipped keys deleted BEFORE decryption attempt (replay protection even on AEAD failure)
- Session state cloned before `ratchetDecrypt` in `decryptMessage` (AEAD failure safety)
- Message padding hides length (PKCS#7-style, min 160 bytes, 16-byte alignment)
- Constant-time padding validation (XOR accumulator, not early-exit)
- Serialized session lock (`withSessionLock`) prevents concurrent encrypt/decrypt on same session
- Header authenticated as AAD in AEAD (tampering detected)
- XChaCha20-Poly1305 with HKDF-derived deterministic nonces (safe because each message key is unique)
- Previous sessions kept for simultaneous initiation resolution (capped at 5)
- MMKV serialization uses Base64 for Uint8Array fields with AEAD integrity wrapper
- `Uint8Array.slice()` used (not `subarray`) so derived keys are independent copies
