## Summary

The PQXDH implementation is **non-functional end-to-end**. It consists of:
- Interface definitions and wrapper functions (pqxdh.ts) that are structurally correct
- Integration points in x3dh.ts that are unreachable because no PQ prekeys are ever generated, uploaded, or served
- A Go server that hardcodes `SupportedVersions: [1]` with no PQ schema support
- An API adapter (e2eApi.ts) that drops PQ fields from server responses
- Dead code (`deriveHybridSecret`, `negotiatePQVersion`) with misleading documentation

Even if all infrastructure were in place, two CRITICAL flaws would undermine the implementation:
1. **PQ prekey signature is never verified** -- server/MITM can substitute PQ keys, defeating post-quantum protection entirely
2. **Dead hybrid derivation function** uses a different HKDF info string than the actual protocol path, creating a latent interop trap

The feature is correctly described in CLAUDE.md as requiring "version negotiation [1, 2]" but the presentation alongside the list of "What's built" implies it is operational. It is not -- it is stub infrastructure with zero functional paths.

**Minimum viable path to real PQXDH:** (1) Install @noble/post-quantum, (2) Generate and upload PQ prekeys with Ed25519 signatures, (3) Add PQ fields to Go server schema/endpoints, (4) Pass PQ fields through the API adapter, (5) Verify PQ prekey signature in initiateX3DH, (6) Abort on PQ failure when both parties advertise v2, (7) Zero PQ secret keys after use, (8) Set protocolVersion:2 on PQXDH sessions, (9) Remove dead code.
