## Analysis Notes

### What the code does well
- Constant-time comparison in crypto.ts (V7-F13 fix) with pre-padded arrays is textbook correct
- assertNonZeroDH with the full 7-point low-order check is defense-in-depth done right
- The fromBase64 V7-F12 fix (copying from Buffer pool to standalone array) prevents a real slab-sharing vulnerability
- AEAD key/nonce length validation is present at both encrypt and decrypt entry points
- Padding is PKCS#7-style with MIN_PADDED_SIZE to hide short message lengths -- proper traffic analysis resistance
- zeroOut in constantTimeEqual (V8-F2) prevents padded copies from leaking to GC
- The native adapter auto-detection is synchronous (V6-F9 fix), preventing race conditions at startup

### Attack surfaces in scope
The most exploitable finding is #2 (AEAD fallback timing oracle). A network-level attacker measuring response latency to tampered ciphertext can distinguish "native decrypted and failed tag check, then @noble also failed" (two decrypt attempts, ~2x latency) from "native not available, @noble failed" (one attempt). This is a measurable timing difference even over the network. The fix is straightforward: throw on auth failure, only return null for "feature not available."

Finding #1 (timing side-channel in adapter fallback) is exploitable only when quick-crypto is partially installed (available but missing timingSafeEqual). This is an unlikely but possible configuration, and the impact is length leakage of compared values.

Finding #4 (missing assertNonZeroDH in session init) is mitigated by @noble/curves rejecting low-order points at the library level, but the codebase explicitly documents these checks as defense-in-depth and applies them everywhere EXCEPT these three call sites.
