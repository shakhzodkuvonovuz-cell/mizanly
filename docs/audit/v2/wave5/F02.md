## Summary

The native crypto adapter has one **critical** bug: the HChaCha20 key schedule function is called with wrong argument types (Uint8Array instead of Uint32Array, 2 args instead of 4), causing a silent TypeError that is caught and discarded. This means the entire native AEAD acceleration path has NEVER worked -- every XChaCha20-Poly1305 operation falls back to pure JavaScript `@noble/ciphers`. The fallback is cryptographically correct, so there is no security vulnerability from this bug, but the performance claims in comments and documentation are false.

The `secureZero` function claims to use `OPENSSL_cleanse` but actually uses `randomFillSync + arr.fill(0)`, which is subject to JIT dead-store elimination. The `constantTimeCompare` fallback has a known timing side-channel (nullish coalescing `??` on out-of-bounds access) that was already fixed in `crypto.ts` but not propagated here. All 13 `Buffer.from()` calls create unzeroed copies of key material in the Buffer pool slab.

The file has zero test coverage. The critical HChaCha20 bug has persisted undetected because: (a) the catch blocks silently swallow the TypeError, (b) the fallback to @noble produces correct output, and (c) there are no tests that verify the native path executes.

**What actually works as advertised:**
- `timingSafeEqual` correctly delegates to `CRYPTO_memcmp` in C++ (verified in source)
- `hkdfSync` correctly delegates to native OpenSSL HKDF
- `createHash('sha256')` and `createHmac('sha256', ...)` correctly delegate to native OpenSSL
- `randomFillSync` and `randomBytes` correctly delegate to native CSPRNG
- Synchronous `require()` detection (V6-F9 fix) correctly avoids race conditions

**What does NOT work as advertised:**
- Native AEAD (XChaCha20-Poly1305) -- dead code due to hchacha20 bug
- `OPENSSL_cleanse` -- not called, documentation is false
- Constant-time fallback comparison -- has timing leak fixed elsewhere but not here
- Buffer pool key material zeroing -- 13 unzeroed copies per crypto operation
