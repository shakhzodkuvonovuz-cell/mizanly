## Findings

| F07-01 | **CRITICAL** | `storage.ts` | **`cleanupOrphanedOTPKeys` and `exportAllState` read pre-key registry via raw `mmkv.getString()`, bypassing AEAD -- silent data loss after HMAC migration** | Lines 1066 and 1172-1173: `mmkv.getString('prekey_registry:opk')` and `mmkv.getString('prekey_registry:spk')` read the raw plaintext key. But `trackPreKeyId` (line 1314) and `untrackPreKeyId` (line 1329) write via `secureStore(HMAC_TYPE.PREKEY_REGISTRY, 'prekey_registry:opk', ...)` which stores data under an HMAC-hashed key name with AEAD encryption. After the first `trackPreKeyId` call migrates the data, the raw key `prekey_registry:opk` is deleted and the data lives only under `r:<HMAC hash>`. From that point: (1) `cleanupOrphanedOTPKeys` returns 0 (registry appears empty) -- orphaned OTP private keys permanently leak in SecureStore, (2) `exportAllState` exports zero pre-key private keys -- backup is incomplete and restoring it loses all pre-key state, breaking all active sessions. Similarly, line 1120: `mmkv.set('prekey_registry:opk', JSON.stringify([]))` writes to the wrong key, achieving nothing. This is not a theoretical issue -- any device that has generated pre-keys and triggered HMAC migration (which happens on first `getMMKV()` after the F4 fix) will hit this. |
| F07-02 | **HIGH** | `key-transparency.ts` | **Root signature covers only the Merkle root hash, not `updatedAt` or `treeSize` -- a compromised signing key holder can replay old roots with forged timestamps** | Line 68: `ed25519Verify(publicKey, root, signature)` verifies the signature over `root` bytes only. The `updatedAt` and `treeSize` fields are unsigned -- the client trusts them as-is from the server. Attack: a compromised server (or anyone who obtains the signing key) takes a legitimately signed root from when the tree was honest, pairs it with `updatedAt: new Date().toISOString()` and `treeSize: <monotonically larger>`, and serves this to the client. The root signature verifies, the freshness check passes, the tree size check passes, and the Merkle proof for the user's original (honest) key verifies. Meanwhile, the server is serving different users a substituted key. The signature must cover `SHA-256(root || updatedAt || uint64BE(treeSize))` or a structured signed message binding all fields. Without this, the freshness and monotonic growth checks (V7-F3) are pure theater against a signing key compromise. |
| F07-03 | **HIGH** | `key-transparency.ts` | **`verifyRootSignature` returns `true` when `TRANSPARENCY_PUBLIC_KEY_B64` is empty string -- dead code path that could be triggered by build misconfiguration** | Lines 55-58: `if (!TRANSPARENCY_PUBLIC_KEY_B64) { return true; }`. The constant is currently set to a 32-byte key, so this branch is dead. However, the check is `!TRANSPARENCY_PUBLIC_KEY_B64` which is falsy for `''` (empty string). If a developer ever clears this constant (e.g., for debugging, build variant, or accidental commit), ALL transparency verification silently passes for ALL users -- the entire key transparency system becomes a no-op. A `const` assignment guarantees compile-time value, but the runtime check should throw an error (fail-closed) rather than return true (fail-open). A compromised build pipeline that strips this constant would completely defeat key transparency. |
| F07-04 | **HIGH** | `prekeys.ts` | **`checkAndRotateSignedPreKey` does not zero identity key pair private key bytes after use** | Line 296: `const identityKeyPair = await loadIdentityKeyPair()` loads the Ed25519 private key from SecureStore into a JS `Uint8Array`. The function uses this key at line 321 via `generateSignedPreKey(identityKeyPair, newKeyId)` which calls `ed25519Sign(identityKeyPair.privateKey, ...)`. After the function returns, `identityKeyPair.privateKey` (32 bytes of the permanent identity key) remains in the JS heap until GC collects the closure. For non-ephemeral keys (the identity key is the MOST sensitive key -- compromise = permanent impersonation), this is a meaningful window. The double-ratchet module correctly zeroes old ratchet keys (`zeroOut(oldKeyPair.privateKey)` at double-ratchet.ts:365), but the identity key -- which is orders of magnitude more sensitive -- gets no zeroing anywhere after load. Every call to `loadIdentityKeyPair()` across 6+ call sites (x3dh.ts:127, x3dh.ts:332, sealed-sender.ts:163, sealed-sender.ts:207, safety-numbers.ts:93, prekeys.ts:296) leaves the identity private key on the heap. |
| F07-05 | **HIGH** | `storage.ts` | **`loadIdentityKeyPair` returns identity private key via `fromBase64(privB64)` but the base64 string `privB64` is an immutable JS string that cannot be zeroed** | Line 452-464: `const privB64 = await SecureStore.getItemAsync(...)` returns a JS string (immutable). `fromBase64(privB64)` creates a new `Uint8Array` (zeroable), but `privB64` itself persists until GC. Even if the caller zeros the returned `Uint8Array`, the base64-encoded identity private key remains as an unreachable-but-not-collected immutable string on the V8/Hermes heap. This is the same class of issue as the LiveKit E2EE key (documented in CLAUDE.md "Technical Debt"). For the PERMANENT identity key (not ephemeral session keys), this is more severe -- every `loadIdentityKeyPair()` call leaves another immutable string copy. The `fromBase64` function (crypto.ts:419) correctly copies bytes to an isolated buffer (V7-F12), but the source base64 string is the unzeroable artifact. Fix: JSI native module that loads from SecureStore directly into a C++ buffer, never exposing the base64 string to JS. |
| F07-06 | **MEDIUM** | `storage.ts` | **`getMMKV` reads the master encryption key from SecureStore twice in the same initialization** | Lines 121 and 151: The MMKV initialization reads `SECURE_STORE_KEYS.MMKV_ENCRYPTION_KEY` from SecureStore twice -- once to derive the MMKV CFB key and once to derive the HMAC key. The first `fromBase64(encryptionKey)` result on line 140 is never zeroed before the second read. Both decoded byte arrays and the base64 string `encryptionKey` remain in JS heap. This doubles the exposure window for the master key material. Should derive MMKV key and HMAC key from a single SecureStore read, zero the decoded bytes immediately after both derivations, and null out the base64 string variable. |
| F07-07 | **MEDIUM** | `prekeys.ts` | **`generateOneTimePreKeys` returns array of `OneTimePreKey` objects containing private key bytes that are never zeroed** | Lines 179-199: After generating 100 OTP key pairs and storing private keys in SecureStore, the function returns the full `OneTimePreKey[]` array containing both public and private keys. The caller (`prepareOneTimePreKeysUpload`) only needs public keys. The 100 private keys (3,200 bytes of key material) persist as part of the returned array until the caller's scope ends and GC collects them. No zeroing occurs. The `generateSignedPreKey` function (line 118) has the same pattern -- returns `SignedPreKey` containing the full X25519 private key. These are ephemeral DH keys (lower severity than identity keys), but 100 OTP private keys on the heap simultaneously is a large attack surface for a memory dump. |
| F07-08 | **MEDIUM** | `key-transparency.ts` | **`lastSeenTreeSize` in-memory Map is lost on app restart -- tree size monotonic check resets, allowing one-time tree shrink attack per app restart** | Lines 159 and 227-231: The `lastSeenTreeSize` Map is populated from persistent storage (`secureLoad`) on first verification per userId, but only IF a prior verification stored a value. The initial load uses `parseInt(stored, 10) || 0` which returns 0 for any parse failure (including `NaN` from corrupted data). If the persisted value is corrupted (AEAD tamper caught at a different layer), the fallback is 0 -- allowing the server to present a tree of any size. More importantly: the persistence at line 271 only happens on successful verification. If the app is killed between a successful verification and the `secureStore` write, the tree size check resets to the previously persisted value. A compromised server can exploit this timing window by causing app crashes (e.g., OOM via large proof arrays) right after verification but before persistence. |
| F07-09 | **MEDIUM** | `key-transparency.ts` | **Merkle leaf hash includes only `userId` and `identityKey` -- no `deviceId` binding, enabling cross-device key confusion** | Line 104-108: `concat(LEAF_PREFIX, TextEncoder.encode(userId), identityKey)` hashes `userId || identityKey` as the leaf. In a multi-device world (which the types support via `deviceId`), two devices for the same user would have different identity keys but the same `userId` prefix. If device A's key and device B's key are both in the tree, the proof for device A's key proves "some key for userId exists in the tree" -- not "THIS SPECIFIC device's key exists." A compromised server could serve device A's (older, potentially compromised) key when asked about device B, and the Merkle proof would verify because the userId matches. The leaf should be `SHA-256(0x00 || userId || deviceId || identityKey)`. |
| F07-10 | **MEDIUM** | `storage.ts` | **`importAllState` restores MMKV entries via raw `mmkv.set()`, bypassing AEAD -- restored sessions have no integrity protection until re-encrypted** | Lines 1287-1289: `for (const [key, val] of Object.entries(state.mmkvEntries ?? {})) { mmkv.set(key, val as string); }`. The exported backup stores raw MMKV strings (already AEAD-wrapped). The restore writes them back as-is via `mmkv.set()`, which is correct for AEAD-wrapped values (the `A1:` prefix and ciphertext are preserved). However, there is no validation that the restored values are still valid AEAD ciphertext. A tampered backup (bit-flip in the encrypted archive after Argon2id decryption succeeds due to a different integrity layer being absent -- note: XChaCha20-Poly1305 AEAD on the backup itself protects against this, but partial corruption in the JSON parse step could pass through truncated values) could inject entries that fail AEAD verification on first read, corrupting all sessions. The backup format should include a per-entry HMAC or verify AEAD integrity on restore. |
| F07-11 | **MEDIUM** | `prekeys.ts` | **`checkAndRotateSignedPreKey` uses dynamic `import('./storage')` creating a new module binding on each call instead of using top-level imports** | Lines 301: `const { secureLoad, secureStore: secStore, HMAC_TYPE } = await import('./storage')`. The top of the file already imports `storeSignedPreKeyPrivate`, `loadSignedPreKeyPrivate`, etc. from `'./storage'`. This dynamic import is unnecessary (the module is already loaded) and creates a code path that behaves differently if module resolution is mocked in tests. The `SecureStore` import at line 24 is a top-level static import, proving the module dependency is not circular. This is not a security issue per se, but the inconsistency between the top-level static imports and the dynamic imports suggests the function was written in isolation without checking existing imports. |
| F07-12 | **LOW** | `prekeys.ts` | **Registration ID generation has modular bias: 14-bit mask then non-zero clamp** | Lines 97-99: `const raw = ((bytes[0] << 8) | bytes[1]) & 0x3fff` masks 16 bits to 14 bits (0-16383). Then `raw === 0 ? 1 : raw` shifts the probability. Value 0 maps to 1, so ID=1 has twice the probability of any other value (2/16384 vs 1/16384). The bias is negligible for a non-security-critical identifier (registration IDs are not secret), but the correct approach is rejection sampling: generate until non-zero. |
| F07-13 | **LOW** | `storage.ts` | **`clearAllE2EState` does not delete the MMKV encryption key from SecureStore** | Lines 1338-1413: `clearAllE2EState` deletes identity keys, registration ID, pre-key private keys, sender signing keys, and all MMKV entries. But it does NOT delete `SECURE_STORE_KEYS.MMKV_ENCRYPTION_KEY`. After logout, the MMKV encryption key persists in SecureStore. If the user re-registers, `getMMKV()` reuses the old encryption key. This means the MMKV base encryption key and the derived AEAD key survive logout. A forensic analyst who compromises SecureStore post-logout can decrypt any MMKV data written before the `clearAll()` that wasn't overwritten (flash storage wear-leveling may retain old pages). The `panicWipe` function (line 1439) does handle this correctly -- it overwrites and deletes the MMKV key. But normal logout via `clearAllE2EState` does not. |
| F07-14 | **LOW** | `key-transparency.ts` | **`verifyConsistencyProof` accepts `oldSize === 0` as trivially consistent -- allows bootstrapping a fully forged tree** | Line 320: `if (oldSize === 0) return true`. An empty tree is mathematically consistent with any new tree. But from a security perspective, this means the very first consistency check after app install always passes, regardless of what the server serves. Combined with the `no_local_key` early return in `verifyKeyTransparency` (line 184), a new user has zero protection against key substitution until TOFU establishes a local reference. This is inherent to TOFU + Merkle transparency systems, but the code should at minimum log/alert when this bootstrapping path is taken, rather than silently returning `true`. |
| F07-15 | **LOW** | `storage.ts` | **`aeadGet` error message leaks the HMAC type prefix of the corrupted key** | Line 271: `'MMKV integrity check failed for key "${(aadKey ?? storageKey).split(':')[0]}:..."'`. If `aadKey` is the original unhashed key (e.g., `session:user_abc:1`), `split(':')[0]` leaks `session` -- revealing the key type. For HMAC-hashed keys where `aadKey` is not passed, `storageKey` is `s:A7f3...` and the split leaks the type prefix `s`. While this doesn't reveal user IDs, it tells an attacker (reading crash logs or Sentry) which type of data had integrity failure, helping them target their tampering to specific key types. Error messages for security-sensitive storage should not include any key metadata. |
| F07-16 | **LOW** | `prekeys.ts` | **No rate limiting or cap on OTP batch generation -- a bug in replenishment logic could generate unlimited SecureStore entries** | The `generateOneTimePreKeysBatch` function (line 265) generates `count` keys (default 100) without checking how many OTP keys already exist in SecureStore. If the `needsReplenishment` check is called with stale `serverCount` data, or if the server always reports count=0, the client will generate 100 new keys on every app open. SecureStore has platform-dependent limits (iOS Keychain is practically unlimited, Android Keystore has ~1000 key entries on some devices). After 10 faulty replenishments, 1000 OTP private keys consume SecureStore space and the pre-key registry grows to 1000 entries in MMKV. There is no dedup or cap in `trackPreKeyId`. |

## Architecture Observations (Not Findings)

1. **Identity key storage in SecureStore is correct.** The Ed25519 private key never touches MMKV -- it goes directly to `expo-secure-store` which maps to iOS Keychain (Secure Enclave on supported devices) and Android Keystore. This is the right design.

2. **AEAD + HMAC key name pattern is comprehensive.** All security-sensitive MMKV values (sessions, identity keys, sender keys, dedup sets, OTP registry, queue) use `secureStore`/`secureLoad` with HMAC-hashed key names and per-value AEAD. The implementation is thorough and consistent across all storage paths.

3. **Session mutex (chained promises) is a correct single-threaded async lock.** The `withSessionLock` implementation correctly serializes operations on the same session while allowing parallelism across sessions. No race condition.

4. **Signed pre-key rotation lifecycle is sound.** 7-day rotation, 30-day retention of old keys, monotonic keyId increment, proper cleanup of old private keys from SecureStore. Matches Signal spec.

5. **Key transparency implements RFC 6962 correctly.** Domain-separated leaf/internal nodes (0x00/0x01 prefixes), full consistency proof verification (not a stub), hash version tracking for protocol evolution, and monotonic tree size enforcement. The implementation is well-documented and matches the spec.

6. **AEAD key derivation via HKDF with unique info strings is correct.** Three derived keys from one master: `MizanlyMMKVCFB` (MMKV base encryption), `MizanlyHMACKeyNames` (key name hashing), `MizanlyMMKVAEAD` (value encryption). HKDF with distinct info strings guarantees domain separation -- compromising one derived key does not compromise the others.

7. **Backup encryption (Argon2id + XChaCha20-Poly1305) is well-parameterized.** m=64MB, t=3, p=4 matches OWASP minimum. 32-byte salt, 24-byte nonce, versioned format byte. Password entropy enforcement (12 chars, mixed case + digit). Restore correctly resets all singletons.
