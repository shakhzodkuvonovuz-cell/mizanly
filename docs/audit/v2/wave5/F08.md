## Findings

| # | Severity | File | Finding | Detail |
|---|----------|------|---------|--------|
| F08-01 | **CRITICAL** | `media-crypto.ts` | **HKDF nonce derived from (mediaKey, chunkIndex) with static salt is deterministic -- but chunkKey also uses zero salt** | Line 119: `hkdfDeriveSecrets(ctx.mediaKey, new Uint8Array(32), CHUNK_KEY_INFO, 32)` derives the per-file chunk encryption key with a **zero salt**. Line 140: `hkdfDeriveSecrets(ctx.mediaKey, uint32BE(i), CHUNK_NONCE_INFO, 24)` derives the nonce from (mediaKey, chunkIndex). Since each file has a unique random mediaKey, the nonces are unique across files. HOWEVER: if the same mediaKey is ever reused (forward-to-self, retry-after-failure that re-uses the same context object, cache rehydration), every chunk will have an **identical (chunkKey, nonce) pair**, producing identical keystreams. XChaCha20 nonce reuse under the same key is **catastrophic** -- XOR of two ciphertexts reveals plaintext XOR. The `prepareMediaEncryption` function generates a fresh mediaKey, but the `MediaEncryptionContext` object holding it is returned to the caller who could (incorrectly) reuse it. There is no nonce-reuse guard or used-key tombstone. |
| F08-02 | **CRITICAL** | `multi-device.ts` | **Device-link 6-digit code derived from only 3 bytes of secret, brute-forceable in <1 second** | Lines 241-242: `const num = ((secret[0] << 16) | (secret[1] << 8) | secret[2]) % 1000000;`. Despite generating a 32-byte CSPRNG secret, only the first 3 bytes (24 bits of entropy = 16.7M values) are used, then reduced mod 1M to a 6-digit code (20 bits effective entropy = ~1M values). The `MAX_LINK_ATTEMPTS = 5` client-side rate limit is trivially bypassable -- an attacker can patch the binary, use Frida to reset `linkAttemptsCache`, or simply restart the app (the `ensureLinkCacheLoaded` is async and non-blocking on first call, line 294). The comment at line 257 says "Server-side rate limiting (V4-F20) remains the authoritative control" but there is **no evidence in this file or e2eApi.ts that the Go server enforces rate limiting on device-link verification attempts**. If server-side rate limiting is missing or bypassable, an attacker with the device in hand (or who intercepted the link request) can brute-force all 1M codes in seconds. |
| F08-03 | **HIGH** | `multi-device.ts` | **`verifyDeviceLinkCode` is synchronous but depends on async state load -- race window** | Line 287: `verifyDeviceLinkCode` is a synchronous function (`function`, not `async function`). Line 294: It calls `ensureLinkCacheLoaded().catch(() => {})` fire-and-forget. On the first verification attempt after app restart, `linkAttemptsCache` is 0 (not yet loaded from MMKV). An attacker can make verification attempts before the async load completes, getting up to `MAX_LINK_ATTEMPTS` free attempts that don't count against the persisted counter. This effectively doubles the allowed attempts from 5 to 10 on every app restart. |
| F08-04 | **HIGH** | `media-crypto.ts` | **Decrypted media temp file persists for 5 seconds on filesystem -- extractable by root/forensic** | Lines 376-383: After decryption, the full decrypted file is written to `cacheDirectory` and a 5-second `setTimeout` schedules deletion. On a rooted/jailbroken device, an attacker can: (1) watch the cacheDirectory for new `decrypted_*` files, (2) copy them before the 5-second timer fires. The `cleanupDecryptedMediaFiles()` in `index.ts` runs on app launch and backgrounding, but NOT continuously. If the user views multiple media files in a session, each one persists for 5 seconds. There is no overwrite-before-delete (secure deletion) -- `FileSystem.deleteAsync` only unlinks the inode, data remains recoverable on flash storage until sectors are reused. |
| F08-05 | **HIGH** | `media-crypto.ts` | **SHA-256 verification happens AFTER full decryption -- malicious file fully processed** | Lines 320-364: The decrypt loop decrypts ALL chunks AND accumulates them in memory, then verifies SHA-256 at line 358. If a server-side attacker provides a file with valid AEAD per-chunk (they have the key if the server is compromised) but wrong overall SHA-256, all chunks are decrypted and held in memory before verification fails. For a 50MB file, this means 50MB of attacker-controlled decrypted data is in memory. While AEAD protects individual chunk integrity, the SHA-256 check guards against chunk reordering/truncation at the file level. The order of operations should be: verify SHA-256 of the *encrypted* chunks first (which is what happens since the hasher runs on encrypted data), then decrypt. But the current code decrypts during the hash loop, so decrypted data accumulates even if the file hash will fail. |
| F08-06 | **HIGH** | `message-cache.ts` | **`isMessageCached` checks raw MMKV key presence, bypassing AEAD integrity verification** | Line 183: `mmkv.contains(hashed) || mmkv.contains(originalKey)` checks if a key exists in MMKV without decrypting/verifying the AEAD tag. A forensic attacker who can flip bits in MMKV storage can make `isMessageCached` return `true` for entries whose integrity is actually compromised (AEAD tag invalid). This causes the caller to skip re-decryption of messages that have been tampered with, displaying stale/corrupt data or silently dropping messages that should have been re-fetched from the server. |
| F08-07 | **HIGH** | `search-index.ts` | **Tokens are lowercased plaintext stored as HMAC input -- HMAC is deterministic, enabling dictionary attack** | Lines 97-98: `const tokenOriginalKey = INDEX_TOKEN_PREFIX + token` where `token` is the lowercased plaintext word. This original key is used as input to `hmacKeyName()` which produces a deterministic HMAC. An attacker with MMKV filesystem access can: (1) extract all HMAC-hashed token keys, (2) compute `hmacKeyName(HMAC_TYPE.SEARCH_TOKEN, "searchidx:t:" + word)` for a dictionary of common words, (3) match against stored hashes to determine which words appear in the user's messages. The HMAC key is derived from the MMKV encryption key in SecureStore, so this requires either the HMAC key or SecureStore compromise. But the HMAC key IS cached permanently in memory (line 150 of storage.ts: `hmacKeyForNames`), extractable via Frida/memory dump. |
| F08-08 | **MEDIUM** | `offline-queue.ts` | **`markMessageFailed` uses `as any` cast to bypass telemetry type safety** | Line 62: `recordE2EEvent({ event: 'message_encrypt_failed' as any, ... })`. The event `message_encrypt_failed` is not in the `E2EEvent` union type in `telemetry.ts`. The `as any` cast silences the type checker, meaning this telemetry event is untracked and won't appear in `getE2ETelemetrySnapshot()` aggregation. This is a code quality issue that could hide failure patterns. |
| F08-09 | **MEDIUM** | `message-cache.ts` | **LRU eviction cannot decrypt HMAC-keyed entries -- eviction degrades over time** | Lines 239-277: `evictOldestMessages` iterates MMKV keys and attempts to parse cached messages to determine `createdAt` for LRU ordering. Line 263: `if (raw.startsWith('A1:') && key.startsWith(HMAC_TYPE.MSG_CACHE)) continue;` -- HMAC-keyed AEAD entries are SKIPPED because the original key (needed as AAD) is unknown. This means once all legacy entries are evicted, the LRU eviction becomes a no-op. Only the per-conversation cap (500 messages) provides eviction. With 100 conversations x 500 messages = 50K messages, the MAX_CACHED_MESSAGES=50K cap is reached but cannot evict anything, causing MMKV to grow unbounded as new conversations are added. |
| F08-10 | **MEDIUM** | `search-index.ts` | **Same LRU eviction failure as message-cache -- HMAC-keyed entries unevictable** | Lines 239-277 of `search-index.ts`: `evictOldestIndexEntries` has the identical pattern -- skips HMAC-keyed entries at line 262: `if (key.startsWith(INDEX_MSG_PREFIX))` processes only legacy keys. Once legacy entries are gone, eviction stops working. The 50K index cap becomes a hard ceiling that can only grow via legacy cleanup, never shrink new entries. |
| F08-11 | **MEDIUM** | `streaming-upload.ts` | **Multipart upload progress not persisted to MMKV -- resume after kill restarts from scratch** | Lines 166-173: The `progress` object tracking `completedParts` is in-memory only. The doc comment (lines 12-14) claims "Resume support: completed parts are tracked in MMKV" but the code never writes to MMKV. If the app is killed mid-upload of a 2GB file, all completed parts are lost and the upload restarts from part 1. The R2 multipart upload itself persists for 7 days, but without knowing the `uploadId`, `key`, and completed part ETags, the client cannot resume. |
| F08-12 | **MEDIUM** | `notification-handler.ts` | **`encryptPreview` has default empty-string conversationId AAD** | Line 134: `conversationId: string = ''`. Looking at the test file, line 260 of `v5-audit-fixes.test.ts` and lines 343-412 of `safety-numbers.test.ts` show multiple call sites passing no conversationId. | F08-13 | **MEDIUM** | `multi-device.ts` | **`knownDeviceCache` is module-level Map that grows unbounded** | Line 55: `const knownDeviceCache = new Map<string, number[]>()`. While each entry is small (userId string + array of 1-10 numbers), over the lifetime of an active user with thousands of contacts, this Map grows without bound. No eviction, no WeakMap, no cap. Combined with the MMKV persistence at line 117, this is a memory leak that survives hot reloads but not app restart. |
| F08-14 | **MEDIUM** | `media-crypto.ts` | **`encryptSmallMediaFile` returns deprecated `mediaKey: zeroedKey` that shares buffer identity confusion** | Lines 220-227: The function returns both `mediaKeyB64` (the actual key as base64) and `mediaKey: zeroedKey` (a zeroed-out Uint8Array for backward compat). | F08-15 | **LOW** | `telemetry.ts` | **`sanitizeErrorType` fallback regex may leak partial sensitive data** | Line 131: `return errorType.replace(/[a-z0-9_-]{20,}/gi, '[REDACTED]').slice(0, 50)`. ## Architecture Observations (Not Findings)

1. **Media crypto nonce derivation is sound IF mediaKey is never reused.** Each file gets `generateRandomBytes(32)` as mediaKey. HKDF with different chunk indices produces unique nonces. The critical assumption is one-time use of `MediaEncryptionContext` -- this is not enforced.

2. **Offline queue correctly avoids plaintext storage.** The `QueuedMessage` type has no plaintext field, and the comment at types.ts:239 explicitly documents this decision. The queue only holds encrypted payloads.

3. **Safety number algorithm matches Signal spec.** 5200 HMAC-SHA256 iterations, 2-byte version prefix, sorted by userId for determinism. The `computeSafetyNumberFromKeys` export allows cross-validation with the Go server.

4. **Telemetry sanitization is thorough.** The `sanitizeErrorType` function maps known error patterns to generic codes, and the `recordE2EEvent` function explicitly avoids logging metadata for failure events.

5. **AEAD + HMAC key name pattern is consistently applied** across message-cache.ts, search-index.ts, offline-queue.ts, and notification-handler.ts. All delegate to `secureStore`/`secureLoad`/`secureDelete` from storage.ts.

## Recommended Fixes (Priority Order)

| # | Fix | Effort |
|---|-----|--------|
| F08-01 | Add a `consumed: boolean` flag to `MediaEncryptionContext`. Set to `true` after `encryptMediaChunked` yields its first chunk. Throw if the generator is called again on a consumed context. Alternatively, move mediaKey generation inside the generator so it's impossible to reuse. | Small |
| F08-02 | Replace the 6-digit device link code with a 8-digit code derived from HKDF(secret, "DeviceLink", 4) mod 10^8, AND require the server-side Go endpoint to enforce a hard 5-attempt rate limit with exponential backoff per link session. Verify the Go server rate limit exists. | Medium |
| F08-03 | Make `verifyDeviceLinkCode` async and `await ensureLinkCacheLoaded()` before checking the counter. The synchronous API is a false economy when security depends on the loaded state. | Small |
| F08-04 | Use `expo-file-system`'s streaming read + in-memory buffer instead of writing decrypted files to disk. For images, decode directly from the Uint8Array. If file must be written, overwrite with random data before unlinking. | Medium |
| F08-05 | This is partially mitigated by per-chunk AEAD -- SHA-256 is a secondary check. The current order is actually correct for performance (single-pass). Document that AEAD is the primary integrity mechanism and SHA-256 catches chunk reordering only. | Small |
| F08-09/10 | Fix LRU eviction for HMAC-keyed entries: store a separate time-sorted eviction index mapping `createdAt -> originalKey` (encrypted in MMKV). This allows eviction without needing to decrypt every entry. | Medium |
| F08-11 | Persist `UploadProgress` to MMKV (AEAD-encrypted) on each completed part. On app restart, check for in-progress uploads and resume. | Medium |
| F08-12 | Remove the default `''` value from `conversationId` parameter -- make it required. Fix all callers. | Small |
