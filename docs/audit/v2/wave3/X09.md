## Summary

The search pipeline is architecturally sound -- it has a Prisma fallback, a Meilisearch fast path, queue-based indexing, weekly reconciliation, and publish/unpublish workflows. However, there are significant gaps:

**Critical privacy gap (Findings 1-2):** The main search endpoint never passes userId, so blocked/muted user filtering is impossible. When Meilisearch is active, no server-side filtering for removed/private/banned content occurs on the fast path -- raw index hits are returned directly.

**Incomplete index lifecycle (Findings 3-5):** Channels are entirely missing from the Meilisearch pipeline. Admin bans remove the user but not their content from the index. Report-based bans remove posts/reels/threads but not videos.

**Stale data in index (Findings 7-8, 10):** The reconciliation and sync services push private posts, future-scheduled content, and non-public threads into Meilisearch because they lack the same `visibility`/`scheduledAt` filters that the search queries use.

**Feature divergence (Finding 17):** The dedicated `/search/reels` endpoint searches both caption and hashtags, while the universal `/search?type=reels` only searches caption. Users get different results depending on which endpoint the mobile app calls.
