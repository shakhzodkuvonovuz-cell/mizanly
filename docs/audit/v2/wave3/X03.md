## Summary of Cross-Module Trace

### Stripe Checkout -> Webhook -> Dedup -> Credit Flow
The overall architecture is sound: PaymentIntent created client-side, webhook handler routes by `metadata.type`, dedup uses dual Redis+DB. However, the tip fallback matching (finding #2) and fire-and-forget DB writes (findings #1, #9) create a window where money is misrouted when Redis state is lost. The coin purchase dedup (finding #4) relies on string matching in a description field rather than a dedicated indexed column.

### Webhook Dedup via ProcessedWebhookEvent
Correctly implemented with dual Redis (7-day TTL) + DB (permanent) dedup. The only gap is that on handler failure, the event is NOT marked as processed, causing Stripe retries to re-execute the handler (finding #22). This is correct for transient failures but problematic for permanent ones.

### Dual CoinBalance System (coins + diamonds)
The CoinBalance table is correctly the single source of truth. Gifts service uses `$transaction` properly for atomic debit/credit. However, the diamond credit in the tip webhook handler (finding #3) lacks its own idempotency guard. The DB CHECK constraints for non-negative balances exist in a SQL file but may not be applied (finding #25).

### Cascade Behaviors on Financial Records
- **Tip:** `onDelete: SetNull` on senderId/receiverId — correct, preserves financial records.
- **CoinBalance:** `onDelete: Cascade` — when user is deleted, their balance is destroyed. This is fine if there's no pending cashout.
- **CoinTransaction:** `onDelete: SetNull` on userId — correct, preserves transaction history.
- **Order:** `onDelete: SetNull` on buyerId — correct. But `onDelete: Cascade` on productId means product deletion destroys order records (finding #30).
- **MembershipSubscription:** `onDelete: Cascade` on both tier and user — tier deletion destroys subscription records, which may be needed for financial auditing.

### Race Conditions
The conditional `updateMany({ where: { coins: { gte: amount } }, data: { decrement: amount } })` pattern used in gifts and cashout is the correct way to prevent negative balances without explicit locking. The post-check reversal code that follows is dead/harmful (findings #6, #8). The order stock decrement in `createOrder` properly uses this pattern inside a `$transaction`. The order status update does NOT (finding #5).

### Disconnected Subscription Endpoints
The `monetization.subscribe()` and `payments.createSubscription()` endpoints are completely independent — different code paths, different Stripe integration levels. Similarly, `monetization.unsubscribe()` and `payments.cancelSubscription()` diverge on Stripe cancellation. This creates a serious consistency gap (findings #14, #15).
