## Summary of Critical Path Issues

### send_sealed_message is under-validated (Findings 1-2)
The `send_sealed_message` handler is the weakest point in the gateway. Unlike `send_message` which uses a DTO with class-validator decorators and MaxLength constraints, `send_sealed_message` uses only TypeScript interface types (erased at runtime). There is no size limit on `ephemeralKey`, `sealedCiphertext`, or `encryptedContent`. Additionally, the sender's conversation membership is never verified -- only the recipient's. The sealed envelope is forwarded to the recipient BEFORE persistence is attempted, so even if `sendMessage` rejects the sender, the message is already delivered.

### Media processing pipeline is disconnected (Finding 7)
The `MediaProcessor` with EXIF stripping and BlurHash generation exists but is unreachable. `QueueService` has no public method to enqueue media jobs. This means uploaded images retain GPS coordinates and other EXIF metadata, which is a GDPR/privacy concern documented in the processor's own comments but never addressed at the enqueue layer.

### DLQ Sentry fallback is dead code (Finding 4)
The `moveToDlq` method's fallback to Sentry when both Redis and DB fail can never trigger because the `.catch()` handlers on `redisDone` and `dbDone` convert rejections to fulfilled promises. The second `Promise.allSettled` always sees both as fulfilled.

### Presence privacy bypass (Finding 6)
Any authenticated user can track any other user's online/offline status by subscribing to their presence room. No block-list check, no privacy setting check, no mutual-follow requirement. The `activityStatus` setting only controls whether the server BROADCASTS presence, but if a user subscribes to the room, they receive all events regardless.
