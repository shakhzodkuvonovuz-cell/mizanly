## Summary of Key Concerns

### Critical: Sealed Sender Metadata Leak via Presence Rooms
Finding #1 is the most severe issue. The `subscribe_presence` handler has zero access control, allowing any authenticated user to eavesdrop on sealed sender message events in another user's room. This completely defeats the metadata privacy that sealed sender is designed to provide. Fix: either (a) route sealed messages via a separate, non-joinable transport, or (b) add access-control to `subscribe_presence` that prevents joining rooms of non-contacts, or (c) emit sealed messages on a separate event namespace that presence subscribers don't receive.

### High: Sealed Sender Code Path Inconsistency
Findings #2, #3, #6 show the sealed sender implementation has diverged: the gateway (Codex-V7-F2) changed to pass a real senderId but the service still has a dead code path expecting `senderId=null`. The gateway also tries to persist non-existent schema columns. The entire sealed sender persistence path needs reconciliation.

### High: E2E Bypass via Scheduled Messages
Finding #18 shows that `scheduleMessage` creates messages directly in the database without the `isE2E` enforcement check, allowing plaintext injection into E2E conversations.

### High: Disappearing Message Duration Bug
Finding #4 causes messages to disappear 60x faster than configured when set via `setMessageExpiry`. This is a straightforward unit conversion bug.

### Medium: Expired/View-Once Messages Retain Crypto Material
Findings #8 and #9 show that the message expiry cron jobs don't clear E2E fields, unlike the manual delete path which correctly wipes all crypto material.
