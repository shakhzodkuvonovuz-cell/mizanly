## Critical Path Summary

The two Critical findings form a combined GDPR disaster path:

1. **Finding #1:** When Clerk deletes a user (webhook `user.deleted`), `deactivateByClerkId` sets `isDeleted: true` immediately. The `processScheduledDeletions` cron only processes `isDeleted: false` users. Result: the data purge (`deleteAllUserData`) is NEVER executed. PII, media, messages, and encryption keys persist indefinitely.

2. **Finding #2:** When a user requests deletion via the API (30-day grace period), the system sets `isDeactivated: true` and promises "You can cancel before then." But the cancellation endpoint requires authentication through `ClerkAuthGuard`, which rejects deactivated users. The grace period is a lie — users cannot cancel.

Together, these mean: Clerk-path deletions never purge data, and API-path deletions cannot be cancelled. Both deletion flows are fundamentally broken.
