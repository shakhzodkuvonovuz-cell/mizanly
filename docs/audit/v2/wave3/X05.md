## Detailed Analysis

### Critical Findings

**#1 — TAG/COLLAB_INVITE push types missing:** Posts service creates these notification types at `posts.service.ts:650` (TAG) and `posts.service.ts:673` (COLLAB_INVITE). They include `title` and `body` in the `create()` call, so they DO fire via the `default` fallback in `PushTriggerService`. However, the `data` field in the fallback push is empty (`{}`), which means the mobile `usePushNotificationHandler` gets a notification with no routing data. Since `type` would be `undefined` in the data, it falls to the `default` case and navigates to the generic notifications screen.

**#2 — No banned/deleted user filtering:** The `create()` method at line 197 fetches `user.notificationsOn` but not `user.isBanned`, `user.isDeactivated`, or `user.isDeleted`. The select clause at line 204 is `{ notificationsOn: true }`. A simple fix: add `isBanned`, `isDeactivated`, `isDeleted` to the select and check them before proceeding.

### High-Severity Findings

**#3/#4 — Plaintext leak in forward and scheduled message push:** The normal `sendMessage()` path at line 481 correctly branches: `data.e2eVersion ? 'New message' : content`. But `forwardMessage()` and `publishScheduledMessages()` bypass this check and pass raw `original.content` / `msg.content` to `notifyConversationMembers()`. Fix: extract the E2E-safe body logic into a helper and use it in all three paths.

**#6 — markRead increments badge instead of syncing it:** The `markRead` method publishes to `notification:new` channel, but the mobile client treats ALL events on that channel as "new notification arrived" and increments by 1. The server sends `{ type: 'UNREAD_COUNT_UPDATE', unreadCount: N }` but the client ignores the type and count fields entirely. Fix: either use a separate Redis channel for count syncs, or have the client inspect the notification type before deciding whether to increment.

### Architecture Notes

The notification pipeline has a well-structured flow: caller -> `NotificationsService.create()` (dedup + settings + block/mute) -> DB insert -> `PushTriggerService.triggerPush()` (type routing) -> `PushService.sendToUser()` (Expo HTTP) + Redis pub/sub -> `ChatGateway` -> Socket.io -> mobile. The dedup, batching, block/mute, and settings checks are all in the right place. The main gaps are in edge cases (E2E forward/scheduled, mobile type coverage, badge sync direction) rather than structural problems.
