## Detailed Critical Findings

### C1: `handlePaymentIntentFailed` — ZERO tests
**File:** `payments.service.ts:779-814`
**Risk:** When a customer's payment fails, the tip stays as 'pending' forever. No test verifies the tip is marked 'failed', the error message is recorded, or the Redis mapping is cleaned up. This is a money-related error path.

### C2: `handleInvoicePaymentFailed` — ZERO tests
**File:** `payments.service.ts:816-846`
**Risk:** When a subscription invoice fails, the 3-day grace period logic (`past_due` status + `graceDeadline`) is completely untested. This is the path that determines whether subscribers lose access.

### C3: `handleSubscriptionUpdated` — ZERO tests
**File:** `payments.service.ts:848-887`
**Risk:** The Stripe-to-internal status mapping (`active`, `past_due`, `canceled`, `unpaid`, `paused`) is untested. A mapping error would silently give/revoke access.

### C4: `updateProduct` — ZERO tests (service + controller)
**File:** `commerce.service.ts:81-102`
**Risk:** Product owner can update price/status. No test for ForbiddenException on non-owner, no test for negative price rejection, no test for invalid status values.

### C5: `deleteProduct` — ZERO tests (service + controller)
**File:** `commerce.service.ts:104-119`
**Risk:** Active order guard (cannot delete product with pending/paid/shipped orders) is completely untested. A bug here would orphan active orders.

### C6: `updateOrderStatus` — ZERO tests (service + controller)
**File:** `commerce.service.ts:316-359`
**Risk:** THE most complex method in commerce: status transition validation, stock restoration on cancel/refund, seller ownership check, buyer notification. Zero test coverage for any of it.

### C7: `getSellerOrders` — ZERO tests
**File:** `commerce.service.ts:269-288`
**Risk:** Seller's view of their orders (with status filter) is untested. This is a primary seller dashboard endpoint.

### C8: `getSellerAnalytics` — ZERO tests
**File:** `commerce.service.ts:293-314`
**Risk:** Revenue aggregation endpoint untested. Could silently return wrong totalRevenue.

### C9-C14: Commerce controller delegation gaps
Six commerce controller endpoints lack any test: `PATCH products/:id`, `DELETE products/:id`, `GET orders/selling`, `PATCH orders/:id/status`, `GET seller/analytics`, `PATCH businesses/:id`, `DELETE businesses/:id`. Without controller tests, even the routing/param extraction is unverified.

### Webhook controller event branches (C in service, C in controller)
The `StripeWebhookController` only tests 3 of 8 event types: `payment_intent.succeeded`, `invoice.paid`, `customer.subscription.deleted`. The remaining 5 event dispatches (`payment_intent.payment_failed`, `invoice.payment_failed`, `customer.subscription.updated`, `charge.dispute.created`, `payment_method.attached`) have zero controller-level tests.
