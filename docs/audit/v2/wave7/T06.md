## Summary by Module

### Messages Module
- **Controller**: 17 of 40+ endpoints have controller-level tests. All tests are delegation-only (verify service call happens). The remaining 23+ endpoints have zero controller tests.
- **Service**: Core CRUD well-tested (getConversations, getConversation, getMessages, sendMessage, deleteMessage, editMessage, createDM, createGroup, updateGroup, addGroupMembers, removeGroupMember, leaveGroup, markRead, muteConversation, archiveConversation, reactToMessage, removeReaction, setMemberTag, setLockCode, verifyLockCode, setNewMemberHistoryCount). E2E fields extensively tested. But 28+ methods have zero tests including: all starred message operations, all pin operations, scheduled messages + cron, DM notes, invite links, group roles (promote/demote/ban/changeRole), wallpaper, tone, disappearing timer, media gallery, mark delivered, archive/unarchive per-user, search (empty query path).
- **Internal E2E Controller**: Tested in e2e-fields.spec.ts — HMAC verification, missing signature, invalid format, missing rawBody, system message creation all covered.

### Chat Export Module
- **Best coverage in scope**. Both controller and service tested. Edge cases (empty conversation, deleted user sender, chunked loading, 10K cap, includeMedia=true/false) all covered. Only gap: no test for export with very long message content or special characters.

### Stickers Module
- **Basic CRUD tested**, but AI sticker generation (the most complex feature: content moderation, rate limiting, Claude API call, SVG sanitization, save-to-pack transaction) has zero tests. This is ~50% of the service code.
- Controller spec is missing 4 endpoints (generate, save, myGenerated, islamicPresets) from its mock service.

### Chat Gateway
- **Well-tested overall** with 1400+ lines of tests covering: join/leave conversation, send_message, typing (with auto-clear), read, presence tracking, online status, message_delivered, Quran rooms (join/leave/sync/reciter with host transfer), sealed sender, content rooms, presence subscriptions, activity status privacy, connection eviction, IP rate limiting, token extraction, module lifecycle.
- **Gap**: `onModuleInit` Redis pub/sub message handlers (6 channels with routing logic) are untested.

## Severity Counts

**C: 28** | **H: 37** | **M: 28** | **L: 7** | **I: 6** | **Total: 106**

### Critical (C) Breakdown
- 28 public service methods with zero tests at any layer, many with non-trivial logic (crypto invite links, scheduled message cron, pin max-3 enforcement, admin role changes, DM notes with expiry)

### High (H) Breakdown
- 37 controller endpoints with no controller-level test (the most common gap pattern — service may be tested but REST routing/param-passing is not)
- 4 sticker controller endpoints with missing mock methods
- 4 sticker service methods (AI generation pipeline) with zero tests

### Recommendation
Priority fix order:
1. `publishScheduledMessages` cron (C8) — critical production path, zero tests, distributed lock + batch logic
2. `generateSticker` + `sanitizeSvg` (H65, M92) — security-critical SVG sanitization is untested
3. All starred/pin message operations (C10-C15) — 6 methods, zero tests
4. Invite link generate + join (C22-C23) — crypto + dual-storage lookup
5. Admin role operations (C5-C7, C24) — promote/demote/ban/changeRole
6. DM notes (C25-C28) — 4 methods, zero tests
7. `onModuleInit` pub/sub handlers (M97-M99) — 6 channels of routing logic
