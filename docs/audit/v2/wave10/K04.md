# K04 — Queue Processing Audit

**Scope:** `apps/api/src/common/queue/` (6 processors, QueueService, QueueModule) + all `.add()` call sites
**Auditor:** Opus 4.6
**Date:** 2026-03-30

## Findings

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 1 | C | queue.service.ts:93-110 | **Webhook secret stored in Redis job payload.** `addWebhookDeliveryJob` receives `secret` (HMAC signing key) as job data. BullMQ serializes the entire `data` object to Redis as JSON. The secret is stored in plaintext in Redis for the job's entire lifetime (up to 7 days for failed jobs per `removeOnFail` config). If Redis is compromised, all webhook secrets are exposed. Also stored in `FailedJob.data` column in Postgres via DLQ `moveToDlq`. | Webhook secret exposure on Redis/DB compromise. Attacker can forge webhook signatures for all integrations. | M -- sign at enqueue time and pass only the signature, or encrypt the secret field before serializing |
| 2 | C | queue.service.ts:190-201 | **DLQ double-await bug -- Sentry fallback never triggers.** Lines 191 and 194 both call `Promise.allSettled([redisDone, dbDone])`. The first `await` at line 191 waits for both promises. By line 194, `redisDone` and `dbDone` are already settled (fulfilled with `undefined` because their `.catch()` handlers swallow errors and return `undefined`). The `.catch()` on lines 173/186 converts rejections to fulfilled promises. So `Promise.allSettled` at line 194 will always show both as `status: 'fulfilled'`, never `'rejected'`. The Sentry fallback on line 195 is dead code -- it can never execute. | If both Redis DLQ and DB DLQ fail simultaneously, the failure is silently lost with no Sentry alert. The CRITICAL log on line 200 never fires. | S -- remove `.catch()` wrappers from `redisDone`/`dbDone`, or restructure to track success/failure separately |
| 3 | H | media.processor.ts:ALL | **Media processor is a dead processor -- no producer exists.** `QueueService` has no `addMediaJob()`, `addImageResizeJob()`, `addBlurHashJob()`, or any method that calls `this.mediaQueue.add()`. The media queue is registered (line 31 of queue.service.ts), the worker is created (media.processor.ts), but no code anywhere in the codebase enqueues jobs to it. The entire 240-line processor (image resize, EXIF strip, BlurHash generation, R2 upload) is unreachable dead code. | Zero images are processed. No EXIF stripping, no BlurHash generation, no resize variants. Users' GPS coordinates remain in uploaded photos (GDPR Article 25 violation noted in the file's own comment). | M -- add `addMediaProcessingJob()` to QueueService and call it from upload/create flows |
| 4 | H | notification.processor.ts:46 | **`bulk-push` job handler exists but no producer.** The processor handles `'bulk-push'` jobs (line 46-48), but QueueService has no `addBulkPushJob()` method. No code in the codebase enqueues `bulk-push` jobs. The entire `processBulkPush` method is unreachable. | System announcements, broadcast notifications to multiple users cannot be sent via queue. Feature is built but never wired. | S -- add `addBulkPushJob()` to QueueService |

| 6 | H | ai-tasks.processor.ts:55 | **`generate-caption` job handler exists but no producer.** The processor handles `'generate-caption'` (line 55), but QueueService has no method to enqueue caption jobs. The handler itself (line 145-153) immediately throws `"Caption generation not implemented"`. Any job that somehow reaches this will fail on first attempt, retry, then fail again and go to DLQ. | Dead code path that would cause DLQ pollution if accidentally triggered. | S -- remove the case branch or add producer when feature is built |
| 7 | M | media.processor.ts:126,200 | **No timeout on `fetch(mediaUrl)` in media processor.** Both `processImageResize` (line 126) and `processBlurHash` (line 200) call `fetch(mediaUrl)` without `AbortSignal.timeout()`. The webhook processor correctly uses `AbortSignal.timeout(15000)` (webhook.processor.ts:118), but media processor does not. A slow/hanging CDN response will hold the worker slot indefinitely. With concurrency=3, three hanging fetches = complete media worker stall. | Worker slot starvation from slow upstream responses. Media processing stops entirely. | S -- add `signal: AbortSignal.timeout(30000)` to both fetch calls |
| 8 | M | media.processor.ts:131 | **Unbounded memory consumption from `response.arrayBuffer()`.** Line 131: `Buffer.from(await response.arrayBuffer())` loads the entire image into memory before processing. There is no size check. A 500MB image will allocate 500MB+ on the heap (buffer + arrayBuffer). With concurrency=3, this could cause OOM. Same pattern at line 205 for BlurHash. | OOM crash on large image uploads. Node process dies, all in-flight jobs across ALL queues are lost. | M -- add Content-Length check before downloading, reject files > 50MB, or use streaming |
| 9 | M | ALL processors | **No `on('error')` handler on any worker.** BullMQ Worker emits `'error'` events for connection errors, unexpected exceptions outside job processing, etc. None of the 6 processors register an `on('error')` handler. Unhandled `'error'` events on Node.js EventEmitters crash the process. | Unhandled Redis connection error crashes the entire API server. | S -- add `this.worker.on('error', (err) => { this.logger.error(...); Sentry.captureException(err); })` to all 6 processors |
| 10 | M | ALL processors | **Duplicate `on('completed')` event handlers.** Every processor registers TWO `on('completed')` handlers: one for debug logging (e.g., notification.processor.ts:66-68) and one for slow-job warning (e.g., notification.processor.ts:76-79). Both fire for every completed job. This is wasteful and confusing -- combine into one handler. | Minor performance overhead (double callback per completed job) and maintenance confusion. | S -- merge into single handler |
| 11 | M | ALL processors | **`on('failed')` calls Sentry on EVERY failure, not just final.** BullMQ's `'failed'` event fires on every attempt failure, including retries. Each processor calls `Sentry.captureException(err)` in the `'failed'` handler (e.g., notification.processor.ts:72). A job with 5 attempts will generate 5 Sentry events. The `moveToDlq` correctly guards on `attemptsMade < maxAttempts`, but Sentry does not. | Sentry event flood. A transient Redis blip causing 100 webhook jobs to fail once each = 500 Sentry events instead of 0 (since they all succeed on retry). Wastes Sentry quota. | S -- only call `Sentry.captureException` when `job.attemptsMade >= (job.opts?.attempts ?? 1)` |
| 12 | M | analytics.processor.ts:46-68 | **Analytics worker has no retry config.** The worker is created without `backoffStrategy` or per-job `attempts` config. Jobs enqueued via `addGamificationJob` set `attempts: 2` with exponential backoff at the Queue level, but the Worker itself has no `settings.backoffStrategy`. For `'custom'` type backoff this would matter. Currently using `'exponential'` so it works, but inconsistent with notification/webhook processors that define custom strategies. | Not a bug today, but fragile -- if someone adds custom backoff to gamification jobs, it will silently use BullMQ's default backoff (0ms). | S |
| 13 | M | search-indexing.processor.ts:46-49 | **Search indexing worker has no retry config.** Same as #12. Worker created without backoff strategy. Jobs use `attempts: 3` with `exponential` delay, which works because BullMQ handles exponential natively. But no custom strategy is defined. | Same as #12 -- fragile if custom backoff is needed later. | S |
| 14 | M | media.processor.ts:56-86 | **Media worker has no retry config.** Worker has no `attempts` or `backoff` in default job options, and the QueueService has no media add method to set per-job retries. If a media job is somehow enqueued (e.g., directly via Redis), it defaults to `attempts: 1` (no retry). A transient network error fetching the image = permanent failure. | No retry on media processing failure. Transient errors cause permanent data loss (no EXIF strip, no BlurHash). | M -- moot until #3 is fixed, but should be addressed together |
| 15 | M | queue.module.ts:70-76 | **No `maxStalledCount` or `stalledInterval` configuration.** BullMQ workers default to `stalledInterval: 30000` and `maxStalledCount: 1`. A job that takes >30s (e.g., processing a large image, slow Meilisearch indexing, slow AI moderation API) will be marked as stalled and re-executed by another worker, causing duplicate processing. The AI processor's limiter (10 calls/min) means jobs could wait in the limiter queue and appear stalled. | Duplicate job execution for long-running jobs. Duplicate push notifications, duplicate webhook deliveries, duplicate AI moderation reports. | M -- set `stalledInterval` and `lockDuration` appropriately for each processor based on expected job duration |
| 16 | M | queue.service.ts:60-70 | **No job deduplication for push notifications.** `addPushNotificationJob` does not set a `jobId` based on `notificationId`. If the same notification is queued twice (e.g., race condition, retry at application level), two push notifications are sent to the user. | Duplicate push notifications to users. | S -- use `{ jobId: notificationId }` in job options for natural deduplication |
| 17 | M | queue.service.ts:93-110 | **No job deduplication for webhooks.** `addWebhookDeliveryJob` does not deduplicate. If a webhook event fires twice for the same entity update (common in event-driven systems), two deliveries go out. | Duplicate webhook deliveries to subscribers. Receivers must implement idempotency themselves. | S -- use `{ jobId: \`${webhookId}:${event}:${hash(payload)}\` }` |
| 18 | L | notification.processor.ts:103 | **`'SYSTEM' as any` type cast.** Line 103 uses `as any` to bypass the Prisma enum type for `type: 'SYSTEM'`. If `SYSTEM` is not a valid `NotificationType` enum value in Prisma, this will fail at runtime with a cryptic DB error. If it IS valid, the cast is unnecessary. | Runtime DB error if enum doesn't match, or unnecessary type suppression. | S -- use the Prisma-generated enum type or verify 'SYSTEM' exists in schema |
| 19 | L | queue.module.ts:54 | **Dynamic require in factory.** Line 54: `new (require('@nestjs/common').Logger)(...)` uses dynamic `require()` inside a factory function. This works but defeats tree-shaking and is inconsistent with the rest of the codebase which uses ES imports. | Code smell. Minor bundle impact. | S -- import Logger at top of file (already imported on line 1 of queue.service.ts) |
| 20 | L | webhook.processor.ts:133 | **Empty catch block.** Line 133: `catch { }` silently swallows errors from `prisma.webhook.update`. While the comment says "Non-critical -- webhook may have been deleted", completely empty catch blocks hide bugs. At minimum, log at debug level. | Silent failure. If this consistently errors (e.g., wrong column name), you'll never know. | S -- add `this.logger.debug(...)` |
| 21 | L | media.processor.ts:160-161 | **S3Client created per-job.** Each `processImageResize` call creates a new `S3Client` instance (line 163) inside the resize loop, once per size variant (3 times per job). S3Client should be created once and reused. | Unnecessary object creation, TCP connection overhead, potential connection pool exhaustion under load. | S -- create S3Client in constructor or lazily once |
| 22 | L | media.processor.ts:221-227 | **Silent `.catch(() => {})` on BlurHash DB writes.** Lines 221-227 silently swallow Prisma update errors for all content types (reel, post, story, video). If the `where: { id: contentId }` fails because the content was deleted between job enqueue and processing, this is fine. But if it fails due to a schema mismatch or connection error, the BlurHash is generated but never persisted -- and nobody knows. | Lost BlurHash data with no trace in logs or monitoring. | S -- log the error at warn level |
| 23 | L | ALL processors | **No `on('stalled')` handler.** No processor listens for the `'stalled'` event. When BullMQ detects a stalled job (see #15), the event fires silently. No logging, no Sentry, no monitoring. | Stalled jobs are invisible to operations. | 24 | I | queue.service.ts:ALL | **No addMediaProcessingJob, addBulkPushJob, addEngagementTrackingJob, or addCaptionGenerationJob.** QueueService exposes 5 methods (push notification, gamification, webhook, search index, moderation) but the processors handle 10+ job types. 5 job types have no producer. | 25 | I | queue.module.ts:70-76 | **Queue connection config is minimal.** The Queue constructor only sets `connection: { url: redisUrl }` and `defaultJobOptions`. | 26 | I | async-jobs.service.ts:ALL | **Deprecated AsyncJobService still injected.** Marked `@deprecated` with zero callers of `enqueue()`, but still instantiated by DI, still injected into HealthController for `getStats()`, and still has `setTimeout`-based retry logic that loses jobs on restart. | 27 | I | queue.service.ts:47-55 | **`onModuleDestroy` only closes Queues, not Workers.** QueueService closes Queue instances (line 48-55), but Workers are closed separately by each processor's own `onModuleDestroy`.
