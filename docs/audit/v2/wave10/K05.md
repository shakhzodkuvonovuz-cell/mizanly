## Findings

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 1 | C | `apps/api/railway.json:6` | **`prisma db push --accept-data-loss` runs on every production build.** This command is explicitly documented by Prisma as destructive — it can drop columns, tables, or rename fields to match the schema, silently deleting production user data. The `--accept-data-loss` flag auto-confirms these destructive changes. If a developer renames a field in schema.prisma, the next deploy wipes that column's data in the live Neon DB. | Production data loss on any schema rename/type change. No rollback possible. | S — Replace with `prisma migrate deploy` which uses versioned migration files and refuses destructive changes unless explicitly written. Build command should be: `npx prisma generate && npx prisma migrate deploy && rm -rf dist && npx nest build`. Requires creating initial migration with `prisma migrate dev`. |
| 2 | C | `apps/e2e-server/Dockerfile:12` | **Container runs as root.** No `USER` directive in runtime stage. The Go binary executes as PID 1 with full root privileges inside the container. If the binary has an RCE vulnerability (e.g., via malformed prekey bundle input), attacker gets root in the container. | Container escape risk, privilege escalation, violates CIS Docker Benchmark 4.1. | S — Add `RUN adduser -D -u 1001 appuser` and `USER appuser` in runtime stage before CMD. |
| 3 | C | `apps/livekit-server/Dockerfile:12` | **Container runs as root.** Same as #2 — no `USER` directive. LiveKit server handles webhook payloads from external sources, making it a higher-risk target for malicious input. | Same as #2. | S — Same fix as #2. |
| 4 | H | `apps/livekit-server/Dockerfile:1` | **Go version mismatch: Dockerfile uses `golang:1.26-alpine` but `go.mod` declares `go 1.25.0`.** Building with a compiler version that doesn't match go.mod can cause subtle behavior differences, new default behaviors, or compilation failures when Go 1.26 introduces breaking defaults (like a new default GOEXPERIMENT). The CI (`.github/workflows/ci.yml:139`) uses Go 1.25 — so CI passes but Docker build may diverge. | Binary built in Docker may behave differently from CI-tested binary. Could silently introduce bugs that only appear in production. | S — Change Dockerfile to `golang:1.25-alpine` to match go.mod. |
| 5 | H | `.github/workflows/ci.yml` | **livekit-server has no CI job at all.** The CI workflow has jobs for `lint-and-typecheck`, `build-mobile`, `test-api`, `test-api-integration`, `build-api`, and `e2e-server` — but zero mention of `livekit-server`. Its 123 tests never run in CI. A regression in livekit-server would be merged to main without detection. | Broken livekit-server code can ship to production uncaught. | M — Add a `livekit-server` CI job identical to the `e2e-server` job but pointing at `apps/livekit-server`. |
| 6 | H | `apps/e2e-server/Dockerfile:1` | **Base image not pinned to digest — uses `golang:1.25-alpine`.** Tag-based references are mutable; the image behind `golang:1.25-alpine` changes when Alpine patches are released. A rebuild weeks later may produce a different binary due to different libc, ca-certificates, or toolchain patches. Same issue for `alpine:3.21` on line 8. | Non-reproducible builds. A "rebuild" may introduce new bugs without any code change. | M — Pin to SHA256 digest: `FROM golang:1.25-alpine@sha256:<digest>`. Or at minimum pin the Alpine patch version. |
| 7 | H | `apps/livekit-server/Dockerfile:1` | **Same unpinned base image issue as #6** — `golang:1.26-alpine` and `alpine:3.21` used without digest. | Same as #6. | M — Same fix as #6. |
| 8 | H | `apps/e2e-server` | **No `.dockerignore` file.** The entire `apps/e2e-server/` directory (including `.env` files if present, `.git` if nested, test files, docs) is sent as Docker build context. While no `.env` currently exists in this directory, any developer who creates one for local testing will have it baked into the Docker image layer, leaking secrets. The `COPY . .` on Dockerfile line 5 copies everything. | Potential secret leakage in image layers. Slower builds due to larger context. | S — Create `.dockerignore` with: `.git`, `*.md`, `*_test.go`, `.env*`, `.dockerignore`, `railway.json`. |
| 9 | M | `apps/api/.dockerignore` | **`.dockerignore` does not exclude `.git` directory.** The livekit-server `.dockerignore` correctly excludes `.git` but the API's does not. While the API uses Nixpacks (not Dockerfile), if a Dockerfile is ever added, the entire `.git` history would be included in the build context. | `.git` directory (potentially large) would bloat build context. Git history could leak commit messages containing sensitive info. | S — Add `.git` to `apps/api/.dockerignore`. |
| 10 | M | `apps/e2e-server/Dockerfile:8`, `apps/livekit-server/Dockerfile:7` | **No `HEALTHCHECK` instruction in Dockerfiles.** While Railway has its own health check config (`healthcheckPath`), the Dockerfiles themselves lack `HEALTHCHECK`. If these containers are ever run outside Railway (local Docker, Kubernetes, ECS), there's no built-in health monitoring. Docker marks the container as always healthy. | Container orchestrators outside Railway won't know when the service is unhealthy. | S — Add `HEALTHCHECK --interval=30s --timeout=3s CMD wget -qO- http://localhost:8080/health || exit 1` (adjust port per service). |
| 11 | M | `apps/api/railway.json:5` | **Build command uses `npm install --legacy-peer-deps` instead of `npm ci`.** `npm install` can modify `package-lock.json` during the build, potentially installing different dependency versions than what was tested in CI. `npm ci` is the correct command for reproducible CI/CD builds — it installs exactly what's in the lockfile and fails if the lockfile is out of sync. | Non-reproducible builds. Production may run different dependency versions than CI tested. | S — Change to `npm ci --legacy-peer-deps`. |
| 12 | M | `apps/api/railway.json:6` | **Build command includes `ls dist/main.js` as a build step.** This `ls` at the end of the build command is a debugging artifact left in the production build configuration. If `main.js` doesn't exist, this produces a non-zero exit that fails the build (which is OK as a guard), but it logs file metadata to build logs and is not the idiomatic way to verify build output. | Minor: debug artifact in production config. The `nest build` already fails if compilation errors exist. | S — Remove `&& ls dist/main.js` or replace with `&& test -f dist/main.js`. |
| 13 | M | `docker-compose.yml:11-12` | **Hardcoded credentials in docker-compose.yml.** `POSTGRES_PASSWORD: mizanly_dev` and `MEILI_MASTER_KEY: mizanly_dev_master_key` are committed to the repo. While the file header says "Local development only", these credentials could be copy-pasted to staging/production by mistake. Anyone with repo access knows the dev DB password. | Credential exposure in version control. If dev DB is accessible externally, it's compromised. | S — Use `.env` file for docker-compose variables: `POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-mizanly_dev}` and add a `docker-compose.env.example`. |
| 14 | M | `docker-compose.yml:18-19` | **Postgres port bound to 0.0.0.0:5432.** The `ports: ['5432:5432']` binding exposes PostgreSQL to all network interfaces, not just localhost. On a developer machine connected to public WiFi, the database is accessible from the network. | Dev database exposed to LAN/internet on the developer's machine. | S — Change to `'127.0.0.1:5432:5432'`. Same for Redis (`127.0.0.1:6379:6379`) and Meilisearch (`127.0.0.1:7700:7700`). |
| 15 | M | `docker-compose.yml:1` | **Uses deprecated `version: '3.9'` key.** Docker Compose V2 (which is now the default) ignores the `version` key entirely and logs a deprecation warning. This is cosmetic but signals the file hasn't been updated for modern Docker. | Deprecation warning on every `docker compose up`. Misleads developers about which Compose features are available. | S — Remove the `version: '3.9'` line. |
| 16 | M | `apps/e2e-server/Dockerfile`, `apps/livekit-server/Dockerfile` | **Go binaries are not stripped.** The `go build` commands don't include `-ldflags="-s -w"` to strip debug symbols and DWARF info. This makes the binary 20-30% larger than necessary and includes symbol names that help attackers reverse-engineer the code. | Larger container images, debug info available to attackers who extract the binary. | S — Change to `go build -ldflags="-s -w" -o /server ./cmd/server/`. |
| 17 | L | `apps/api/railway.json:9` | **`restartPolicyMaxRetries: 10` is excessive.** If the API crashes 10 times in a row, there is a systemic issue (DB down, missing env var, OOM). Restarting 10 times just delays the alert while the service repeatedly crashes. The livekit-server uses 3, e2e-server uses 5 — API should be comparable. | Delayed incident detection. Service flaps for minutes before Railway gives up. | S — Reduce to 5. |
| 18 | L | `apps/livekit-server/railway.toml`, `apps/e2e-server/railway.json` | **Inconsistent Railway config formats.** livekit-server uses `railway.toml`, e2e-server uses `railway.json`, API uses `railway.json`. Pick one format for consistency. | Developer confusion, harder to audit. | S — Convert `railway.toml` to `railway.json` for consistency, or vice versa. |
| 19 | L | `apps/e2e-server/railway.json:6` | **Start command `/e2e-server` duplicates the Dockerfile CMD.** The Dockerfile already specifies `CMD ["/e2e-server"]`. The Railway `startCommand` overrides this, making the Dockerfile CMD dead code. If someone updates one but not the other, they'll be confused. | Confusion about which command actually runs. | S — Remove `startCommand` from railway.json (let Dockerfile CMD take effect) or remove CMD from Dockerfile and rely solely on Railway config. |
| 20 | L | `apps/api/railway.json` | **No healthcheckTimeout configured for API.** The livekit-server railway.toml sets `healthcheckTimeout = 30` but the API railway.json only sets `healthcheckPath` without a timeout. Railway's default is 300 seconds, which means a hung health endpoint won't be detected for 5 minutes. | Slow detection of unhealthy API instances. | S — Add `"healthcheckTimeout": 30` to the deploy section. |
| 21 | I | `apps/e2e-server/Dockerfile`, `apps/livekit-server/Dockerfile` | **No `LABEL` metadata in Dockerfiles.** Best practice is to include `LABEL maintainer`, `LABEL version`, `LABEL description` for image identification in registries and `docker inspect` output. | Hard to identify image purpose and owner in a container registry. | S — Add appropriate LABEL directives. |
| 22 | I | `apps/api/src/main.ts` | **No Dockerfile for the API service.** The API relies entirely on Railway's Nixpacks builder, which auto-detects the build process. This means the build is not reproducible outside Railway — you cannot `docker build` the API locally or deploy it to any other container platform (ECS, GKE, Fly.io) without creating a Dockerfile. | Vendor lock-in to Railway. Cannot reproduce the production build environment locally for debugging. | M — Create a multi-stage Dockerfile for the API service for portability. Low priority while Railway works. |
| 23 | I | `docker-compose.yml` | **No docker-compose service for the Go microservices.** The compose file includes postgres, redis, meilisearch but not the e2e-server or livekit-server. Developers must run these manually during local development. | Incomplete local development environment. | M — Add e2e-server and livekit-server services with Dockerfile builds and proper networking. |

### Critical Path

The single most dangerous finding is **#1**: `prisma db push --accept-data-loss` in the production Railway build. This is a ticking time bomb. The moment anyone renames a Prisma schema field, the next deploy silently drops that column from the production database. Replace with `prisma migrate deploy` immediately. This requires a one-time setup of Prisma migrations (`npx prisma migrate dev --name init`) but is non-negotiable for a production service.

Findings #2 and #3 (root containers) are standard security hardening — 2-line fixes each.
