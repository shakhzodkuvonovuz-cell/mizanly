## Summary by Check

### 1. Overlap prevention
- **1 of 27** crons has a distributed lock (`publishScheduledMessages`).
- **26 of 27** have zero overlap prevention.
- The high-frequency crons (every minute: #1, #2, #3) are the most critical.

### 2. Error handling
- **All 27** crons have try/catch at the top level. None will crash the server.
- Errors are caught and logged. Good.

### 3. Sentry logging on failure
- **All 27** crons call `Sentry.captureException(error)` in their catch blocks. Good.
- **2 crons** (#12 `sendVerseOfTheDay`, #13 `checkIslamicEventReminders`) have early-return paths that silently skip without Sentry.

### 4. Idempotency
- **Good:** Most DB operations use `updateMany` (idempotent), `upsert` (idempotent), or `deleteMany` (idempotent).
- **Bad:** Side effects (notifications, gamification XP, push delivery queuing) are NOT idempotent — they fire every time the cron runs, leading to duplicates on multi-instance.
- **Critical:** `sendVerseOfTheDay` picks a DIFFERENT random verse on each instance, so multi-instance produces two different verse notifications.

### 5. Distributed locks (Redis NX) for multi-instance
- **1 of 27** has a Redis NX lock (`publishScheduledMessages`).
- **1** has a Redis dedup key but with a TOCTOU race (`checkIslamicEventReminders`).
- **1** has a Redis dedup key that works correctly (`sendWeeklyScreenTimeDigest`).
- **24** have zero distributed locking.

### 6. Active and registered
- All 27 crons are active. `ScheduleModule.forRoot()` is registered. All service classes are in modules imported by `app.module.ts`.
- No dead/orphaned crons found.

### 7. Dead crons / missing crons
- **Missing:** Story cleanup, DM note cleanup, circle invite cleanup, download cleanup, cron health monitoring.
- **Duplicate:** `notifyScheduledPostsPublished` duplicates notification logic already in `publishOverdueContent`.

## Severity Counts

**C:5 H:3 M:12 L:8 I:7 Total: 35**

The 5 Critical findings (#1-5) are all the same root cause: raw SQL UPDATE statements using Prisma model names (`"Post"`, `"Reel"`, `"Thread"`, `"Video"`, `"Hashtag"`) instead of actual PostgreSQL table names (`"posts"`, `"reels"`, `"threads"`, `"videos"`, `"hashtags"`). These cause every counter reconciliation UPDATE to fail at runtime with `relation "X" does not exist`, meaning counter drift is never corrected despite the cron appearing to work (the SELECT succeeds, finds drifted rows, but the UPDATE silently fails and the catch block swallows the error).
