## Analysis Notes

### What the code does well
- All SQL queries use parameterized placeholders (`$1`, `$2`, etc.) — no SQL injection vectors found
- Auth middleware is consistently applied to all authenticated routes in main.go routing table
- Rate limiting is present on the two highest-risk endpoints (room creation, token generation)
- SDK calls consistently use `context.WithTimeout` with `defer cancel()` — no context leak
- The advisory lock pattern in `CreateCallSession` (sorted by user ID) correctly prevents AB-BA deadlock
- E2EE key generation uses `crypto/rand` (CSPRNG), not `math/rand`
- `decodeBody` uses `io.LimitReader(r.Body, maxBodySize)` to prevent request body amplification
- `WipeE2EEKey` sets both key and salt to NULL (not just empty bytes)
- Webhook deduplication via Redis SETNX prevents replay attacks
- The F8/F11 fixes (caller-only deletion, kick target validation) are correctly implemented
- `resp.Body.Close()` is called immediately after checking the response in `postInternalPush`

### Patterns of concern
1. **Silent error swallowing** is the dominant pattern. At least 8 places discard DB operation errors with `_`. This violates the project's own "All errors explicit" rule and creates silent data inconsistency.
2. **No cleanup for orphaned ACTIVE calls.** The `CleanupStaleRingingSessions` ticker only handles RINGING sessions older than 60s. ACTIVE calls that become orphaned (all participants left but race condition prevented status update) persist indefinitely. A second cleanup ticker for ACTIVE sessions with 0 remaining participants is needed.
3. **E2EE key material handling is good in the happy path but fragile on errors.** Key generation, distribution, and wiping are all present, but every error path that skips `WipeE2EEKey` leaves key material in the DB.
