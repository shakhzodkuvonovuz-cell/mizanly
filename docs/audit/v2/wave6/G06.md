# G06 — Infrastructure Audit: livekit-server store, config, middleware, model, main, Dockerfile

**Auditor:** Claude Opus 4.6 (1M context)
**Scope:** `apps/livekit-server/internal/store/`, `internal/config/`, `internal/middleware/`, `internal/model/`, `cmd/server/main.go`, `Dockerfile`
**Date:** 2026-03-30

## Findings

| # | Severity | File | Line(s) | Finding | Detail |
|---|----------|------|---------|---------|--------|
| 1 | **CRITICAL** | `cmd/server/main.go` | 73-75 | Redis TLS forced on non-TLS URLs — breaks local/non-TLS connections | `redis.ParseURL` sets `TLSConfig` only for `rediss://` scheme. The code checks `if redisOpt.TLSConfig == nil` and forces TLS, meaning `redis://localhost:6379` gets TLS forced on it and the connection will fail with a TLS handshake error. In development or any non-TLS Redis setup, the server will exit on startup. Should only set TLS config when the URL scheme is `rediss://`, or remove the override entirely and trust `ParseURL`. |
| 2 | **CRITICAL** | `Dockerfile` | 1 | Invalid Go version `golang:1.26-alpine` — does not exist | `go.mod` declares `go 1.25.0`. The Dockerfile uses `golang:1.26-alpine` which is a nonexistent image tag (Go latest stable as of early 2026 is 1.25.x). Docker build will fail with `manifest unknown`. Should be `golang:1.25-alpine`. |
| 3 | **HIGH** | `Dockerfile` | 1-12 | Container runs as root — no `USER` directive | The final stage has no `RUN adduser` / `USER` directive. The `/server` binary runs as `root` inside the container. If an attacker exploits a vulnerability, they have full root access. Standard practice is `RUN adduser -D appuser` + `USER appuser`. |
| 4 | **HIGH** | `internal/store/store.go` | 196-209 | `UpdateSessionStatus` does not validate status parameter — allows arbitrary string injection into DB | The `status` parameter is a raw string with no validation against an allowed set. Although it uses parameterized queries (no SQL injection), any caller can write arbitrary status values (e.g., `"HACKED"`) into the `status` column. The handler does validate in some paths, but this is defense-in-depth missing at the store layer. The column is an enum in Prisma (`CallStatus`), so PostgreSQL will reject invalid values, but the error will be opaque (`invalid input value for enum`). Add a Go-side validation. |
| 5 | **HIGH** | `cmd/server/main.go` | 193-199 | Goroutine calls `os.Exit(1)` — bypasses deferred cleanup | If `ListenAndServe` fails (e.g., port already in use), the goroutine calls `os.Exit(1)` which skips ALL deferred functions: `sentry.Flush`, `db.Close`, `rdb.Close`. This causes unclean shutdown. Should signal the main goroutine to exit instead (e.g., send to `quit` channel or `log.Fatal` from main after receiving error). |
| 6 | **HIGH** | `internal/store/store.go` | 100-193 | `CreateCallSession` holds transaction + advisory locks for N user existence checks | The transaction acquires advisory locks for ALL participants, then the handler (not the store) does N sequential `UserExists` checks BEFORE calling `CreateCallSession`. But inside `CreateCallSession`, the sorted advisory locks + active-call checks run sequentially. For a 30-person group call, this holds the transaction open for 30 sequential queries. Under high concurrency this could cause connection pool exhaustion (pool max is 10). |
| 7 | **MEDIUM** | `internal/store/store.go` | 388-472 | `GetHistory` — `DISTINCT ON` with `ORDER BY DESC` requires a subquery or it may error | PostgreSQL requires that `DISTINCT ON` expressions match the leftmost `ORDER BY` expressions in direction. `DISTINCT ON (cs."createdAt", cs.id)` with `ORDER BY cs."createdAt" DESC, cs.id DESC` works in practice because the DISTINCT ON columns match the ORDER BY columns, but the result ordering within each DISTINCT group is undefined. This is technically correct but fragile — if someone adds a third ORDER BY column or changes the DISTINCT ON, it will break silently. |
| 8 | **MEDIUM** | `internal/middleware/ratelimit.go` | 42-49 | `CheckCreateRoom` silently passes when Redis is nil | If `rl.rdb == nil`, rate limiting is bypassed with `return nil`. The comment says "No Redis in test mode" but in production, if Redis becomes nil (e.g., failed initialization path), all rate limits are silently disabled. Defense-in-depth: fail closed in production. |
| 9 | **MEDIUM** | `internal/store/store.go` | 262-272 | `GetActiveParticipantCount` counts only `livekitJoinedAt IS NOT NULL` — misses pre-join state | A participant who has a DB record but hasn't connected to LiveKit yet (`livekitJoinedAt IS NULL`) is not counted. This is used by the webhook handler to check if `count >= 2` to transition from RINGING to ACTIVE. If both participants have DB records but only one has connected to LiveKit, the transition is delayed. This is probably intentional but is not documented. |
| 10 | **MEDIUM** | `internal/store/store.go` | 592-604 | `SplitCursor` does not validate the timestamp portion | `SplitCursor` splits on `|` but never validates that the first part is a valid RFC3339 timestamp. A malformed cursor like `not-a-date|some-id` would be passed directly to the SQL query as `$2::timestamptz`, causing a PostgreSQL cast error. The error propagates as a generic 500 instead of a 400. |
| 11 | **MEDIUM** | `cmd/server/main.go` | 100-127 | CORS middleware does not set `Access-Control-Max-Age` | Preflight responses lack `Access-Control-Max-Age`, causing browsers to send a preflight OPTIONS request before every cross-origin request. This doubles request count for non-simple requests. Should add `Access-Control-Max-Age: 86400` to cache preflight results. |
| 12 | **MEDIUM** | `internal/middleware/requestid.go` | 14-15 | Trusts client-supplied `X-Request-ID` without validation | If the client sends `X-Request-ID: <script>alert(1)</script>`, that value is stored in context, set as a response header, and logged. While XSS via headers is low-risk, log injection is possible. Should validate format (hex/UUID only) or always generate server-side. |
| 13 | **MEDIUM** | `internal/store/store.go` | 236-242 | `WipeE2EEKey` sets key to NULL but does not verify the update affected a row | `Exec` returns a `CommandTag` with `RowsAffected()`, but it's ignored. If the session ID doesn't exist, the wipe silently succeeds. For a security-critical operation (key destruction), the caller should verify exactly 1 row was affected. |
| 14 | **MEDIUM** | `cmd/server/main.go` | 55 | `clerk.SetKey` uses package-level global state — not test-safe | `clerk.SetKey()` mutates a package-level variable. If multiple tests or goroutines call this, it's a data race. Not an issue for the single production binary, but problematic for integration tests. |
| 15 | **LOW** | `internal/store/store.go` | 36-39 | Connection pool `MaxConns: 10` is hardcoded — not configurable | For production, 10 max connections may be insufficient under load (especially with the 30-participant group call path holding transactions). This should be configurable via environment variable. Neon's free tier allows 100 connections; paid plans allow more. |
| 16 | **LOW** | `internal/config/config.go` | 28-69 | No validation on port number format or range | `Port` is read as a string with no numeric validation. A typo like `PORT=abc` will cause `http.Server` to fail at `ListenAndServe` with an opaque address parse error. Should validate it's a number in 1-65535 at config load time. |
| 17 | **LOW** | `internal/model/types.go` | 7-25 | `CallSession` model does not include `e2eeKey`/`e2eeSalt` fields | The Prisma schema has `e2eeKey Bytes?` and `e2eeSalt Bytes?` on `call_sessions`, but the Go `CallSession` struct omits them. This means `GetSessionByID` and `GetSessionByRoomName` can never accidentally leak key material in JSON responses (the struct simply doesn't have the fields). This is GOOD security design but should be documented as intentional. |
| 18 | **LOW** | `internal/store/store.go` | 506-523 | `CleanupStaleRingingSessions` CTE: `updated_participants` result is unused | The CTE `updated_participants` does an `UPDATE` but its result is never referenced. PostgreSQL still executes it (CTEs are optimization fences in older versions), but the naming suggests the author expected to return affected participant count. This is functionally correct but could mislead future readers. |
| 19 | **LOW** | `Dockerfile` | 6 | Build does not use `-ldflags="-s -w"` — binary includes debug symbols | Without strip flags, the binary is ~30% larger than necessary and includes symbol/debug information that aids reverse engineering. Add `CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /server ./cmd/server/` |
| 20 | **LOW** | `Dockerfile` | 8 | Final image uses `alpine:3.21` — should pin with `@sha256:` digest for reproducibility | Using a tag alone is mutable (the registry can update the image behind the tag). For production, pin with a sha256 digest: `FROM alpine:3.21@sha256:<digest>`. |
| 21 | **LOW** | `cmd/server/main.go` | 164-172 | `http.Server` has no `ErrorLog` configured | Without `ErrorLog`, the server logs HTTP errors (bad requests, TLS failures) to the default `log` package logger, which outputs unstructured text to stderr. This bypasses the structured `slog.Logger` configured for the rest of the application. Set `ErrorLog: slog.NewLogLogger(logger.Handler(), slog.LevelWarn)`. |
| 22 | **LOW** | `internal/store/store.go` | 316-336 | `getActiveParticipantsLight` does not scan `livekitJoinedAt` — inconsistent with `getParticipants` | `getActiveParticipantsLight` SELECTs 5 columns (`sessionId, userId, role, joinedAt, leftAt`) and scans into the same `CallParticipant` struct that has a `LivekitJoinedAt` field. The field is left as zero value. Meanwhile, `getParticipants` and `getParticipantsBatch` SELECT and scan 6 columns including `livekitJoinedAt`. This inconsistency could cause bugs if callers expect `LivekitJoinedAt` to be populated. |
| 23 | **INFO** | `internal/store/store.go` | 87-93 | `UserExists` queries by `id` on `users` table without verifying `isBanned` | A banned user (`isBanned = true`) still passes the existence check and can be added as a call participant. Whether banned users should be callable is a product decision, but currently there's no enforcement. |
| 24 | **INFO** | `internal/middleware/auth.go` | 18-32 | Auth middleware delegates entirely to Clerk SDK — no timing-safe comparison needed | The `RequireAuth` middleware uses `clerkhttp.RequireHeaderAuthorization()` which handles JWT verification internally via the Clerk SDK. No manual string comparison is done, so timing attacks are not applicable here. The internal service key comparison (`X-Internal-Key` in handler.go) uses `http.DefaultClient` + Clerk SDK on the NestJS side, not this middleware. This is correctly structured. |
| 25 | **INFO** | `internal/store/store.go` | 1-616 | All SQL queries use parameterized queries (`$1`, `$2`, etc.) — no SQL injection vectors found | Every query uses `pgx` parameterized placeholders. No string concatenation or `fmt.Sprintf` is used to build SQL. The `ErrUserInCall.UserID` field is not included in any SQL — it's only in the error message struct. SQL injection risk: **none detected**. |

### Critical items requiring immediate action:
1. **Dockerfile Go version mismatch** (#2) — build is completely broken, cannot produce container
2. **Redis TLS forced on non-TLS URLs** (#1) — server fails to start in development or any non-TLS Redis environment

### Security positives noted:
- All SQL is parameterized (no injection vectors)
- E2EE key/salt fields deliberately omitted from the Go model struct (prevents JSON serialization leaks)
- E2EE key wipe is called on every call-end path (ENDED, MISSED, DECLINED, cleanup)
- Advisory lock ordering prevents deadlocks in concurrent room creation
- Webhook HMAC validation delegated to LiveKit SDK (no manual comparison needed)
- Rate limiting fails closed on Redis errors
- Request body size limited to 64KB
- HTTP timeouts properly configured (read, write, idle, header)
