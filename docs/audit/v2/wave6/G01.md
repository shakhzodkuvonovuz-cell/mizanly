## Findings

| 1 | M | handler.go:513-514 | **`defer rCancel()` inside for-loop leaks contexts.** In `notifyIdentityChanged`, the retry loop at lines 506-515 creates a new `context.WithTimeout` on each retry attempt (attempts 1 and 2) and calls `defer rCancel()`. `defer` executes when the *function* returns, not when the loop iteration ends. This means: (a) the context from attempt 1 is NOT cancelled when attempt 2 starts, (b) up to 2 context timers remain live simultaneously. With max 3 iterations the leak is bounded, but it is a correctness bug — the HTTP request made on attempt 1's context remains cancellable by attempt 1's timer even after the code has moved on to attempt 2. **Fix:** call `rCancel()` explicitly at the end of each iteration, or restructure to call cancel before creating the next context. | Context leak; stale timer could cancel a later retry's in-flight request if the function takes long enough | Small |
| 2 | M | handler.go:188-205 | **`HandleGetBundle` has no defense-in-depth auth check.** Unlike every other mutating/sensitive handler (`HandleRegisterIdentity`, `HandleUploadSignedPreKey`, `HandleGetPreKeyCount`, etc.), `HandleGetBundle` does NOT call `middleware.UserIDFromContext(r.Context())` to verify the requester is authenticated. It relies entirely on the auth middleware wired in `main.go` (line 129). If routing is ever refactored or the handler is mounted without auth middleware (e.g., test server, new router), bundle fetches become unauthenticated. The handler consumes OTP keys on every call, so unauthenticated access would enable OTP pool draining. | OTP drain if handler mounted without auth middleware | Small |
| 3 | M | handler.go:208-267 | **Batch bundle `req.UserIDs` not validated for length/chars.** `HandleGetBundlesBatch` checks `len(req.UserIDs) > 100` (line 221) and deduplicates (lines 237-245), but individual userID strings are never validated with `validatePathParam` or equivalent. A request with userIDs containing megabyte-length strings, null bytes, or control characters passes directly to `h.store.GetPreKeyBundle` which executes SQL queries. The pgx parameterized query prevents SQL injection, but oversized strings waste DB query time and memory, and control characters in userIDs could pollute structured logs if any downstream code logs them. | DoS via oversized userID strings in batch; log injection via control chars | Small |
| 4 | L | handler.go:322-347 | **`HandleGetSenderKeys` does not verify group membership.** `HandleStoreSenderKey` (line 306-307) correctly calls `h.store.VerifyGroupMembership` before allowing a sender key write. However, `HandleGetSenderKeys` does not perform the same check. A user who was removed or banned from a group can still fetch sender keys that were addressed to them. The store filters by `recipientUserId`, so they only see their own keys, but a banned user should arguably lose access to newly distributed sender keys. This is an inconsistency between the write and read paths. | Banned/removed user can still read sender keys addressed to them | Small |
| 5 | L | handler.go:73-74, 127-128, 157-158 | **Silent clamping of out-of-range `deviceID` instead of rejecting.** Three handlers (`HandleRegisterIdentity`, `HandleUploadSignedPreKey`, `HandleUploadOneTimePreKeys`) silently default `deviceID` to 1 when the value is < 1 or > 10. This masks client bugs: a client sending `deviceId: 0` or `deviceId: -1` gets a silent success on device 1, potentially overwriting device 1's keys without realizing it. The store layer validates key material strictly (returns errors for bad base64, wrong sizes), but deviceID is silently corrected. An explicit 400 error would surface client integration bugs faster. | Silent key overwrite on device 1 when client sends invalid deviceID | Small |
| 6 | L | handler.go:571-581 | **`concat` function is dead code.** The `concat` helper (lines 571-581) is defined and has 4 test cases in `handler_test.go`, but is never called anywhere in the handler file or any other file in the codebase (`grep` confirms zero call sites outside tests). Dead code increases maintenance burden and confuses readers about its purpose. | Dead code in production module | Trivial |
| 7 | I | handler.go:460-529 | **`notifyIdentityChanged` goroutine has no `recover`.** The goroutine launched at line 102 (`go h.notifyIdentityChanged(...)`) has no `defer recover()`. If any panic occurs (e.g., nil pointer in JSON marshal, slice bounds), the entire server process crashes. The function is defensively written and unlikely to panic today, but Go best practice for fire-and-forget goroutines is to include a recover to prevent one webhook notification from taking down the server. | Server crash on unexpected panic in webhook goroutine | Small |
| 8 | I | handler.go:550-554 | **`writeJSON` silently swallows `json.Encode` errors.** `json.NewEncoder(w).Encode(v)` at line 554 can fail (e.g., if `v` contains a channel, func, or the connection is broken). The error is silently discarded. If encoding fails after `WriteHeader` has been called (line 553), the client receives a truncated or empty response with a 200 status code. This is rare but violates the project's "All errors explicit, never silently swallowed" rule. | Client receives truncated JSON with 200 status on encode failure | Small |
| 9 | I | handler.go:560-569 | **`extractPathParam` is fragile with URL-encoded paths.** `extractPathParam` uses `r.URL.Path` which is the decoded path. However, if a userId contains `%2F` (encoded slash), it gets decoded to `/` which `TrimRight(param, "/")` at line 566 would strip. Combined with `validatePathParam` allowing most printable characters, a userId like `user%2F123` would be silently truncated to `user` (everything after the decoded `/` is treated as a sub-path by the mux, so `extractPathParam` never sees it). This is an edge case since Clerk userIDs are controlled format, but the pattern is brittle for any future path parameter extraction. | Silent truncation of userIDs containing encoded slashes | Trivial |
| 10 | I | handler.go:92-96 | **`UpsertIdentityKey` store error mapped to 400 without distinguishing error types.** Line 95 returns `http.StatusBadRequest` for all store errors, but `UpsertIdentityKey` can fail for reasons beyond bad input: DB connection timeout, transaction failure, constraint violation on duplicate concurrent writes. A DB timeout is a 500, not a 400. The generic error message to the client is correct (no info leak), but the status code is misleading for monitoring/alerting dashboards that track 4xx vs 5xx rates. | Incorrect 4xx status for server-side failures; misleading error rate metrics | Small |
