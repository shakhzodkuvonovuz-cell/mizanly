## Findings

### CRITICAL

| ID | Finding | File | Line(s) | Details |
|----|---------|------|---------|---------|
| G02-C1 | **Schema mismatch: `ON CONFLICT ("userId", "deviceId")` but Prisma schema has `@unique` on `userId` only** | e2e-server/store/postgres.go | 121-128 | `UpsertIdentityKey` uses `ON CONFLICT ("userId", "deviceId")` which requires a unique index on that pair. But the Prisma schema at `schema.prisma:3024` declares `userId String @unique` (single column). There is NO `@@unique([userId, deviceId])` constraint. This means: (1) the `ON CONFLICT` clause will fail at runtime with `there is no unique or exclusion constraint matching the ON CONFLICT specification`, or (2) if the DB was manually migrated to have the composite unique, Prisma migrations will try to revert it. Either way, the multi-device identity key story is broken in production. The comment on line 115 says "Requires a UNIQUE index on (`userId`,`deviceId`) in the DB schema" but this was never added to the Prisma schema. |

### HIGH

| ID | Finding | File | Line(s) | Details |
|----|---------|------|---------|---------|
| G02-H1 | **`GetTransparencyProof` returns `time.Now()` as UpdatedAt, inconsistent with `GetTransparencyRoot`** | e2e-server/store/postgres.go | 658 vs 685 | `GetTransparencyProof` (line 658) returns `time.Now().UTC()` as `UpdatedAt`, while `GetTransparencyRoot` (line 685) correctly returns `s.cacheRebuiltAt`. This means the proof endpoint always looks fresh even when the cache is stale, defeating the V6-F13 fix that was specifically added to prevent this. A compromised server could serve an old Merkle tree via the proof endpoint and the client would never notice staleness from the timestamp. |
| G02-H2 | **`SELECT COUNT(*) ... FOR UPDATE` locks ALL matching rows in `InsertOneTimePreKeys`** | e2e-server/store/postgres.go | 228-231 | `SELECT COUNT(*) FROM e2e_one_time_pre_keys WHERE "userId" = $1 AND "deviceId" = $2 FOR UPDATE` acquires row-level exclusive locks on every OTP row for that user+device. With up to 500 OTPs, this locks 500 rows. Any concurrent `ClaimOneTimePreKey` (which also uses `FOR UPDATE`) for the same user will block until the insert transaction commits. This creates unnecessary contention between OTP upload and OTP claim operations. A `SELECT COUNT(*) ... FOR SHARE` would suffice since the count check only needs read consistency, or better yet use an advisory lock on the user+device pair instead of locking every individual row. |
| G02-H3 | **`SimpleProtocol` mode sends SQL as plain text strings -- all `$1` parameters are string-interpolated by pgx, not server-side prepared** | e2e-server/store/postgres.go | 58 | Both servers use `QueryExecModeSimpleProtocol` (required for Neon pooler). In this mode, pgx does NOT use PostgreSQL prepared statements. Instead, parameters are escaped and interpolated client-side by pgx before sending as a simple query string. While pgx's escaping is well-tested and not vulnerable to trivial SQL injection, this means the wire protocol has no server-side parameter binding. If a pgx bug is ever found in its escaping logic, all queries would be vulnerable. This is a known trade-off with Neon, but should be documented as a trust dependency on pgx's client-side escaping correctness. Not an active vulnerability, but an elevated risk posture. |
| G02-H4 | **`UpdateSessionStatus` does not verify the session exists or that the status transition is valid** | livekit-server/store/store.go | 196-209 | `UpdateSessionStatus` blindly executes `UPDATE ... WHERE id = $1` without checking `RowsAffected()`. If the session ID doesn't exist, it silently succeeds (updates 0 rows). Additionally, there is no validation of status transitions (e.g., preventing `ENDED` -> `RINGING`). Any status can be set on any session. The handler may enforce some of this, but the store layer should be defensive. |

### MEDIUM

| ID | Finding | File | Line(s) | Details |
|----|---------|------|---------|---------|
| G02-M1 | **No context timeout on `rebuildMerkleCacheLocked` -- holds write lock for duration of full table scan** | e2e-server/store/postgres.go | 537-599 | `rebuildMerkleCacheLocked` reads ALL rows from `e2e_identity_keys` while holding `s.mu` write lock. If the DB is slow or the table is large, ALL concurrent `GetTransparencyProof` and `GetTransparencyRoot` calls block. The query has no `LIMIT` or timeout. With 100K+ users, this could block the entire transparency subsystem for seconds. Should use a derived context with timeout, or build the new cache in a temporary variable and swap atomically. |
| G02-M2 | **`CleanupExpiredSignedPreKeys` uses a correlated subquery that may be slow without index on `createdAt`** | e2e-server/store/postgres.go | 468-482 | The `DELETE ... WHERE id NOT IN (SELECT DISTINCT ON (...) id ... ORDER BY "createdAt" DESC)` subquery requires sorting all SPKs per (userId, deviceId). The Prisma schema has `@@index([userId, deviceId])` but not a composite index including `createdAt`. On large tables, `DISTINCT ON` with `ORDER BY createdAt DESC` will require a sort per group. Additionally, `NOT IN` with a subquery performs poorly compared to `NOT EXISTS` or a `LEFT JOIN ... WHERE NULL` pattern in PostgreSQL. |
| G02-M3 | **`GetHistory` uses `DISTINCT ON` with composite cursor pagination -- potential for incorrect deduplication** | livekit-server/store/store.go | 399-420 | The query uses `SELECT DISTINCT ON (cs."createdAt", cs.id)` but `DISTINCT ON` requires that the `ORDER BY` leading columns match the `DISTINCT ON` columns exactly. The query has `ORDER BY cs."createdAt" DESC, cs.id DESC` which matches, so it works. However, `DISTINCT ON` is technically unnecessary here since `cs.id` is already unique (it's the primary key). The JOIN with `call_participants` could produce duplicates (if a user appears multiple times in participants for the same session, which shouldn't happen due to `@@id([sessionId, userId])`). The `DISTINCT ON` is defensive but adds sort overhead on every history query. |
| G02-M4 | **`MarkParticipantLeft` and `MarkAllParticipantsLeft` don't check `RowsAffected` -- silent no-ops** | livekit-server/store/store.go | 246-260 | Both functions return `nil` error even if no rows matched. `MarkParticipantLeft` for a nonexistent user/session combination silently succeeds. The caller (handler) may need to know if the participant was actually found and updated. |
| G02-M5 | **`GetDeviceIDs` returns `[1]` when no keys exist -- masks missing registration** | e2e-server/store/postgres.go | 507-509 | When a user has no registered identity keys, `GetDeviceIDs` returns `[1]` instead of an empty list. This makes it impossible for callers to distinguish "user has device 1 registered" from "user has no devices at all." A bundle fetch for this phantom device 1 will then fail with a confusing "identity key not found" error instead of "user has no registered devices." |

### LOW

| ID | Finding | File | Line(s) | Details |
|----|---------|------|---------|---------|
| G02-L1 | **`UpsertIdentityKey` RETURNING subquery may return the NEW value, not the OLD value** | e2e-server/store/postgres.go | 120-129 | The `RETURNING (SELECT "publicKey" FROM e2e_identity_keys WHERE "userId" = $1 AND "deviceId" = $2)` subquery executes AFTER the upsert. In PostgreSQL, the RETURNING clause sees the row as it is after the INSERT/UPDATE. The subquery reads from the same table, but due to MVCC snapshot isolation, it should see the post-update value. However, since the subquery uses a separate SELECT (not `EXCLUDED` or `OLD`), behavior depends on the transaction isolation level and PostgreSQL version. The intent is to get the previous value, but this is fragile. A safer approach: use a CTE to first SELECT the old key, then perform the upsert. |
| G02-L2 | **Merkle tree leaf construction includes userId as raw string -- no length prefix** | e2e-server/store/postgres.go | 561-565 | Leaf data is `0x00 || userId || publicKey`. Since userId is a variable-length CUID string (typically 25-28 chars), and publicKey is always 32 bytes, there is a theoretical second-preimage concern: a userId "abc" + key starting with "d..." would hash the same as userId "abcd" + key starting with "...". In practice this is near-impossible since CUIDs have fixed format, but a length-delimited construction (e.g., `0x00 || len(userId) || userId || publicKey`) would be formally correct per RFC 6962 leaf format expectations. |
| G02-L3 | **`notifyIdentityChanged` runs in a goroutine with no context propagation** | e2e-server/handler/handler.go | 102 | `go h.notifyIdentityChanged(userID, oldFP, req.PublicKey)` launches a fire-and-forget goroutine. It creates its own `context.Background()` timeouts internally (good), but if the server is shutting down via `context.Done()`, these goroutines may outlive the shutdown deadline. Consider using a `sync.WaitGroup` or bounded worker pool for graceful shutdown. |
| G02-L4 | **`GetSenderKeys` has no pagination -- returns ALL sender keys for a group+recipient** | e2e-server/store/postgres.go | 338-362 | If a group has many senders and frequent key rotations, this query could return thousands of rows. There is no `LIMIT` clause. In practice, groups are unlikely to have thousands of distinct senders, but a malicious actor who joins a group and rapidly rotates sender keys could bloat the response. |
| G02-L5 | **`ErrUserInCall.Error()` does not include the UserID -- debugging difficulty** | livekit-server/store/store.go | 613-615 | The error message is always "user is already in a call" without including the user ID. When multiple users are checked in `CreateCallSession`, the caller only knows SOME user was in a call, not which one. The handler at least gets the structured error and can type-assert for `.UserID`, but the `.Error()` string itself is unhelpful for logging. |

## Positive Observations

| Area | Detail |
|------|--------|
| SQL injection | All queries use parameterized `$1, $2, ...` placeholders. No string concatenation for SQL. Even with SimpleProtocol, pgx handles escaping correctly. |
| Row cleanup | `rows.Close()` is called via `defer` in all multi-row query functions (`GetSenderKeys`, `GetDeviceIDs`, `rebuildMerkleCacheLocked`, `getParticipants`, `getParticipantsBatch`, `GetHistory`). No resource leaks. |
| Transaction discipline | `InsertOneTimePreKeys` and `GetPreKeyBundle` both use transactions with `defer tx.Rollback()`. OTP claim within bundle fetch prevents partial failure wasting keys. |
| Batch operations | `InsertOneTimePreKeys` uses `pgx.Batch` for efficient bulk insert. `SendBatch` results are iterated and errors checked. `br.Close()` is called on both success and error paths. |
| Concurrency | Merkle cache uses proper `sync.RWMutex` with double-checked locking in `ensureMerkleCache`. Advisory locks in `CreateCallSession` are sorted to prevent deadlocks. |
| Error wrapping | Errors are wrapped with `fmt.Errorf("context: %w", err)` throughout, preserving the error chain for callers. |
| SKIP LOCKED | OTP claiming uses `FOR UPDATE SKIP LOCKED` which eliminates contention between concurrent claim operations -- correct pattern for queue-like operations. |
| E2EE key lifecycle | `WipeE2EEKey` NULLs both key and salt after call ends. `CleanupStaleRingingSessions` also wipes keys atomically via CTE. |

## Missing Index Analysis

| Query Pattern | Table | Current Indexes | Missing Index |
|--------------|-------|----------------|---------------|
| `WHERE "userId" = $1 AND "deviceId" = $2` (identity keys) | `e2e_identity_keys` | `@unique userId` only | **Need `@@unique([userId, deviceId])` or at minimum `@@index([userId, deviceId])`** (blocked by G02-C1) |
| `ORDER BY "createdAt" ASC` (Merkle rebuild) | `e2e_identity_keys` | None on `createdAt` | Could benefit from `@@index([createdAt])` for large trees |
| `WHERE "groupId" = $1 AND "recipientUserId" = $2 ORDER BY "createdAt" DESC` | `e2e_sender_keys` | `@@index([groupId, recipientUserId])` | Covered by existing index + sort on small result set |
| `WHERE status = 'RINGING' AND "createdAt" < ...` (cleanup) | `call_sessions` | `@@index([status])`, `@@index([createdAt])` | Composite `@@index([status, createdAt])` would be more efficient than intersecting two indexes |
| `WHERE "livekitRoomName" = $1` (multiple queries) | `call_sessions` | `@@index([livekitRoomName])` | Covered |
