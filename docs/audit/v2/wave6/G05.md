| 1 | H | handler.go:661-698 | **`HandleStopEgress` allows ANY participant to stop recording, not just the caller.** `HandleStartEgress` (line 620) correctly restricts recording start to `isCaller(session, userID)`. However, `HandleStopEgress` (line 681) only checks `isCallerOrParticipant`, meaning any callee can stop a recording they did not start. This is an authorization asymmetry: if recording requires caller permission to start (consent/legal), it should also require caller permission to stop. A malicious participant could stop an ongoing recording to destroy evidence or disrupt the session. | Any participant can stop a recording started by the caller; potential evidence tampering | Small |
| 2 | H | handler.go:700-747 | **`HandleCreateIngress` has no session status check.** `HandleStartEgress` (line 624) correctly rejects non-ACTIVE sessions (`session.Status != "ACTIVE"`). `HandleCreateIngress` does not check session status at all. A caller can create an ingress/RTMP stream for a session that is RINGING, ENDED, MISSED, or DECLINED. If the session is ENDED, the ingress would be created on a dead LiveKit room. If the session is RINGING, the ingress could start pumping media into a room before any participant has answered. | Ingress creation on non-ACTIVE sessions; media streaming into dead or unanswered rooms | Small |
| 3 | M | handler.go:782-893 | **Webhook handler DB operations swallow errors without logging.** Multiple DB calls inside the webhook handler (lines 816, 831, 834, 848, 849, 861, 867, 877, 885) discard errors silently. Examples: `h.db.UpdateSessionLivekitSid(ctx, ...)` at line 816 -- return value ignored. `h.db.UpdateSessionStatus(ctx, session.ID, "ACTIVE")` at line 867 -- error discarded. `h.db.MarkAllParticipantsLeft(ctx, session.ID)` at line 848 -- error discarded. `h.db.WipeE2EEKey(ctx, session.ID)` at line 849 -- error discarded. If any of these fail (DB timeout, connection pool exhaustion), the call session will be in an inconsistent state (e.g., E2EE key not wiped after call ends, session stuck in RINGING forever, duration not recorded). The project rule says "All errors explicit, never silently swallowed." | Silent DB failures in webhook handler leave call sessions in inconsistent state; E2EE key may persist after call ends | Medium |
| 4 | M | handler.go:343-345 | **`HandleDeleteRoom` ignores errors from `UpdateSessionStatus`, `MarkAllParticipantsLeft`, `WipeE2EEKey`.** Lines 343-345 make three critical DB calls (set status ENDED, mark participants left, wipe E2EE key) without checking any error returns. If `WipeE2EEKey` fails, the E2EE key persists in the database after the call ends, directly contradicting the forward-secrecy guarantee documented in the trust model (line 12: "key is wiped after call"). The handler returns `{"success": true}` regardless. Same issue in `HandleLeaveRoom` at lines 404-406, 414-416. | E2EE key material persists in DB on silent failure; false forward-secrecy guarantee | Medium |
| 5 | M | handler.go:907-941 | **`sendCallPush` goroutine uses `context.Background()` without bounds.** Line 910 calls `h.db.GetUserDisplayName(context.Background(), callerUserID)` with an unbounded context. If the database is slow or deadlocked, this goroutine blocks indefinitely. The goroutine is fire-and-forget (launched at line 233 via `go h.sendCallPush(...)`) with no timeout, no recover, and no way to cancel it. The `postInternalPush` call at line 936 has its own 5s timeout, but the DB call at line 910 has none. Under DB pressure, leaked goroutines accumulate. | Goroutine leak under DB pressure; no panic recovery in fire-and-forget goroutine | Small |
| 6 | M | handler.go:944-965 | **`sendMissedCallPush` silently swallows JSON marshal error.** Line 957: `if err != nil { return }` discards the error without logging. The function's sibling `sendCallPush` at line 932 logs the same error: `h.logger.Error("marshal push body failed", ...)`. This inconsistency means if JSON marshalling fails for missed-call push, there is no log trace. Additionally, `sendMissedCallPush` (called at line 844 via goroutine) has the same no-recover problem as `sendCallPush`. | Silent failure for missed-call notifications; no debugging trail | Trivial |
| 7 | M | handler.go:987 | **`http.DefaultClient` used for internal push requests.** `http.DefaultClient` has no timeout by default (the per-request context provides one, but `DefaultClient` also has no connection pool limits, no TLS configuration enforcement, no keep-alive tuning). If the NestJS server is slow to respond at the TCP level (SYN flood, DNS hang), the 5s context timeout may not cover the connection phase on all Go versions. A dedicated `http.Client` with `Transport` settings (max idle conns, TLS minimum version, dial timeout) would provide defense-in-depth. Also, `http.DefaultClient` is global mutable state, so any third-party library modifying it affects push delivery. | Unbounded connection phase; shared mutable global client | Small |
| 8 | M | handler.go:500-547 | **`HandleMuteParticipant` does not validate `TrackSid` or `Identity` for empty strings.** The handler decodes `identity`, `trackSid`, and `muted` from the body (lines 518-522), validates that `identity` is a participant (line 528), but never checks that `trackSid` is non-empty. An empty `trackSid` is passed directly to LiveKit's `MutePublishedTrack` API (line 540). Depending on LiveKit SDK behavior, this could mute all tracks, no tracks, or return a confusing error. The `identity` field is also not checked for empty string before the `isCallerOrParticipant` check (line 528), which would match against an empty-string participant if one existed in the session (unlikely but undefined). | Undefined LiveKit SDK behavior with empty trackSid; potential unintended mute-all | Trivial |
| 9 | M | handler.go:633 | **Egress recording filepath allows room name injection.** Line 633: `filepath := fmt.Sprintf("recordings/%s/{time}.mp4", req.RoomName)`. The `req.RoomName` comes from user input (the request body). Although `req.RoomName` is validated against the database (line 614, must match an existing session), room names are generated server-side with a crypto suffix (store.go line 153: `call_roomNameBase_hexSuffix`). If the store-generated room name ever contained path traversal characters (e.g., `../`), the S3 filepath could escape the `recordings/` prefix. Today the room name format is safe, but the handler does not independently sanitize the filepath. Defense-in-depth would validate no `..` or `/` sequences beyond the expected format. | Path traversal in S3 recording filepath if room name format changes | Trivial |
| 10 | L | handler.go:207, 294 | **`GetSessionE2EEMaterial` error silently discarded on CreateRoom and CreateToken.** Line 207: `e2eeMaterial, _ := h.db.GetSessionE2EEMaterial(r.Context(), actualRoomName)`.
