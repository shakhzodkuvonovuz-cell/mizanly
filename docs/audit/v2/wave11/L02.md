## Module Import Cycle Analysis

### Cycle 1: QueueModule <-> NotificationsModule (real, masked by @Global + @Optional)

```
QueueModule (@Global)
  imports: [NotificationsModule, GamificationModule, AiModule, SearchModule]
  providers: [QueueService, NotificationProcessor, ...]

NotificationsModule
  imports: [DevicesModule]
  providers: [NotificationsService]  // <-- injects @Optional() QueueService

NotificationsService.constructor:
  @Optional() private queueService: QueueService | null
```

**Chain:** `QueueModule` -> imports `NotificationsModule` -> `NotificationsService` depends on `QueueService` (from `QueueModule`)

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|

True circular dependency masked by `@Global()` + `@Optional()`. The `@Optional()` means QueueService may be null at runtime if module initialization order is wrong. | Fragile initialization order. ## Root Cause

The entire API has **zero event-driven architecture**. Every cross-cutting concern (notifications, gamification, moderation, search indexing, analytics) is wired via direct DI injection. This creates a star topology with NotificationsService at the center (21 direct consumers, 39 synchronous `.create()` calls). The forwardRef usages and @Optional annotations are symptoms of this fundamental coupling problem.

The fix is structural: introduce `@nestjs/event-emitter`, emit domain events from content services, and have cross-cutting services subscribe via `@OnEvent()`. This eliminates all circular dependencies, enables independent module extraction, and unblocks the Elixir WebSocket migration at 50K users (CLAUDE.md Scale Roadmap).
