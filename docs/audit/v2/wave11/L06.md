Auditor: Claude Opus 4.6 | Date: 2026-03-30 | Scope: apps/mobile — screen architecture, state management, service layer, component reuse

## Methodology
- Analyzed 10 largest screen files (1,000-3,169 lines) and 10 largest component files (355-773 lines)
- Inspected store (499 lines, single file), service layer (api.ts 1,536 lines + 30+ specialized service files), hooks (28 files), providers (1 file)
- Searched for: prop drilling, duplicate API fetching, navigation listener leaks, duplicated utilities, style pattern violations, `as any` in non-test code, raw `Image` imports, hardcoded `colors.dark.*` in JSX

## Architecture Summary

### What works well
- **Service layer is centralized**: `api.ts` (1,536 lines) is a well-structured ApiClient with typed error classes, token refresh, 429 retry, 401 retry, and timeout handling. 30+ specialized service files (`creatorApi.ts`, `giphyService.ts`, etc.) extend it cleanly.
- **State management pattern is sound**: React Query for server cache, Zustand for client state, Context for socket. No unnecessary state duplication between systems.
- **Socket listeners are properly cleaned up**: The conversation screen (most complex socket user) correctly removes all listeners on unmount.
- **Component library is rich**: 35+ shared UI components in `src/components/ui/` are well-used across screens.
- **Hooks directory is organized**: 28 purpose-specific hooks (`useContextualHaptic`, `useThemeColors`, `useLiveKitCall`, etc.) extract reusable behavior.

### What needs work
1. **God screens**: The top 5 screen files (conversation 3,169, video-editor 2,606, create-story 1,645, create-reel 1,613, settings 1,531) contain business logic, UI, styling, and sub-components all in one file. These need hook extraction and component decomposition.
2. **Theme violation at scale**: 643 `colors.dark.*` + 1,628 `colors.text.*` hardcoded references in 199 files mean light theme is fundamentally broken. The `useThemeColors()` infrastructure exists but adoption is incomplete.
3. **Store monolith**: The single 500-line Zustand store with 184 properties will become a maintenance burden. Domain slicing is the standard Zustand pattern for stores this size.
4. **Utility duplication**: `formatTime` (11 copies), draft persistence (4 copies), GIF picking (2 implementations) show a pattern of copy-paste rather than extraction.
