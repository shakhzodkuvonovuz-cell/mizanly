## Pattern Summary

### 1. Silent `.catch(() => {})` — 46 instances found
The dominant anti-pattern. 46 `.catch(() => {})` calls across services. While some are genuinely acceptable (Redis cache invalidation, fire-and-forget notifications), many swallow critical errors:
- **Payment mapping DB writes** (lines 2-3): money collected but not tracked
- **Admin audit logs** (lines 5-6): compliance gap
- **Group invite DB persistence** (line 4): data loss on Redis flush
- **View/loop counters** (lines 7-8): analytics corruption

**Recommendation**: Replace `catch(() => {})` with `catch(err => this.logger.debug('context', err?.message))` universally. This costs nothing at runtime but provides operational visibility. For payment-critical paths, await the write.

### 2. Caught + Logged but Not Re-thrown — 30+ instances
Many services catch errors from external services (Meilisearch, Expo Push, Claude AI), log them, and return fallback values. This is the correct pattern for non-critical side effects (notifications, analytics). However, the `push.service.ts:sendBatch` silently swallowing ALL push failures is concerning.

### 3. Missing try-catch on Network Calls — 2 instances
`stream.service.ts:getPlaybackUrls` and `stream.service.ts:deleteVideo` make `fetch()` calls without try-catch. Network errors propagate as unhandled errors.

### 4. Error Message Leaking — 1 critical instance
`auth.service.ts:104` leaks Clerk SDK error messages directly to the client in a `BadRequestException`. The global exception filter passes 4xx messages through unchanged.

### 5. Inconsistent Error Types — 8 instances
Some services throw `new Error(...)` instead of `HttpException` subclasses. When these escape catch blocks, the global filter converts them to generic 500s, losing context.
