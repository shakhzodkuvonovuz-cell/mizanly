## 1. Pagination: 3 Different Cursor Strategies

Sampled 30+ services. Found **3 incompatible cursor strategies**:

**Strategy A — Prisma ID-based cursor (correct):** `{ cursor: { id: cursor }, skip: 1 }`
Used by: admin, alt-profile, channels, commerce, community, community (boards), circles, chat-export, majlis-lists

**Strategy B — Date-string cursor (fragile):** `where.createdAt = { lt: new Date(cursor) }`
Used by: communities, audio-rooms, broadcast (subscribers), gamification (XP history), mosques, feed, halal, notifications

**Strategy C — Manual `id: { lt: cursor }` without Prisma cursor API:**
Used by: clips, collabs, channel-posts, audio-tracks, broadcast (messages/channels), live, messages, moderation, telegram-features, posts, threads, story-chains, personalized-feed

Additionally, some return `cursor: null` on last page, others return `cursor: undefined`. Some use `||`, others use `??`.

|---|-----|-----------|---------|--------|------------|

## 4. Naming: `create` vs `add` vs `remove` vs `delete`

Sampled all 80 controllers. Found mixed conventions:

**Creation verbs:** `create` (dominant, ~30 uses), `add` (checklists.addItem, alt-profile.addAccess, halal.addReview, stickers.addPack, circles.addMembers), `register` (devices.registerDevice), `start` (live.start, scholar-qa.startSession), `invite` (collabs.invite, discord-features.inviteSpeaker)

The pattern is: `create/delete` for entity CRUD, `add/remove` for membership operations. This is actually **a reasonable convention** but not documented anywhere.

## 5. Response Wrapping: Double-wrapping in communities module

The `TransformInterceptor` globally wraps ALL responses with `{ success: true, data: T, timestamp }`. It detects existing `{ data, meta }` envelopes and merges them.

**Problem:** `communities.service.ts` manually constructs `{ data, success: true, timestamp }` in every method (lines 133, 179, 208, 247, 264, 304, 334, 382). The TransformInterceptor then wraps this AGAIN, producing:
```json
{ "success": true, "data": { "data": { ... }, "success": true, "timestamp": "..." }, "timestamp": "..." }
```

Wait — actually the interceptor checks for `data in data && meta in data`. Since communities returns `{ data, success, timestamp }` without `meta`, the interceptor wraps it in another `data` envelope. This creates **double nesting**.

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 20 | H | communities.service.ts:133 | Returns `{ data, success: true, timestamp }` — TransformInterceptor does NOT detect this as pre-wrapped (no `meta` key), so it wraps again: `{ success: true, data: { data: circle, success: true, timestamp: ... }, timestamp: ... }` | Client gets double-nested `data.data` for all community operations | M — remove manual wrapping from all 8 methods |
| 21 | H | communities.service.ts:179 | Same double-wrapping for list members | Same | S |
| 22 | H | communities.service.ts:208 | Same for join | Same | S |
| 23 | H | communities.service.ts:247 | Same for update | Same | S |
| 24 | H | communities.service.ts:264 | Same for leave | Same | S |
| 25 | H | communities.service.ts:304 | Same for ban member | Same | S |
| 26 | H | communities.service.ts:334 | Same for promote | Same | S |
| 27 | L | discord-features.service.ts:177,259,348,363 | Returns `{ success: true }` without `data` or `meta` — TransformInterceptor wraps to `{ success: true, data: { success: true }, timestamp }` — redundant nesting but less harmful | Minor inconsistency | S |
| 28 | L | Multiple (gamification, audio-rooms, events, downloads, etc.) | ~15 services return `{ success: true }` for mutation endpoints — gets wrapped to `{ success: true, data: { success: true }, timestamp }` | Double `success` field | S — just return void/null from mutations |

## 6. Auth Guards: 3 Strategies, Inconsistent Placement

**Strategy A — Class-level `@UseGuards(ClerkAuthGuard)`:** Applied to entire controller.
Used by: admin, blocks, bookmarks, checklists, chat-export, collabs, alt-profile (both controllers)

**Strategy B — Method-level `@UseGuards(ClerkAuthGuard)` or `@UseGuards(OptionalClerkAuthGuard)`:** Per-endpoint.
Used by: ai, audio-tracks, audio-rooms, broadcast, channels, clips, community, communities, events, discord-features

**Strategy C — No guard at all:**
Used by: webhooks (intentional — receives external webhooks), health/legal

The inconsistency is in Strategy B: some controllers mix `ClerkAuthGuard` and `OptionalClerkAuthGuard` on different methods within the same controller. This is **correct behavior** (some endpoints need auth, others support anonymous). But the following controllers have NO guard on mutation endpoints:

| # | Sev | File:Line | Finding | Impact | Fix Effort |
|---|-----|-----------|---------|--------|------------|
| 29 | L | ai.controller.ts:99 | `routeSpace(@Body() dto)` — no `@UseGuards` on this POST endpoint | Unauthenticated users can hit AI route-space endpoint (AI costs money) | S — add ClerkAuthGuard |
| 30 | L | ai.controller.ts:113 | `@UseGuards(OptionalClerkAuthGuard)` on `getQuranVerse` — GET is fine as optional, but inconsistent with other AI endpoints that require auth | Minor | S |

## Summary

### Severity Counts

**C: 0 | H: 8 | M: 18 | L: 20 | I: 4 | Total: 50**

### Worst Offenders by Category

1. **Pagination (M x8):** 8 services use fragile Date-string cursors that will skip records on duplicate timestamps. The notification service is highest risk due to batch fan-out.
2. **Response wrapping (H x7):** `communities.service.ts` double-wraps all 7+ responses because it manually adds `{ success, timestamp }` which the global TransformInterceptor wraps again.
3. **Validation gaps (H x3):** 5 controller endpoints accept raw inline types with zero class-validator validation — any JSON passes through.
4. **Error format (H x2, M x3):** 7 service methods throw raw `new Error()` which returns unstructured 500 instead of proper NestJS HTTP exceptions.
5. **Mobile state (M x4):** broadcast-channels and broadcast/[id] manually reimplement React Query with 10-11 useState calls each — no cache, no error recovery, race conditions.
6. **Styling (M x1, L x3):** 112 files hardcode `colors.dark.*` in static StyleSheet, blocking any future light theme. 3 screens have 38-155 inline styles.

### Risk Assessment

The pagination and double-wrapping issues are the most impactful. The Date-string cursor in notifications will cause users to miss notifications when batch-created. The communities double-wrapping means every community API call returns `data.data` nesting that the mobile app must work around (or is silently broken).

The validation gaps on messages group-topic, moderation appeal, and mosque creation are exploitable — any string length, any coordinate value.
