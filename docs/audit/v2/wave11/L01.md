### Key Observations

1. **~5,400 lines of completely dead code** across API and mobile. This is ~2% of the 276K LOC codebase. Not catastrophic but creates maintenance burden and confuses new readers.

2. **12 mobile API client files are fully dead** (668 lines). Backend modules exist, controllers exist, API client wrappers exist, but no screen ever calls them. Features like video replies, story chains, discord features, telegram features, mosques, scholar Q&A, checklists, OG previews, thumbnails, privacy settings, retention tracking, and stream uploads are all wired backend-to-client but the last mile (screen -> API call) is missing.

3. **Feature flags and A/B testing are built but decorative.** The entire feature flag infrastructure (Redis+DB dual storage, percentage rollout, consistent user hashing) is never used for production gating. The A/B testing service has 0 callers. These represent ~500 lines of sophisticated but unused infrastructure.

4. **26 of 41 Zustand store convenience hooks are dead.** Components access the store directly via `useStore(s => s.field)` rather than the exported convenience hooks like `useActiveCallId()`.

5. **RTL support (193 lines) is completely dead.** Despite supporting Arabic, Urdu, and other RTL languages in i18n, the RTL utility functions are never imported.

6. **3 legal compliance TODOs are the highest-severity items.** CSAM reporting, terrorism content reporting, and Australian eSafety reporting are marked as TODO but not implemented. These are legal requirements for a UGC app, especially one targeting Muslim communities (terrorism content moderation is heavily scrutinized).
