## Findings

### CRITICAL

| # | Finding | File(s) | Detail |
|---|---------|---------|--------|
| C1 | `resizeForUpload()` only called by 2 of 12+ upload paths | `apps/mobile/app/(screens)/create-story.tsx`, `create-thread.tsx`, `edit-profile.tsx`, `disposable-camera.tsx`, `community-posts.tsx`, `create-broadcast.tsx`, `conversation-info.tsx`, `create-group.tsx`, `profile-customization.tsx` | `resizeForUpload()` exists in `utils/imageResize.ts` and correctly resizes images to max 2048px + 82% JPEG quality before upload. But only `create-post.tsx` and `create-carousel.tsx` call it. **All other upload paths send full-resolution raw blobs directly to R2** -- stories, threads, edit-profile avatars/covers, disposable camera, community posts, broadcasts, group avatars, profile customization. A 12MP iPhone photo is ~4-8MB uncompressed. Users on 4G will wait 3-8s per image. This is the single biggest performance gap in the entire app. |
| C2 | Media processing queue is wired but never enqueued | `apps/api/src/common/queue/queue.service.ts`, `apps/api/src/common/queue/processors/media.processor.ts` | `MediaProcessor` has working code for EXIF stripping, image resize variants (thumb/medium/large), and BlurHash generation. `QueueService` injects `mediaQueue`. But **no code anywhere calls `mediaQueue.add()`**. The entire media processing pipeline is dead code. EXIF GPS coordinates leak in every uploaded photo. BlurHash is never computed from real content. Resize variants are never generated server-side. |
| C3 | No orphaned upload cleanup job or cron | `apps/api/src/modules/upload/` | When a user picks an image, gets a presigned URL, uploads to R2, then abandons the post (back button, crash, app kill), the R2 object persists forever. There is no reconciliation job that compares R2 objects against database references. `deleteFile()` exists but is only called on account deletion (`privacy.service.ts`). Over time, R2 storage accumulates unlimited orphaned files with no way to identify or purge them. |

### HIGH

| # | Finding | File(s) | Detail |
|---|---------|---------|--------|
| H1 | `imagePresets` / `optimizedImageUrl()` used only by Avatar component | `apps/mobile/src/utils/image.ts`, `apps/mobile/src/components/ui/Avatar.tsx` | The Cloudflare CDN Image Resizing utility (`optimizedImageUrl`) with presets for thumbnails, feed images, covers, and video thumbs is defined and correct. But **only Avatar uses it**. `PostMedia`, `ProgressiveImage`, `ImageCarousel`, `ImageGallery`, `ThreadCard`, `LinkPreview`, `StickerPackBrowser`, and all screen-level `<Image>` tags render raw R2 URLs. Users download full-resolution images (1200px+) for 60px thumbnails in lists. |
| H2 | BlurHash values are hardcoded defaults, never content-specific | `apps/mobile/src/utils/blurhash.ts`, `apps/api/src/common/queue/processors/media.processor.ts` | `ProgressiveImage` correctly uses `expo-image`'s blurhash placeholder. The `MediaProcessor` has working BlurHash generation code using `sharp` + `blurhash` encode. But since C2 means no jobs are ever enqueued, the `blurhash` column on Post/Reel/Story/Video is always NULL. The client falls back to `BLURHASH_POST` (a generic dark gray hash). Every post shows the same placeholder before load -- no content-specific blur preview like Instagram/WhatsApp. |
| H3 | Cloudflare Image Resizing gated behind env flag, defaults to OFF | `apps/api/src/common/utils/image.ts` line 90 | `getResponsiveImageUrls()` checks `CF_IMAGE_RESIZING_ENABLED === 'true'`. When not set (which is the default), all variants return the original URL. The presign response includes `variants: { thumbnail, small, medium, large }` but they're all identical to the original. The mobile client's `imagePresets` generates `/cdn-cgi/image/` URLs client-side (bypassing this flag), but only Avatar uses them. The API-side variant system is entirely inactive. |
| H4 | `ImageGallery` loads full-res images for all gallery items simultaneously | `apps/mobile/src/components/ui/ImageGallery.tsx` line 287-306 | `FlatList` with `renderItem` uses `<Image source={{ uri: item }}>` with raw URLs for all gallery images. No `windowSize` optimization, no `maxToRenderPerBatch`, no `initialNumToRender` limits. With a 10-image carousel, all 10 full-resolution images are rendered and decoded simultaneously when the gallery opens. Should use `windowSize={3}` and pass through `optimizedImageUrl` with `feedFull` preset. |
| H5 | Video `shouldPlay` / `autoPlay` not gated by visibility | `apps/mobile/src/components/ui/VideoPlayer.tsx` line 302 | `VideoPlayer` accepts `autoPlay` prop and passes it to `<Video shouldPlay={autoPlay}>`. The `autoPlaySetting` store value (`wifi`/`always`/`never`) exists but is never read by `VideoPlayer` itself. No viewport-based play/pause (Intersection Observer equivalent). If multiple `VideoPlayer` instances exist in a feed, they all start loading/buffering regardless of visibility. `autoPlaySetting` is a settings UI placeholder with no runtime effect. |

### MEDIUM

| # | Finding | File(s) | Detail |
|---|---------|---------|--------|
| M1 | ProgressiveImage does not use `optimizedImageUrl` for CDN resizing | `apps/mobile/src/components/ui/ProgressiveImage.tsx` | The component accepts a raw `uri` and passes it directly to expo-image `<Image source={{ uri }}>`. No width-aware CDN optimization. A 300px-wide thumbnail still downloads the full 4000px original. Should accept the display width and call `optimizedImageUrl(uri, { width })` internally, or accept pre-optimized URIs from the parent. |
| M2 | No `cachePolicy` explicitly set on any expo-image instance | `apps/mobile/src/config/image.ts`, all Image/ProgressiveImage usage | `config/image.ts` has a comment noting "default cachePolicy='memory-disk'" but never actually calls `Image.setCachePolicy()`. The config file exports nothing (`export {}`). expo-image defaults are fine for most cases, but there is no cache size limit, no cache eviction strategy, and no differentiation between avatars (cache forever) vs ephemeral content (cache briefly). Heavy scrollers will OOM as the disk cache grows unbounded. |
| M3 | R2 presigned PUT has no CDN edge cache population | `apps/api/src/modules/upload/upload.service.ts` line 122-126 | The `PutObjectCommand` sets `CacheControl: 'public, max-age=31536000, immutable'` which is correct for CDN caching. However, R2 alone does NOT provide edge caching -- it requires either a Cloudflare Worker binding or R2 custom domain with Cache Reserve. Without this, every image request goes to the R2 origin (single region). The `publicUrl` (`https://media.mizanly.app`) needs to be served through Cloudflare CDN, not direct R2 access. No Worker or custom domain config exists in the repo. |
| M4 | No image dimension hints in raw `<Image>` instances | `apps/mobile/src/components/ui/ImageGallery.tsx` line 296-300, `apps/mobile/src/components/risalah/StickerPackBrowser.tsx` | Several `<Image>` instances use `style={styles.image}` with `width: '100%', height: '100%'` without explicit dimensions. This causes layout shifts on load -- the container collapses to 0 height until the image loads, then jumps to full size. `PostMedia` correctly uses `aspectRatio` prop from post data, but `ImageGallery`, `StickerPackBrowser`, and `LinkPreview` do not provide dimension hints. |
| M5 | Story/reel creation uploads raw video without compression | `apps/mobile/app/(screens)/create-story.tsx` line 508-513, `apps/mobile/app/(screens)/create-reel.tsx` line 415-429 | `create-reel.tsx` and `create-story.tsx` upload raw `video/mp4` blobs directly to R2 via presigned URL with no client-side compression. A 30-second 4K video can be 100-200MB. `create-reel.tsx` has a `100MB` max but no re-encoding. The video editor's FFmpeg integration (from session 3) exists for editing but is not wired into the upload path for output quality/size optimization. |

### LOW

| # | Finding | File(s) | Detail |
|---|---------|---------|--------|
| L1 | `ImageCarousel` prefetches only next 2 images, not thumbnailed | `apps/mobile/src/components/ui/ImageCarousel.tsx` line 112-117 | `Image.prefetch(images[i])` prefetches full-resolution images. Should prefetch CDN-resized versions matching the display size. Minor impact since prefetch is lazy, but wastes bandwidth on hi-res images the user may never swipe to. |
| L2 | Avatar uses CDN optimization but no fallback on CDN failure | `apps/mobile/src/components/ui/Avatar.tsx` line 58 | `optimizedUri` generates a Cloudflare `/cdn-cgi/image/` URL. If Cloudflare Image Resizing is not enabled (requires paid plan), these URLs return 404. No fallback to the original URL on error. expo-image will show the blurhash placeholder indefinitely. Should add `onError` handler that retries with the raw `uri`. |
| L3 | `ViewOnceMedia` loads full-resolution for view-once photos | `apps/mobile/src/components/ViewOnceMedia.tsx` | View-once photos are loaded at full resolution despite being displayed for only 10 seconds and then deleted. The image is never saved or zoomed. Should use a medium-quality CDN variant (600px) since the user can't save or zoom the image anyway. Saves bandwidth for ephemeral content. |

## Impact Assessment

| Area | Current State | Impact |
|------|--------------|--------|
| Upload bandwidth | 8 of 10 image upload paths send full-res (4-8MB per photo) | 3-8s upload on 4G per image, user abandonment |
| Download bandwidth | Feed images served at full-res (except avatars) | ~5x unnecessary data transfer vs proper thumbnails |
| EXIF privacy | GPS coordinates in every uploaded photo | Privacy violation (GDPR Art 25 data protection by design) |
| BlurHash | Generic placeholder for all content | No perceived-speed improvement, inferior to Instagram |
| Orphaned files | No cleanup mechanism | Unbounded R2 storage cost growth |
| Video loading | No viewport-based lazy loading | Multiple videos buffer simultaneously in feed |
| CDN caching | R2 without edge caching | Every request hits R2 origin, ~200ms latency worldwide |

---

## Recommended Fix Priority

1. **C1 + C2** (highest ROI): Wire `resizeForUpload()` into all upload paths AND enqueue media processing jobs (EXIF strip + BlurHash) on every image upload. These two fixes together transform perceived performance.
2. **H1 + M1**: Make `ProgressiveImage` width-aware -- pass display width to `optimizedImageUrl()` internally. This single change makes every feed image, thread image, and gallery thumbnail CDN-optimized.
3. **C3**: Add a daily cron job that lists R2 objects and deletes any not referenced in the database.
4. **M3**: Configure R2 custom domain through Cloudflare dashboard (no code change, just DNS + dashboard config).
5. **H5 + M5**: Wire `autoPlaySetting` into VideoPlayer and add client-side video compression before upload.
