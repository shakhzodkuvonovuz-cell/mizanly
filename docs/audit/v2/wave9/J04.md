# J04 -- Memory Leak Audit

**Scope:** `apps/api/src/` and `apps/mobile/src/` -- setInterval/setTimeout without cleanup, event listeners without removal, subscriptions without unsubscribe, growing Maps/Sets never evicted, stale closures holding references, Redis connections not returned to pool, Prisma client not properly managed, large objects cached without TTL/LRU.

## Findings

### ML-01 -- Signal Service AppState listener never removed (CRITICAL)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/services/signal/index.ts` line 196 |
| **Severity** | Critical |
| **Category** | Event listener without removal |

**Problem:** The `initialize()` function registers an `AppState.addEventListener('change', ...)` listener for cleaning up decrypted media temp files on backgrounding. This listener is **never removed**. The `initialize()` function is called once on app startup, but:

1. The returned subscription is not captured or stored.
2. If `initialize()` were ever called multiple times (e.g., after a re-auth or hot reload), duplicate listeners accumulate.
3. Since the listener holds a closure over `cleanupDecryptedMediaFiles`, it pins that module in memory permanently.

The listener itself calls `require('react-native')` at the module scope and uses the old `addEventListener` API without storing the subscription return value.

```typescript
// line 196 -- no subscription capture, no cleanup
AppState.addEventListener('change', (state: string) => {
  if (state === 'background' || state === 'inactive') cleanupDecryptedMediaFiles();
});
```

**Fix:** Store the subscription and expose a `teardown()` or `deinitialize()` function. Alternatively, guard against double-registration with a module-level flag for the listener itself (separate from the `initialized` flag).

### ML-02 -- ChatGateway typingTimers Map grows unbounded during normal operation (MAJOR)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/gateways/chat.gateway.ts` line 73 |
| **Severity** | Major |
| **Category** | Growing Map never evicted |

**Problem:** The `typingTimers` Map uses composite keys like `userId:conversationId`. Timer entries are added when a user starts typing but there is no evidence that entries are reliably removed when the typing stops normally (only cleared on module destroy). If the typing timeout fires, the timer completes but the Map entry (with its key string) may persist as a stale `undefined`-valued entry depending on how the handler is wired. Over the lifetime of a long-running server with thousands of conversations, this Map grows monotonically.

The `onModuleDestroy` cleanup only clears it at shutdown, not during normal operation. A server running for days or weeks with many active conversations will accumulate entries.

**Fix:** In the typing timeout callback and in the `stopTyping` handler, explicitly `delete` the key from the Map after clearing the timer. Ensure every code path that clears a timer also removes the Map entry.

### ML-03 -- AsyncJobService secondary cleanup timer leaks (MAJOR)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/common/services/async-jobs.service.ts` line 85 |
| **Severity** | Major |
| **Category** | setTimeout without proper cleanup |

**Problem:** Although marked `@deprecated` and unused, the `executeWithRetry` method creates a cleanup timer for each retry:

```typescript
// line 85 -- cleanup timer
setTimeout(() => this.pendingTimers.delete(timer), delay + 100);
```

This secondary `setTimeout` is **not** tracked in `pendingTimers`. If `onModuleDestroy` is called while the retry delay is active, the primary timer is cleared but the cleanup timer (delay + 100ms) continues to fire after the Set is cleared, referencing a now-empty Set. This is a minor leak, but the real problem is that the cleanup timer itself is never cancelable.

**Fix:** Since this service is deprecated and unused (the code explicitly says so), either remove it entirely or track the secondary timer in `pendingTimers` too.

### ML-04 -- Toast dismiss animation setTimeout not cleaned up on unmount (MAJOR)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/components/ui/Toast.tsx` line 111-112 |
| **Severity** | Major |
| **Category** | setTimeout without cleanup |

**Problem:** In the `dismiss` callback, a `setTimeout` fires after the exit animation to remove the toast from the store:

```typescript
setTimeout(() => {
  dismissToast(toast.id);
}, animation.timing.normal);
```

This timer is **not** tracked in a ref and not cleaned up on component unmount. If `ToastItem` unmounts before the animation completes (e.g., rapid toast dismissal, navigation), the `setTimeout` fires and calls `dismissToast` on a potentially stale toast ID. This also calls `setState` on an unmounted component (React warning) and the closure holds references to the callback dependencies.

**Fix:** Store this timer ID and clear it in the useEffect cleanup, or use Reanimated's `withTiming` callback mechanism (which is already animation-thread-safe) instead of a separate `setTimeout`.

### ML-05 -- PostCard setTimeout for animation bounce not cleaned up (MAJOR)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/components/saf/PostCard.tsx` line 95 |
| **Severity** | Major |
| **Category** | setTimeout without cleanup in mutation callback |

**Problem:** Inside `reactMutation.onMutate`:

```typescript
setTimeout(() => { likeCountScale.value = withSpring(1, ...); }, 150);
```

This `setTimeout` is not cleaned up if `PostCard` unmounts within 150ms (e.g., rapid scrolling in FlashList which aggressively recycles cells, or navigating away). It will write to a shared value on a potentially unmounted component. Since `PostCard` is used inside a `FlashList` feed that recycles heavily, this fires very frequently.

**Fix:** Store the timer ref and clear it on unmount, or use Reanimated's `withDelay` to chain the animation on the UI thread without a JS timer.

### ML-07 -- useAutoUpdateTimestamp interval does not readjust to changing ages (MINOR)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/hooks/useAutoUpdateTimestamp.ts` lines 30-37 |
| **Severity** | Minor |
| **Category** | setInterval with stale configuration |

**Problem:** The interval is calculated once at effect creation time based on the current age of the timestamp. But the interval is never adjusted as time passes. A timestamp that starts as "30 seconds ago" (10s interval) will continue ticking at 10-second intervals even when it becomes "30 minutes ago" (should be 60s interval) and beyond. It only re-evaluates when `date` changes in the dependency array, which for a fixed date, never changes.

This means timestamps in long-lived lists (e.g., a chat thread kept open for hours) continue firing 10-second intervals forever, causing unnecessary re-renders across potentially hundreds of components.

**Fix:** Use a recursive `setTimeout` that recalculates the interval on each tick, or add a mechanism to clear and restart the interval when the age crosses a threshold.

### ML-08 -- VideoPlayer seekIndicatorTimeoutRef not cleared on unmount (MINOR)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/components/ui/VideoPlayer.tsx` lines 107-109 |
| **Severity** | Minor |
| **Category** | setTimeout without cleanup on unmount |

**Problem:** `seekIndicatorTimeoutRef.current` is set via `setTimeout` in the `showSeekIndicator` callback but never cleared in the component's unmount useEffect. The main useEffect only clears `controlsTimeoutRef`, not `seekIndicatorTimeoutRef`. If the user double-taps to seek then navigates away within 800ms, the callback fires on an unmounted component.

**Fix:** Add `if (seekIndicatorTimeoutRef.current) clearTimeout(seekIndicatorTimeoutRef.current);` to the cleanup function in the main `useEffect`.

### ML-09 -- useVideoPreloader loadStates Map grows without bound across feed sessions (MINOR)

| Field | Value |
|-------|-------|
| **File** | `apps/mobile/src/hooks/useVideoPreloader.ts` line 25 |
| **Severity** | Minor |
| **Category** | Growing Map/state without eviction |

**Problem:** The `loadStates` state map accumulates entries as the user scrolls through the reel feed. The `preloadedUrls` ref set has a cap at 20, but the `loadStates` state map has no such cap. Every URL that has ever been preloaded during the component's lifetime gets a state entry that is never removed (only `clearAll` wipes it, but that requires explicit invocation). In a long reel browsing session (hundreds of reels), this Map grows continuously.

**Fix:** Evict old entries from `loadStates` in `onViewableChange` (the same place `preloadedUrls` is trimmed), or use the same 20-item bound.

### ML-10 -- REDIS_SHUTDOWN provider onModuleDestroy may not be called (MINOR)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/config/redis.module.ts` lines 96-111 |
| **Severity** | Minor |
| **Category** | Redis connection not properly managed |

**Problem:** The `REDIS_SHUTDOWN` provider is registered with `provide: 'REDIS_SHUTDOWN'` but is never exported from the module and never injected by any other module. NestJS only invokes `onModuleDestroy` lifecycle hooks on providers that are actually resolved and instantiated. Since `REDIS_SHUTDOWN` is never injected, NestJS may optimize it away and never call the shutdown hook, leaving the Redis connection open on app termination.

Note: There are `SIGTERM`/`SIGINT` handlers in `main.ts` for the socket adapter, but none for the main Redis client. The `app.enableShutdownHooks()` call should trigger `onModuleDestroy`, but only on providers that were instantiated.

**Fix:** Either inject `REDIS_SHUTDOWN` in the `RedisModule` constructor or a health controller to force instantiation, or implement `OnModuleDestroy` on `RedisModule` itself and close Redis there.

### ML-11 -- ChatGateway Redis subscriber duplicate connection lifecycle (INFO)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/gateways/chat.gateway.ts` line 91 |
| **Severity** | Info |
| **Category** | Redis connection management |

**Problem:** The `onModuleInit` creates a duplicate Redis connection via `this.redis.duplicate()` for pub/sub subscription. This is correct (pub/sub requires a dedicated connection). The cleanup in `onModuleDestroy` properly unsubscribes and disconnects. However, if `onModuleInit` fails after `duplicate()` but before the subscriber is assigned to `this.redisSubscriber`, the duplicate connection leaks. The current code wraps in try/catch and only catches the error, but `this.redisSubscriber` might be assigned before the subscribe fails, so this is mostly fine. Noting for completeness.

**Impact:** Negligible in practice -- only during startup failure scenarios.

### ML-12 -- FeatureFlagsService localCache replaced not cleared (INFO)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/common/services/feature-flags.service.ts` line 114 |
| **Severity** | Info |
| **Category** | Cache without explicit LRU/bound |

**Problem:** The `localCache` Map is replaced wholesale on every cache refresh (`this.localCache = new Map(Object.entries(all))`). While this releases the old Map to GC (good), the Map itself has no size bound. If the Redis `feature_flags` hash grows to thousands of entries, the local cache holds all of them in memory. There is a `take: 200` on the DB fallback path but no limit on the Redis `hgetall` path.

**Impact:** Low -- feature flags are typically counted in tens, not thousands. But there is no defensive limit.

### ML-13 -- AnalyticsService flush retry can exceed buffer cap (INFO)

| Field | Value |
|-------|-------|
| **File** | `apps/api/src/common/services/analytics.service.ts` lines 107-111 |
| **Severity** | Info |
| **Category** | Buffer re-growth after failure |

**Problem:** When `flush()` fails, events are put back into the buffer:

```typescript
this.buffer.unshift(...events);
```

But the `track()` method's cap check (`this.buffer.length >= this.MAX_BUFFER_SIZE`) only trims from the front when new events arrive. If Redis is down for an extended period, the flush-fail-unshift cycle can temporarily exceed `MAX_BUFFER_SIZE` between `track()` calls, because `unshift` bypasses the cap check. The next `track()` call will trim, so this is bounded, but there is a transient oversize window.

**Impact:** Very low. The buffer is capped at 10K events (~2-5MB). The transient overshoot is bounded by the 100-event auto-flush trigger.
