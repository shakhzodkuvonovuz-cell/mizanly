## Summary

The codebase shows **mixed discipline** on N+1 avoidance. The shared `enrich.ts` utility and `posts.service.ts` hashtag batch increment demonstrate awareness of the problem. However, several high-traffic paths still contain sequential loops with individual Prisma calls.

### Highest-impact findings:

1. **alt-profile.service.ts addAccess()** (H) — 100 sequential upserts in a loop. The simplest and most egregious N+1. Fix: `createMany`.

2. **reels.service.ts hashtag upsert** (H) — N individual upserts instead of the batch pattern already used in posts.service. Inconsistency within the codebase.

3. **scheduling.service.ts hashtag updates** (H, 6 instances) — Six separate `for` loops doing individual hashtag `UPDATE` queries. These fire during cron jobs that process multiple content items, multiplying the impact.

4. **messages.service.ts DM permission checks** (M) — 3 sequential independent queries on every DM send. Easy `Promise.all` fix on the hottest path in the app.

### Patterns observed:

- **Posts service**: Uses batch `$executeRawUnsafe` with `ANY()` for hashtag creation, but N individual `$executeRaw` for hashtag deletion.
- **Reels service**: Uses N individual `prisma.hashtag.upsert()` instead of the batch pattern from posts.
- **Scheduling service**: 6 places with `for (const name of ...) { prisma.hashtag.update(...) }` — none batched.
- **Notification loops**: Multiple services iterate users and call `notifications.create()` sequentially. Each `notifications.create()` internally makes 4+ DB queries (settings, block, mute, dedup), so N mentions = 4N+ queries.
- **Feed tiered queries**: Intentional adaptive windowing, but could be simplified to a single wide-window query.
