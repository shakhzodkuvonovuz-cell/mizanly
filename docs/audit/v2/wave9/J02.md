# J02 — Missing Database Indexes Audit

**Scope:** `apps/api/prisma/schema.prisma` (5,037 lines, ~200 models) cross-referenced against all `*.service.ts` WHERE/ORDER BY patterns.

**Method:** Read every model's `@@index` declarations, then grep every service file for `findMany`, `findFirst`, `count`, `aggregate`, `orderBy`, and `where` patterns. Each finding below identifies a query pattern used in production code that lacks a supporting index.

---

## Critical Findings (High-Traffic Paths)

| # | Severity | Model | Missing Index | Query Pattern | Service File(s) | Impact |
|---|----------|-------|---------------|---------------|-----------------|--------|
| 1 | **CRITICAL** | `Post` | `@@index([isRemoved, visibility, createdAt(sort: Desc)])` | Feed queries: `WHERE isRemoved=false AND visibility='PUBLIC' AND createdAt >= cutoff ORDER BY createdAt DESC` | `posts.service.ts` (getFeed, getTrendingFallback, getBlendedFeed, getChronologicalFeed), `feed.service.ts` (getCommunityTrending, getTrendingFeed), `personalized-feed.service.ts` (getTrendingFeed saf) | Every feed load for every user. Existing index `[isRemoved, visibility, likesCount DESC]` only covers trending-by-likes. The chronological feed path (`ORDER BY createdAt DESC`) falls back to `[isRemoved, createdAt DESC]` which lacks the `visibility` filter, forcing a heap filter on every row. |
| 2 | **CRITICAL** | `Post` | `@@index([visibility, isRemoved, likesCount(sort: Desc)])` | Search/related: `WHERE visibility='PUBLIC' AND isRemoved=false ORDER BY likesCount DESC` | `search.service.ts` (aggregate search, paginated post search), `posts.service.ts` (getRelatedPosts), `personalized-feed.service.ts` (getIslamicEditorialPicks) | Every search query and related-posts call. Existing index `[isRemoved, visibility, likesCount DESC]` has columns in different order -- this matters for Postgres B-tree leading column selectivity. The current index puts `isRemoved` first (boolean, low cardinality) instead of `visibility` (enum, 3 values). While usable, a `[visibility, isRemoved, likesCount DESC]` order would be optimal for queries that always filter `visibility='PUBLIC'`. **However**, since the existing index `[isRemoved, visibility, likesCount DESC]` does cover this pattern (just with a less selective leading column), this is a **MEDIUM** rather than CRITICAL. Downgraded below. |
| 3 | **CRITICAL** | `Reel` | `@@index([status, isRemoved, likesCount(sort: Desc)])` | Search/trending: `WHERE status='READY' AND isRemoved=false ORDER BY likesCount DESC` | `search.service.ts` (reel search, aggregate), `personalized-feed.service.ts` (getIslamicEditorialPicks bakra) | Every reel search. Existing index `[status, isRemoved, isTrial, createdAt DESC]` covers the feed but NOT `likesCount` ordering. Result: filesort on potentially millions of reels. |
| 4 | **CRITICAL** | `Reel` | `@@index([status, isRemoved, viewsCount(sort: Desc)])` | Trending reels: `WHERE status='READY' AND isRemoved=false ORDER BY viewsCount DESC` | `personalized-feed.service.ts` (getTrendingFeed bakra), `channels.service.ts` | Existing `[viewsCount DESC]` index has no filter columns -- Postgres must scan entire index and filter. |
| 5 | **CRITICAL** | `Thread` | `@@index([isRemoved, visibility, isChainHead, likesCount(sort: Desc)])` | Search/trending threads: `WHERE isRemoved=false AND visibility='PUBLIC' AND isChainHead=true ORDER BY likesCount DESC` | `search.service.ts` (thread search, aggregate), `personalized-feed.service.ts` (getIslamicEditorialPicks majlis, getTrendingFeed majlis) | No existing index covers this 3-column filter + likesCount sort. Falls to sequential scan + filesort. |
| 6 | **CRITICAL** | `Video` | `@@index([status, isRemoved, viewsCount(sort: Desc)])` | Video search/trending: `WHERE status='PUBLISHED' AND isRemoved=false ORDER BY viewsCount DESC` | `search.service.ts` (video search), `videos.service.ts` (getRelated, channels.service.ts) | Existing `[status, isRemoved, createdAt DESC]` covers createdAt sorts only. viewsCount sort has no composite index. |
| 7 | **CRITICAL** | `Message` | `@@index([conversationId, isPinned, isDeleted])` | Pinned messages: `WHERE conversationId=X AND isPinned=true AND isDeleted=false` | `messages.service.ts` (pinMessage count, getPinnedMessages) | Existing `[conversationId, createdAt DESC]` doesn't cover isPinned filter. Small result set per conversation but called on every pin action. |
| 8 | **CRITICAL** | `Message` | `@@index([conversationId, senderId, createdAt(sort: Desc)])` | Slow mode check: `WHERE conversationId=X AND senderId=Y ORDER BY createdAt DESC LIMIT 1` | `messages.service.ts` (sendMessage slow mode) | Called on every message send in slow-mode conversations. | 9 | **CRITICAL** | `Message` | `@@index([isScheduled, scheduledAt])` | Scheduled message cron: `WHERE isScheduled=true AND scheduledAt <= now()` | `scheduling.service.ts` (publishOverdueContent), `messages.service.ts` | Cron runs every minute. Existing `[scheduledAt]` index lacks `isScheduled` filter. Without it, Postgres scans all messages with non-null scheduledAt, including already-published ones. |

| CRITICAL | 9 | Missing indexes on hot-path feed, search, message queries. Every feed load, search, or message send. |
| HIGH | 11 | Moderate-traffic paths: scheduling crons, notification sorting, profile views. |

| **Total** | **40** | |

## Top 5 Impact Rankings

1. **#1 Post feed chronological** -- Every user's default feed hits `WHERE isRemoved=false AND visibility IN (...) ORDER BY createdAt DESC` without a composite index covering all three columns. This is the single highest-traffic query in the application.

2. **#3 + #4 Reel trending** -- The reel feed (TikTok-style swipe) orders by `likesCount DESC` or `viewsCount DESC` with status/isRemoved filters. No composite index covers this. At 100K reels, this becomes a full filesort.

3. **#5 Thread search by likes** -- Thread search and trending both use `isChainHead + isRemoved + visibility + ORDER BY likesCount DESC`. Zero index coverage for this 4-column pattern.

4. **#8 Message slow mode** -- Every message send in a slow-mode group triggers `WHERE conversationId=X AND senderId=Y ORDER BY createdAt DESC LIMIT 1`. The existing index scans all messages in the conversation.

5. **#14 Post scheduledAt** -- The Post model has NO index on `scheduledAt` at all (unlike Message). The cron that publishes scheduled posts must full-table-scan for scheduledAt ranges every 5 minutes.

## Notes

- All findings are against the Prisma schema as of commit `f1c595d6`. No code was modified.
- PostgreSQL B-tree indexes are column-order-sensitive. `@@index([a, b])` is NOT equivalent to `@@index([b, a])` for queries that filter on `b` but not `a`.
- Several existing indexes have the right columns but in suboptimal order (e.g., `[isRemoved, visibility, likesCount]` when queries always filter `visibility='PUBLIC'` -- putting the higher-cardinality column first would improve selectivity).
- GIN indexes on array columns (`hashtags`, `tags`) exist but compound conditions (`hasSome + isRemoved + visibility`) still require heap filtering. Partial indexes or materialized views would be the next optimization step.
- Adding indexes has a write-cost tradeoff. Each index slows INSERT/UPDATE by ~5-15%. Prioritize CRITICAL findings first; LOW findings should only be added when the associated queries show up in slow query logs.
