# J08 — API Response Size & Query Efficiency Audit

**Scope:** All `apps/api/src/modules/**/*.service.ts` (~80 service files) scanned for: unnecessary `include` relations, list endpoints without pagination, responses with unused fields (no `select`/`omit`), responses potentially >100KB, missing cursor pagination, full object returns from mutations, and nested includes creating deep object trees.

**Method:** Read every service file's Prisma queries, identified `findUnique` without `select`, `findMany` without `take`, `include` that fetches entire related models, list endpoints missing cursor pagination, and mutation returns that send back more data than the client needs.

---

## Critical Findings (User-Facing Hot Paths)

| # | Severity | File | Method / Query | Problem | Impact |
|---|----------|------|----------------|---------|--------|
| 1 | **CRITICAL** | `messages.service.ts:550` | `deleteMessage` → `findUnique({ where: { id: messageId } })` | Fetches **entire Message row** (30+ columns including `encryptedContent Bytes`, `e2eSenderRatchetKey Bytes`, media fields) just to check `senderId`. Should use `select: { senderId: true }`. | Every message delete loads cryptographic material + binary blobs from DB into Node heap for no reason. With encrypted messages, `encryptedContent` can be 10KB+. |
| 2 | **CRITICAL** | `messages.service.ts:712,725,767,786,807` | `updateGroup`, `addGroupMembers`, `removeGroupMember`, `changeGroupRole` → `findUnique({ where: { id: conversationId } })` | **Five endpoints** fetch the **entire Conversation row** (all columns) just to check `isGroup` and `createdById`. No `select` clause. | Conversation rows contain `encryptedLastMessagePreview Bytes`, `lastMessageText`, `description`, `rules` — all unnecessary for permission checks. Multiplied across every group management action. |
| 3 | **CRITICAL** | `stories.service.ts:57-79` | `getFeedStories` → 4 parallel `findMany` with `take: 10000` each | Loads up to **40,000 rows** (follows + blocks + mutes + restricts) per story feed request. Does NOT use the shared `getExcludedUserIds()` utility that caches in Redis for 60s. | Every story feed load for every user does 4 unbounded queries. At 10K follows this is ~400KB of IDs in memory. The `getExcludedUserIds` utility exists and is used by feed, threads, posts, and reels — but stories was missed. |
| 4 | **CRITICAL** | `feed.service.ts:104` | `getDismissedIds` → `findMany({ ...take: 10000 })` | Loads up to 10,000 dismissed content IDs into memory. Called by trending feed on every page load. No Redis caching. | Power users who dismiss many items load a massive array on every feed request. This should be a `Set` stored in Redis with TTL, similar to the excluded-users pattern. |
| 5 | **CRITICAL** | `community.service.ts:339-356` | `getDataExport` → 6 parallel `findMany` with `take: 100000` each | Loads up to **600,000 rows** across posts, threads, messages, reels, stories into memory. This is a **duplicate** of `privacy.service.ts:exportUserData` (which caps at 10,000 per category). | Two GDPR export endpoints exist. This one has a 100K cap per table — a single request could allocate hundreds of MB. Response body could exceed 100MB. |
| 6 | **CRITICAL** | `messages.service.ts:125` | `getConversations` → `include: { conversation: { select: CONVERSATION_SELECT } }` | `CONVERSATION_SELECT` uses `include` (not `select`) for `members`, fetching the **entire ConversationMember row** (all columns) for up to 5 members per conversation. ConversationMember has fields like `unreadCount`, `lastMessageAt`, `isBanned`, `role`, `lastReadAt`, `isArchived`, `isMuted`, `joinedAt`. Only `user.{id,username,displayName,avatarUrl,isVerified}` is needed for display. | Conversation list is the most-loaded screen in any messaging app. For 50 conversations × 5 members = 250 member records, each with ~15 unused fields. |

## Summary

| Severity | Count | Key Themes |
|----------|-------|------------|
| CRITICAL | 6 | 40K-row story queries without caching, entire binary Message/Conversation rows for permission checks, duplicate GDPR export with 100K cap, conversation list overfetching, uncached dismissal loading |
| HIGH | 12 | 5 services bypassing cached `getExcludedUserIds()` (loading 10-20K rows each), 5K follow lists uncached per feed page, full User rows (with PII) for profile lookups, `include: { host: true }` exposing PII in responses |
| MEDIUM | 16 | 34 concurrent GDPR queries, `findUnique` without `select` across 7+ services (~35 callsites), nested includes with unused fields, leaderboard N+1-like patterns |
| LOW | 4 | Admin/edge-case `findUnique` without `select` |
| **TOTAL** | **38** | |

## Top 5 Fixes by Impact

1. **Stories: Use `getExcludedUserIds()`** — Replace 4× `findMany(take: 10000)` with the cached utility. Saves 4 DB queries + 40K rows per story feed load. (~5 min fix)
2. **Messages: Add `select` to permission-check `findUnique` calls** — 12+ callsites in `messages.service.ts` fetching entire Conversation/Message rows. Add `select: { id: true, senderId: true }` or equivalent. (~30 min fix)
3. **Block/mute loading: Migrate reels, channels, search, feed-transparency to `getExcludedUserIds()`** — 4 services still doing raw 10K-row queries. The cached utility already exists. (~20 min fix)
4. **CONVERSATION_SELECT: Change `include` to `select` for members** — Replace `members: { include: { user: { select: ... } } }` with `members: { select: { user: { select: ... } } }` to stop fetching all ConversationMember columns. (~5 min fix)
5. **getUserPosts/getUserThreads: Add `select` to user lookup** — Stop fetching full User row (with PII) for profile tabs. Use `select: { id: true, isPrivate: true }`. (~5 min fix)
