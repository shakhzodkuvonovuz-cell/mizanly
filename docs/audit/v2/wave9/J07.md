# J07 — Redis Patterns Audit

**Scope:** All Redis usage in `apps/api/src/` — TTL hygiene, oversized values, hot keys, missing pipelines, connection pool risks, serialization overhead, Redis-as-primary-store risks.

**Files audited:** 35+ service files, 4 infrastructure files, ~185 Redis call sites.

## CRITICAL

### C1 — `community:dhikr:total` has NO TTL and NO eviction protection

**File:** `apps/api/src/modules/islamic/islamic.service.ts:824,871`

The global dhikr counter `community:dhikr:total` is set via `redis.incrby()` and `redis.set()` with no TTL. If Redis evicts this key under memory pressure (Upstash uses `noeviction` by default, but if changed to `allkeys-lru` this key vanishes), the total is lost. The recovery path (line 867-873) recomputes from DB, but:

1. The key has no TTL, so it persists forever as intended, but...
2. `redis.set('community:dhikr:total', String(computed))` at line 871 also sets no TTL, creating an immortal key that can only grow.
3. No periodic reconciliation cron ensures Redis and DB stay in sync. A partial Redis restart could lose increments that were not yet reflected in DB (since `incrby` happens before `prisma.dhikrSession.create`).

**Risk:** Counter can silently diverge from DB. If Redis loses the key, the recompute aggregates all DB rows (potentially millions) in a single `_sum` query with no timeout.

**Fix:** Add a reconciliation cron (e.g., hourly) that compares Redis counter against `SELECT SUM(count) FROM dhikr_sessions` and corrects drift. Set a long TTL (30d) and refresh it on every increment, so at least the key participates in TTL tracking.

### C2 — `post:impressions:{postId}` HyperLogLog keys have NO TTL — unbounded growth

**File:** `apps/api/src/modules/posts/posts.service.ts:1849,1862`

Every call to `trackImpression` creates a `post:impressions:{postId}` HyperLogLog key. These keys:
- Have NO TTL set anywhere
- Are created per-post (potentially millions of posts)
- Each HyperLogLog uses ~12KB of memory regardless of cardinality
- Are never cleaned up

At 100K posts with impressions, this is **1.2GB** of HyperLogLog data in Redis with no expiry.

**Fix:** Add `await this.redis.expire(\`post:impressions:${postId}\`, 7 * 86400)` after `pfadd`. 7-day window is reasonable for impression counting. Sync counts to DB periodically.

---

### C3 — `analytics:events` list has NO TTL — grows unbounded if no consumer exists

**File:** `apps/api/src/common/services/analytics.service.ts:25,102-105`

The analytics events list is trimmed to 100K entries via `ltrim`, but:
1. No consumer/worker is implemented to drain this list (the comment says "can be consumed by" but nothing does).
2. 100K JSON-serialized events at ~200-500 bytes each = **20-50MB** sitting permanently in Redis.
3. If the `ltrim` is the only protection, the list acts as a permanent 100K-entry buffer with no TTL.
4. Each event is `JSON.stringify`'d individually — no compression, no batching format.

**Risk:** This is Redis used as a permanent data store, not a cache. On Upstash's free/hobby tier, this alone could consume 10-20% of the memory budget.

**Fix:** Either implement a consumer worker that drains to PostgreSQL/BigQuery and set a TTL on the list, or remove the Redis analytics buffer entirely and write directly to DB via the queue system.

## HIGH

### H1 — `ab:conversions:{experimentId}:{variant}:{eventName}` keys have NO TTL

**File:** `apps/api/src/common/services/ab-testing.service.ts:225`

`trackConversion` calls `redis.incr(key)` without setting a TTL. These conversion counters accumulate forever. With N experiments x M variants x K event types, this creates potentially thousands of immortal keys. The `deleteExperiment` method cleans them up, but only if someone explicitly deletes the experiment.

**Fix:** Add `if (count === 1) await this.redis.expire(key, 90 * 24 * 3600)` after the incr (matching the 90-day experiment TTL).

---

### H2 — `mizanly:dlq` dead letter queue list has NO TTL

**File:** `apps/api/src/common/queue/queue.service.ts:26,171-172`

The DLQ list is trimmed to 1000 entries but has no TTL. Each entry is a full JSON-serialized job payload including `data` (which can contain media URLs, user data, etc.). 1000 entries x ~1KB each = ~1MB permanently in Redis.

The DLQ is also mirrored to the `FailedJob` DB table, making the Redis copy purely redundant after the DB write succeeds. But there is no cleanup mechanism.

**Fix:** Add `pipeline.expire(QueueService.DLQ_KEY, 7 * 86400)` after the `ltrim` to auto-expire the DLQ weekly.

---

### H3 — `handleConnection` performs 5-7 sequential Redis round-trips without pipelining

**File:** `apps/api/src/gateways/chat.gateway.ts:247-316`

Every WebSocket connection executes these Redis commands sequentially:
1. `incr(connKey)` — connection rate limit
2. `expire(connKey, 60)` — conditionally
3. `smembers(presenceKey)` — get existing sockets
4. `srem(presenceKey, staleId)` — per stale socket (loop)
5. `publish('socket:evict', ...)` — eviction event
6. `sadd(presenceKey, client.id)` — add new socket
7. `expire(presenceKey, PRESENCE_TTL)` — refresh TTL

Each is a separate network round-trip to Upstash (remote Redis). At ~1-3ms per round-trip over the internet, this adds **5-21ms of latency** to every WebSocket connection. Under load (reconnect storms), this serializes and creates a bottleneck.

**Fix:** Pipeline steps 1-2 together, and steps 6-7 together. Steps 3-5 must remain sequential (need result of smembers). Could reduce round-trips from 7 to 4.

### H4 — `excluded_users:{userId}` can be oversized — up to 30K user IDs serialized as JSON

**File:** `apps/api/src/common/utils/excluded-users.ts:32-66`

The function fetches up to 10,000 blocks + 10,000 mutes + 10,000 restricts = **30,000 user IDs**. Each CUID is ~25 characters. JSON-serialized: `30000 * 27 bytes = ~810KB` per user. This is cached with `redis.set(cacheKey, JSON.stringify(result), 'EX', 60)`.

For a user who has been heavily blocked/muted (e.g., a spammer), this single key can be **nearly 1MB**. On every cache miss, this entire blob is deserialized with `JSON.parse`.

**Risk:** A few hundred such users could consume hundreds of MB of Redis memory. The 60s TTL means these keys churn frequently, causing repeated large allocations.

**Fix:** For users with >1000 exclusions, skip Redis caching and query DB directly (it's already fast with indexes). Or use a Redis Set instead of a JSON blob, avoiding serialization.

### H5 — `feature_flags` hash TTL is reset on EVERY `setFlag` call to 90 days

**File:** `apps/api/src/common/services/feature-flags.service.ts:60-61`

```typescript
await this.redis.hset(this.HASH_KEY, flagName, value);
await this.redis.expire(this.HASH_KEY, 90 * 24 * 3600);
```

This is a single hash for ALL feature flags. Every `setFlag` call resets the TTL to 90 days from now. The hash is effectively immortal as long as any flag is ever updated. While this is intentional (flags should persist), it means:

1. Deleted flags via `hdel` don't reduce the hash lifetime
2. The entire hash lives or dies together — no per-flag TTL

More importantly, the `getFlagValue` method (line 113) calls `hgetall` to load ALL flags on every cache miss (every 30s). If there are 100+ flags, this is a large response being deserialized every 30 seconds.

**Fix:** This is acceptable at current scale but document that per-flag TTL requires migrating from hash to individual keys if flag count grows past ~50.

### H6 — INCR + EXPIRE race condition in rate limiters

**Files:**
- `apps/api/src/gateways/chat.gateway.ts:242-243`
- `apps/api/src/modules/auth/auth.service.ts:68-69`
- `apps/api/src/modules/notifications/notifications.service.ts:278-279`
- `apps/api/src/modules/retention/retention.service.ts:202-205`

Pattern:
```typescript
const count = await this.redis.incr(key);
if (count === 1) await this.redis.expire(key, windowSec);
```

If the process crashes between `incr` and `expire`, the key has no TTL and persists forever. This is a classic Redis anti-pattern. The key will accumulate indefinitely, eventually causing the rate limiter to permanently block the user/IP.

**Fix:** Use a Lua script or `MULTI/EXEC` to atomically set the key with TTL:
```typescript
const count = await this.redis.eval(
  "local c = redis.call('INCR', KEYS[1]); if c == 1 then redis.call('EXPIRE', KEYS[1], ARGV[1]) end; return c",
  1, key, windowSec
);
```
Alternatively, check with `TTL` command after incr and set if -1.

## MEDIUM

### M1 — User profile cache (`user:{username}`) stores full Prisma select result as JSON

**File:** `apps/api/src/modules/users/users.service.ts:194-228`

The cached user profile includes `profileLinks` (array of objects with `orderBy`), `channel` data, and all `PUBLIC_USER_FIELDS`. A user with 10 profile links and a channel with description could be **2-5KB** per cached profile.

While individually acceptable, popular users will be accessed frequently, and the 300s TTL means constant churn. 10K active users x 3KB = 30MB of profile cache.

**Impact:** Moderate. Acceptable at current scale, but should be monitored.

---

### M2 — `ScoredFeedCache` stores full item payloads in companion hash

**File:** `apps/api/src/common/utils/scored-feed-cache.ts:104-109`

Each scored feed item is `JSON.stringify`'d and stored in `{cacheKey}:data` hash. For a "For You" feed with 500 candidates, each being a full post/reel/thread object (~1-3KB), the companion hash can be **500KB-1.5MB per user**.

With 1000 concurrent users each having a scored feed cache: **500MB-1.5GB**. The 120s TTL (typical) helps, but a burst of users refreshing feeds simultaneously could spike memory.

**Fix:** Store only IDs in the sorted set (already done) but fetch full objects from DB on pagination read instead of pre-caching everything. This trades latency for memory.

---

### M3 — `trending_feed:{cursor}:{limit}` cache stores full paginated response

**File:** `apps/api/src/modules/feed/feed.service.ts:290,416`

The unauthenticated trending feed is cached as a full JSON response. A 20-item page with enriched post objects (user, media, counts) can be **10-20KB**. With multiple cursor positions cached: potentially **hundreds of KB**.

The risk: this is a hot key pattern — ALL unauthenticated users hit the same `trending_feed:first:20` key. Under high load, this becomes a Redis hot key.

**Fix:** Consider client-side caching with `ETag`/`Cache-Control` headers to reduce Redis reads, or use Redis `CLIENT NO-EVICT` on this key if available.

---

### M4 — `islamic:quran:{surah}:{ayah}:{lang}` and similar Islamic cache entries are 30 days (2592000s)

**Files:** `apps/api/src/modules/islamic/islamic.service.ts:1203,1260,1429`

Quran verse caches, hadith caches, and other Islamic reference data are cached for 30 days. Since this content is static (Quran text never changes), this is reasonable. However:

1. Cache keys are per-surah-per-ayah-per-language. 6236 ayahs x 8 languages = **49,888 potential keys**.
2. At ~500 bytes per entry: **~25MB** if all verses are cached.
3. This data is immutable — it should be cached permanently or not at all.

**Fix:** Either increase TTL to 365 days (effectively permanent) since the data never changes, or remove Redis caching for static Islamic content and use in-memory LRU cache (since it's read-only).

---

### M5 — `device_accounts:{deviceId}` has 365-day TTL — essentially permanent

**File:** `apps/api/src/modules/auth/auth.service.ts:170-171`

Device fingerprint counters persist for a full year. If an attacker rotates device IDs (trivial on Android), these keys accumulate with no cleanup. 1M unique device IDs x ~50 bytes = ~50MB of permanent Redis data.

**Impact:** Low at current scale, but the pattern doesn't scale. Consider moving to DB-only storage for device tracking since it's already a long-lived counter.

---

### M6 — No Redis memory/key monitoring or alerting

**Files:** All Redis-using services

There is no service or cron that monitors:
- Total Redis key count
- Memory usage approaching Upstash limits
- Key count by prefix (to detect leaking patterns)
- Slow log analysis

The health controller (`health.controller.ts`) pings Redis but doesn't report memory stats.

**Fix:** Add a periodic health check (e.g., in the health controller) that calls `INFO memory` and logs `used_memory_human`, `used_memory_peak_human`, and `db0:keys` count. Alert if memory exceeds 80% of Upstash plan limit.

### M7 — Session data (`session:{userId}`) uses HSET + JSON inside hash field

**File:** `apps/api/src/modules/feed/personalized-feed.service.ts:64-66`

```typescript
await this.redis.hset(key, 'json', JSON.stringify(session));
await this.redis.expire(key, PersonalizedFeedService.SESSION_TTL);
```

This stores a single JSON blob inside a hash with one field `json`. This is semantically identical to a plain `SET` but with the overhead of a hash data structure. The `viewedIds` array in the session can contain up to 1000 entries (25KB+ when serialized).

**Fix:** Use `redis.setex(key, ttl, JSON.stringify(session))` instead of `hset` + `expire`. This is simpler, atomic (no race between hset and expire), and uses less Redis memory (string vs hash overhead).

## LOW

### L1 — Single Redis connection for all operations (no read/write splitting)

**File:** `apps/api/src/config/redis.module.ts:29-33`

A single ioredis client is created with `maxRetriesPerRequest: 3`. All reads and writes share this connection. ioredis uses command pipelining internally, but:

1. No read replicas configured (Upstash supports them)
2. No separate connection for pub/sub subscriber (correctly done in chat.gateway.ts with `.duplicate()`)
3. `lazyConnect: true` means the connection isn't established until first use — if the first use is in a critical path, the user experiences connection latency

**Impact:** Acceptable at current scale. ioredis multiplexes well over a single connection. Becomes a concern at >10K RPM.

---

### L2 — Socket.io Redis adapter creates 2 additional Redis connections with no max retry config

**File:** `apps/api/src/config/socket-io-adapter.ts:29-30`

The pub/sub adapter creates two new Redis connections (`pubClient`, `subClient`) without specifying `maxRetriesPerRequest`, `retryStrategy`, or `reconnectOnError`. These use ioredis defaults (infinite retries with exponential backoff), which is fine, but different from the main client's config.

---

### L3 — `cacheAside` lock holder failure leaves stale lock for 10 seconds

**File:** `apps/api/src/common/utils/cache.ts:22-34`

The `finally` block deletes the lock, but if the process crashes (OOM kill, unhandled exception in a different async context), the lock persists for 10 seconds. During this window, other callers wait 100ms and then bypass the lock (line 38-43), which is correct behavior. However, the 100ms single-retry is fragile — if the fetcher takes 2-5 seconds, the waiting caller will skip the cache entirely and hit the DB.

**Fix:** Increase retry attempts (e.g., 3 retries with 100ms spacing) before falling through to direct fetch.

---

### L4 — Multiple `expire` calls on Quran room keys are not pipelined

**File:** `apps/api/src/gateways/chat.gateway.ts:799-813`

When joining a Quran room:
```typescript
await this.redis.hmset(roomKey, {...});
await this.redis.expire(roomKey, this.QURAN_ROOM_TTL);
// ...
await this.redis.sadd(partKey, client.id);
await this.redis.expire(partKey, this.QURAN_ROOM_TTL);
```

Four sequential Redis commands that could be pipelined into one round-trip.
