### Finding 1: QR Code / Scanner Format Mismatch — Scanning Own QR Always Fails

**File:** `apps/mobile/app/(screens)/qr-code.tsx`, line 30
**File:** `apps/mobile/app/(screens)/qr-scanner.tsx`, line 43

**Code (qr-code.tsx line 30):**
```ts
const qrValue = `mizanly://profile/${username}`;
```

**Code (qr-scanner.tsx lines 42-44):**
```ts
const handleBarCodeScanned = ({ data }: BarcodeScanningResult) => {
    // ...
    const mizanlyProfileRegex = /^https:\/\/mizanly\.app\/@([a-zA-Z0-9_]+)$/;
```

**Problem:** The QR code generates a `mizanly://profile/username` deep link format, but the scanner expects `https://mizanly.app/@username` URL format. These will NEVER match. Scanning any QR code generated by the app will always show "Invalid QR Code". Meanwhile, `share-profile.tsx` (line 38) generates the URL format `https://mizanly.app/@${user.username}`, creating an inconsistency between the two QR generation screens.

**Impact:** Core QR sharing feature is completely broken. Users scanning each other's QR codes will always see an error.

**Fix:** Either change `qr-code.tsx` to use `https://mizanly.app/@${username}` or update the scanner regex to also accept `mizanly://profile/` format.

### Finding 2: Location Picker Result Lost After router.back()

**File:** `apps/mobile/app/(screens)/location-picker.tsx`, lines 132-145

**Code:**
```ts
const handleShareLocation = useCallback(() => {
    if (!location) return;
    haptic.success();
    const locationData: LocationData = {
      latitude: location.latitude,
      longitude: location.longitude,
      address,
    };
    router.back();
    // Pass location data back via global event or params
    // The conversation screen should handle this via router params
    router.setParams({
      sharedLocation: JSON.stringify(locationData),
    });
  }, [location, address, haptic, router]);
```

**Problem:** `router.back()` is called BEFORE `router.setParams()`. After `router.back()`, the current route context changes to the parent screen. Calling `router.setParams()` after navigation has already occurred means the params are set on the wrong route (or lost entirely). The comment even acknowledges "The conversation screen should handle this via router params" but the implementation is broken.

**Impact:** Location sharing from conversation never works. The picked location is always lost.

**Fix:** Use a shared state (Zustand store, event emitter, or pass data via `router.push` to the conversation with pre-populated location) instead of back + setParams.

### Finding 14: Contact Sync Uploads Raw Phone Numbers to Server

**File:** `apps/mobile/app/(screens)/contact-sync.tsx`, lines 110-132

**Code:**
```ts
const { data } = await Contacts.getContactsAsync({
    fields: [Contacts.Fields.PhoneNumbers],
  });

  const phoneNumbers: string[] = [];
  for (const contact of data) {
    if (contact.phoneNumbers) {
      for (const phone of contact.phoneNumbers) {
        if (phone.number) {
          phoneNumbers.push(phone.number);
        }
      }
    }
  }
  // ...
  const result = await usersApi.syncContacts(phoneNumbers);
```

**Problem:** Raw phone numbers from the user's contacts are sent directly to the server without any hashing (e.g., SHA-256). **Impact:** Privacy violation. App Store rejection risk. Legal liability.

**Fix:** Hash each phone number with SHA-256 before sending to the server. The server should store and compare hashed phone numbers.

### Finding 15: Contact Sync — Anti-Pattern: State Mutation in Render

**File:** `apps/mobile/app/(screens)/contact-sync.tsx`, lines 151-155

**Code:**
```ts
const [didMount, setDidMount] = useState(false);
if (!didMount) {
    setDidMount(true);
    fetchContacts();
  }
```

**Problem:** Calling `setDidMount(true)` and `fetchContacts()` directly during render is an anti-pattern in React. State updates during render can cause unexpected behavior, double-renders in StrictMode, and potential infinite loops. This should use `useEffect` instead.

**Impact:** Potential double-fetch on StrictMode, unexpected behavior.

**Fix:** Replace with `useEffect(() => { fetchContacts(); }, []);`

### Finding 16: Voice Recorder — Stale Closure on `stop` Function

**File:** `apps/mobile/app/(screens)/voice-recorder.tsx`, lines 61-62, 71

**Code:**
```ts
// Inside start callback:
timer.current = setInterval(() => {
      setTime((t) => t >= MAX_TIME ? (stop(), t) : t + 1);
    }, 1000);

}, [haptic]); // stop is NOT in dependency array
```

**Problem:** The `start` function references `stop` inside the interval callback, but `stop` is not listed in `start`'s dependency array (only `[haptic]`). **Impact:** When recording reaches MAX_TIME, the auto-stop may behave incorrectly depending on React closure behavior.
