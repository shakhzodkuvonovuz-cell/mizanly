## Overview
Bakra (بكرة) is the TikTok-style short video space, one of Mizanly's five core spaces. Currently a placeholder screen showing "coming soon", this design implements the full V1.1 feature set matching the Instagram (Saf), X/Twitter (Majlis), and WhatsApp (Risalah) spaces.

## Design Goals
- Deliver feature parity with existing spaces
- Follow established codebase patterns exactly
- Provide seamless video experience (upload → feed → interaction)
- Integrate with existing auth, storage, caching, and search systems

## Backend Architecture

### Reels Module (`apps/api/src/modules/reels/`)
Follows NestJS modular pattern identical to Posts, Threads, Stories modules.

#### Controller Endpoints (13 total)
- `POST /reels` — Create reel (throttled: 10/min)
- `GET /reels/feed` — Get feed (with optional auth for personalized)
- `GET /reels/:id` — Get single reel
- `DELETE /reels/:id` — Delete reel (author only)
- `POST /reels/:id/like` — Like reel
- `DELETE /reels/:id/like` — Unlike reel
- `POST /reels/:id/comment` — Add comment
- `GET /reels/:id/comments` — Get paginated comments
- `POST /reels/:id/share` — Increment share count
- `POST /reels/:id/bookmark` — Bookmark reel
- `DELETE /reels/:id/bookmark` — Remove bookmark
- `POST /reels/:id/view` — Record view (deduplicated)
- `GET /reels/user/:username` — Get user's reels
- `POST /reels/:id/report` — Report reel

#### Service Patterns
- **Feed caching:** 30 seconds for "foryou" feed (Redis)
- **Counter increments:** `$executeRaw` tagged template literals with `GREATEST(0, ...)` clamping (same as posts/threads)
- **Block/mute filtering:** Exclude blocked/muted users from feed
- **Status filtering:** Exclude `PROCESSING`/`FAILED` reels from feeds
- **Pagination:** Cursor-based using `PaginatedResponse<T>` pattern

## Data Flow

### Upload Sequence
```
Mobile → pick video → get presigned URL → upload to R2 →
call reelsApi.create() → backend creates Reel (PROCESSING) →
return to feed → feed excludes PROCESSING reels
```

### Feed Loading
```
Mobile → reelsApi.getFeed() → backend checks cache →
query DB with block/mute filters → return paginated →
render with auto-play based on viewableItems
```

### Interaction Flow
```
Like → optimistic update → call API → update count →
on error revert optimistic update
```

## Technical Constraints

### Storage
- Videos stored in R2 `reels/` folder via presigned URLs
- Upload service already supports `reels` folder
- No Stream integration — using R2 presigned URLs only

### Video Processing
- Initial implementation: `status: PROCESSING` → `READY` (no actual processing)
- Future: Could add thumbnail generation, transcoding

### Performance
- Feed cache: 30 seconds for "foryou", no cache for "following"
- Video preloading: Load next video while current plays
- Memory: Unmount video components when not visible
